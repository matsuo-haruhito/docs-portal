# テスト方針

## 基本方針

- 新規機能や不具合修正では、可能な限りテストも同一ターンで追加する
- `shiroyama-server` に寄せて RSpec を標準とし、まずは request spec を優先する
- controller 単体の内部実装より、利用者が踏む画面導線と権限制御を優先して守る
- ローカルに `ruby` や `node` が無くても確認を諦めず、まず Docker 上での実行確認を試みる
- 変更内容に対応する実行確認手段がある場合は、`ruby -c` のような構文確認だけで終えず、`docker compose exec app ...` または `docker compose run --rm app ...` で意味のある確認まで行う

## 優先度

### current `main` で継続監視している主要 request spec

- `internal` は管理画面を利用でき、`company_master_admin` は `/admin` から `会社` 管理へ redirect され、`external` は管理画面を利用できないこと
- `company_master_admin` は自社の `会社` / `ユーザー` のみ管理でき、他 admin surface へは入れないこと
- 公開側では `Project.code` / `Document.slug` / `DocumentVersion.public_id` / `DocumentFile.public_id` を使い、numeric id 直指定に依存しないこと
- `external` / `company_master_admin` の文書表示境界、`public_with_login` の本文閲覧と添付 download の分離、archived restore 後の再公開境界が current behavior と一致すること
- Docusaurus HTML view は `AccessLog` を記録し、JS / CSS / image asset 配信では記録しないこと

### まず追加するべきテスト

- 認証必須画面への未ログインアクセス
- ログイン / ログアウト
- ログイン成功時に `users.last_login_at` が更新されること
- 文書表示ルート
- 版番号やファイル名に `.` を含む path の解釈
- seed 投入の整合
- seed で `public_id` が設定されること
- seed Docusaurus build で、Docusaurus が同一 id とみなす Markdown が衝突しないこと
- import 処理
- 同一 Document + version_label の再 import が失敗すること
- latest_version が現行ルールどおり更新されること
- file_uploads / zip_uploads / artifact_imports の internal API
  - 旧URLを公開しないこと
  - dry-run 作成と本実行を分けて確認すること
  - file_uploads は `file` があれば `validate_only=true` なしでも dry-run を作ること
  - file_uploads は `file` がある場合、`validate_only=true` を付けても互換的に dry-run を作ること
  - file_uploads は `file` なし・`import_dry_run_id` なしを本実行リクエストとして扱い、dry-run 作成とは誤認しないこと
  - file_uploads は relative_path / original_filename fallback / Windows区切り / traversal拒否を確認すること
  - file_uploads は content_hash の形式、照合、不一致拒否、preview の content_hash / source_commit_hash を確認すること
  - zip_uploads と file_uploads の dry-run mode 取り違えを拒否すること
- 生成ファイルイベント
  - Notifier / Job / EventBuffer / Handler / Model の各入口で safe relative path だけを扱うこと
  - `\\` 区切りを `/` に正規化し、`../`、絶対パス、Windows drive path を拒否または無視すること
  - 管理画面の path filter でも `\\` 入力を `/` として検索できること
- 各マスタ CRUD のバリデーション

## README / docs 索引 link drift の確認

`README.md` と `docs/README.md` の索引 link を更新したときは、README / docs index 専用の軽量 spec を優先して確認します。

```bash
bundle exec rspec spec/docs/readme_link_drift_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec spec/docs/readme_link_drift_spec.rb
```

この check が守る範囲:

- 対象は `README.md` と `docs/README.md` に書かれた repo 内 relative Markdown file link
- URL encoded path、スペースを含む path、日本語ファイル名は decode して実ファイルの存在を確認する
- 外部 URL、anchor-only link、scheme 付き link、画像 link は対象外
- Docusaurus anchor、外部 URL の疎通、docs 全体の link validation はこの spec では保証しない

README や docs index の broken link を直すときは、link 修正だけに閉じることを優先し、runtime code、runbook 再編、Docusaurus build pipeline の変更と混ぜすぎないようにします。

## closed item active queue label drift の確認

merge 済み PR や completed close 済み Issue の確認後は、必要に応じて closed item に残った active queue label を read-only で点検します。

```bash
GITHUB_TOKEN=... ruby .github/scripts/check_closed_active_queue_labels.rb --repo matsuo-haruhito/docs-portal
```

Docker 環境だけ利用できる場合は、app service 経由で同じ script を実行します。

```bash
docker compose run --rm \
  -e GITHUB_TOKEN \
  app ruby .github/scripts/check_closed_active_queue_labels.rb --repo matsuo-haruhito/docs-portal
```

この check が守る範囲:

- GitHub Issues API の closed Issue / PR を read-only で確認する
- `status:*` と `agent:*` だけを active queue label drift として検出する
- `area:*` / `track:*` / `risk:*` / `type:*` / `scope:*` などの履歴 label は検出対象外にする
- drift があれば対象番号、item 種別、state / state_reason、残っている active label を report する
- `GITHUB_TOKEN` がない場合は安全に終了し、GitHub への書き込みは行わない

この script は label cleanup の補助です。自動 label removal、Issue close / reopen、PR merge、required check 化、label taxonomy 変更、Workflow Manager 判断の自動化は行いません。drift が見つかった場合は、対象 Issue / PR の状態を確認してから人間または Workflow Manager の運用として label を整理します。

## 標準 seed showcase docs drift の確認

`db/seeds/support/seed_sample_document_generator.rb` が生成する標準 showcase の代表ファイルを変えたとき、または `docs/標準seedサンプルと確認用途.md` の生成内容表を更新したときは、seed showcase 専用の docs drift spec を優先して確認します。

```bash
bundle exec rspec spec/docs/standard_seed_showcase_drift_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec spec/docs/standard_seed_showcase_drift_spec.rb
```

この check が守る範囲:

- `SeedSampleDocumentGenerator` が一時 directory に生成する `seed-showcase/docs-portal-demo` 配下の file path
- `docs/標準seedサンプルと確認用途.md` の `標準 showcase の生成内容` table に書かれた path
- `提出済/README.md` を含む current / 旧版 showcase の代表 file list
- `ai-usecases` と任意 `external_samples` が標準 showcase generator とは別枠である説明

この spec は代表 path と docs の同期を守るためのものです。binary snapshot、任意 external sample、seed data cleanup、browser visual evidence、Docusaurus build 成果物の完全性はこの spec では保証しません。

## Kroki optional runtime docs/source guard の確認

Kroki の optional compose、Docusaurus mocked smoke、runtime docs、local setup docs を更新したときは、Kroki 専用の docs/source guard を優先して確認します。

```bash
ruby .github/scripts/check_kroki_runtime_docs.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ guard を実行します。

```bash
docker compose run --rm app ruby .github/scripts/check_kroki_runtime_docs.rb
```

CI では `docs-quality` workflow がこの guard を実行した後、Docusaurus package directory で mocked smoke を実行します。

```bash
cd docusaurus
npm run smoke:kroki
```

この check が守る範囲:

- `docusaurus/package.json` に `smoke:kroki` があり、docs-quality workflow からも実行されること
- smoke test が mocked fetch と、`KROKI_ENDPOINT` 未設定時に source file を含む failure を持つこと
- `.env.example`、`docker-compose.kroki.yml`、runtime notes、local setup docs が `KROKI_ENDPOINT=http://kroki:8000` と optional compose の使い方で矛盾しないこと
- Kroki-generated SVG は build artifact として扱い、source file として commit しない境界

この guard は Kroki optional runtime と docs/source の同期を守るためのものです。Mermaid / ELK の visual evidence、Kroki service image pinning、Docusaurus dependency update の採否、renderer redesign、pixel diff、browser screenshot infrastructure はこの guard では保証しません。

## host app visual evidence comment guide の確認

docs-portal 本体の小さな UI / copy / form 変更で、request spec や source guard は確認済みだが実ブラウザ確認を同じ PR で取れない場合は、[host app visual evidence comment guide](./host-app-visual-evidence-comment.md) を確認欄の正本にします。

PR body では、詳細テンプレートを貼る前に次のように短く分けます。

```text
- request spec / source guard: success
- browser visual evidence: not checked in this PR
- follow-up: #xxxx で desktop / narrow viewport を確認予定
```

この guide が守る範囲:

- docs-portal host app の admin / viewer / form / copy 変更で、CI success、request spec success、source guard success、browser visual evidence を混同しないこと
- `request spec only`、`source guard only`、`not checked` を result として明示し、未確認 viewport を確認済み扱いにしないこと
- PR comment、source Issue comment、follow-up visual evidence Issue comment のどこに残すかを短く選べること

この guide は screenshot 必須化、Playwright / system spec 導入、visual regression policy、internal UI gem の static visual artifact evidence を決めるものではありません。internal UI gem 側の evidence は [internal UI gem visual evidence runbook](./internal-ui-gem-visual-evidence-runbook.md) を正本にします。

## 社外ユーザー情報露出 smoke digest の確認

社外ユーザーの可視範囲、AI context export、ZIP / download、外部送付履歴、文書ショートカットなどに触れる PR で、短い evidence を残す場合は、[社外ユーザー向け情報露出点検チェックリスト](./社外ユーザー向け情報露出点検チェックリスト.md) と `bin/external_user_exposure_smoke` を確認入口にします。

代表的な smoke は次を優先します。

```bash
bundle exec ruby bin/external_user_exposure_smoke --format markdown
```

Docker 環境だけ利用できる場合は、app service 経由で同じ smoke を実行します。

```bash
docker compose run --rm app bundle exec ruby bin/external_user_exposure_smoke --format markdown
```

PR body や PR comment には、詳細な payload を貼る前に次のように短く分けます。

```text
- external exposure smoke: passed
- digest: smoke name / executed at / representative specs / next runbook only
- raw payloads and unauthorized document names: not pasted
```

この digest が守る範囲:

- external / company_master_admin の文書表示、検索、関連情報、配信、AI context export、外部送付履歴、文書ショートカットが current user の可視範囲に閉じること
- Markdown digest には smoke 名、実行時刻、RSpec 結果、代表 spec、対象 surface、確認目的、次に見る runbook だけを残すこと
- HTML / JSON / ZIP payload 本文、raw response、token-like value、文書タイトル、添付 metadata、権限外文書名を evidence へ貼らないこと
- admin / integration metadata exposure は [運用 metadata 情報露出点検チェックリスト](./運用metadata情報露出点検チェックリスト.md) と `bin/operational_metadata_exposure_smoke` 側へ分けること

この確認は external visibility に触れる変更時の evidence 候補です。すべての PR への必須 gate 化、CI workflow の大規模変更、代表 spec 一覧の再編、security policy の最終判断、admin / integration metadata smoke との統合は別 Issue で扱います。

## CSV companion metadata JSON の確認

監査ログ、文書利用状況、文書セットの CSV companion metadata JSON を触るときは、[CSV条件metadata JSON 運用メモ](./CSV条件metadata_JSON運用メモ.md) を読んで、CSV 本体、metadata JSON、HTML 表示設定の責務差を確認します。

代表的な request spec は次を優先します。

```bash
bundle exec rspec \
  spec/requests/admin_access_logs_spec.rb \
  spec/requests/admin_document_usage_reports_spec.rb \
  spec/requests/admin_document_sets_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec \
  spec/requests/admin_access_logs_spec.rb \
  spec/requests/admin_document_usage_reports_spec.rb \
  spec/requests/admin_document_sets_spec.rb
```

この確認が守る範囲:

- 監査ログ metadata は current filter の最新 200 件 CSV の companion であること
- 文書利用状況 metadata は選択案件の current usage report の companion であること
- 文書セット metadata は current filter に一致する文書セット CSV の companion であること
- HTML 表示設定は画面上の列表示だけを変え、CSV header / row や metadata 条件を変えないこと
- invalid date や unsupported enum は安全側に正規化または `ignored_filters` で確認できること

この確認は CSV companion metadata JSON の境界を守るためのものです。CSV header / row contract の変更、metadata schema 再設計、scheduled report、background export、retention policy、表示設定連動、admin export 全体の再編は別 Issue で扱います。

## 管理画面 route identifier contract の確認

管理画面の member route、controller lookup、view link helper、request / routing spec を追加・変更したときは、admin route identifier contract の軽量 spec を優先して確認します。

```bash
bundle exec rspec spec/routing/admin_route_identifier_contract_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec spec/routing/admin_route_identifier_contract_spec.rb
```

この check が守る範囲:

- 主要 admin member resource は `:public_id` を URL 識別子として使うこと
- `admin/projects` と project-owned member action controller は `:code` を使うこと
- collection-only controller は dynamic segment を持たないこと
- `admin/model_browser/:model_key` などの個別 dynamic segment が意図的に分類されていること

`PUBLIC_ID_CONTROLLERS` / `CODE_CONTROLLERS` / `COLLECTION_ONLY_CONTROLLERS` の分類を変える場合は、route 定義、controller lookup、view link helper、request / routing spec、`docs/開発・保守ガイド.md` の説明を同じ識別子で見直します。未確認の numeric `:id` 導線が見つかった場合は、広い route 移行にせず、対象 resource と URL を確認して concrete issue に切ります。

この spec は route 変更時の確認用です。route 変更と無関係な docs-only PR のために docs-quality workflow へ常時重い gate を追加するものではありません。

## 関連 gem smoke の代表確認

関連 gem の更新や screen adoption を進めるときは、まず `docs/関連gem連携調査runbook.md` の「3 gem 共通 verification matrix」と「代表 smoke の早見表」を正本にします。`#1026` の採用マトリクスは責務整理、ここでの smoke は docs-portal 側で壊していないことを確認する quality gate として分けて扱います。

- `tree_view-rails`
  - 代表 surface は `app/views/documents/_tree.html.erb` と `app/views/projects/_document_detail_tree.html.erb`
  - まず `spec/requests/document_tree_regressions_spec.rb` で sidebar tree、detail tree、persisted state のどこを確認したかを残す
  - request spec で確認できないブラウザ操作差分がある場合だけ、manual QA または小さな system spec へ広げる
- `rails_table_preferences`
  - 代表 surface は `app/views/admin/document_sets/index.html.slim` の editor + table
  - まず `spec/requests/admin_document_sets_index_spec.rb` と `spec/requests/admin_document_sets_spec.rb` で stable column key、filter / preset、mounted engine save のどこを確認したかを残す
  - embedded table や preview fallback は同じ PR に混ぜず、対象 issue の update log へ未確認または別確認として分ける
- `rails_fields_kit`
  - 代表 surface は `app/views/admin/document_sets/_form.html.slim` の `rfk_select` 群
  - まず `spec/requests/admin_document_sets_spec.rb` で initial load、invalid rerender、selected value / placeholder のどこを確認したかを残す
  - Turbo 再訪や remote search のように request spec だけでは足りない差分がある場合だけ、manual QA または小さな system spec を追加する

update log には、少なくとも `gem 名`、`from SHA / tag`、`to SHA / tag`、`使った代表 smoke`、`確認結果`、`rollback target` を 1 セットで残します。target SHA の確定や bump 実施は各 child issue / PR の責務であり、この方針 docs では runtime code、DB schema、認可、UI redesign は変更しません。

## 実装メモ

- テストデータは FactoryBot を優先し、必要に応じて spec 内で最小限だけ上書きする
- ファイル配信や Docusaurus 公開物のテストでは、`storage/docs_sites/` 配下に一時ファイルを作って検証する
- public_id のテストでは、DB id と public_id を意図的に異なる値として扱う
- Docusaurus build を伴うテストは重くなりやすいため、request spec では必要最小限の HTML / asset ファイルを fixture として配置して確認する
- Docusaurus renderer 連携は request spec では Job enqueue までを確認し、archive / client / artifact install / Job 連携は service spec と job spec で確認する

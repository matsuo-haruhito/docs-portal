# テスト方針

## 基本方針

- 新規機能や不具合修正では、可能な限りテストも同一ターンで追加する
- `shiroyama-server` に寄せて RSpec を標準とし、まずは request spec を優先する
- controller 単体の内部実装より、利用者が踏む画面導線と権限制御を優先して守る
- ローカルに `ruby` や `node` が無くても確認を諦めず、まず Docker 上での実行確認を試みる
- 変更内容に対応する実行確認手段がある場合は、`ruby -c` のような構文確認だけで終えず、`docker compose exec app ...` または `docker compose run --rm app ...` で意味のある確認まで行う

## 優先度

### current `main` で request spec により継続監視している項目

- 管理画面の権限制御
  - `internal` は管理画面を利用できる
  - `company_master_admin` は `/admin` から `会社` へ redirect され、自社の `会社` / `ユーザー` だけ管理できる
  - `external` は管理画面を利用できない
- 文書表示ルート
- public_id / code / slug による URL 解決
- numeric id 直指定では外部公開 route を解決しないこと
- external / company_master_admin は未所属 Project を参照できないこと
- view 権限のみでは DocumentFile を取得できないこと
- download 権限では Docusaurus HTML と DocumentFile の両方を取得できること
- `public_with_login` は ProjectMembership があれば閲覧できること
- `public_with_login` は DocumentPermission なしでも本文閲覧できるが、添付 download には `download` 権限が要ること
- draft / archived の DocumentVersion は external / company_master_admin に表示されないこと
- `company_master_admin` の文書閲覧境界は `external` と同じであること
- archived Document は社外導線から隠れ、admin 管理画面から restore すると再表示されること

### 次に追加するべきテスト

- 認証必須画面への未ログインアクセス
- ログイン / ログアウト
- ログイン成功時に `users.last_login_at` が更新されること
- 版番号やファイル名に `.` を含む path の解釈
- seed 投入の整合
- seed で `public_id` が設定されること
- seed Docusaurus build で、Docusaurus が同一 id とみなす Markdown が衝突しないこと
- import 処理
- 同一 Document + version_label の再 import が失敗すること
- latest_version が現行ルールどおり更新されること
- file_uploads / zip_uploads / artifact_imports の internal API
  - 旧URLを公開しないこと
  - dry-run 作成と本実行を分けて確認すること
  - file_uploads は `file` があれば `validate_only=true` なしでも dry-run を作ること
  - file_uploads は `file` がある場合、`validate_only=true` を付けても互換的に dry-run を作ること
  - file_uploads は `file` なし・`import_dry_run_id` なしを本実行リクエストとして扱い、dry-run 作成とは誤認しないこと
  - file_uploads は relative_path / original_filename fallback / Windows区切り / traversal拒否を確認すること
  - file_uploads は content_hash の形式、照合、不一致拒否、preview の content_hash / source_commit_hash を確認すること
  - zip_uploads と file_uploads の dry-run mode 取り違えを拒否すること
- 生成ファイルイベント
  - Notifier / Job / EventBuffer / Handler / Model の各入口で safe relative path だけを扱うこと
  - `\\` 区切りを `/` に正規化し、`../`、絶対パス、Windows drive path を拒否または無視すること
  - 管理画面の path filter でも `\\` 入力を `/` として検索できること
- 各マスタ CRUD のバリデーション
- AccessLog の記録対象
  - Docusaurus HTML view は記録する
  - DocumentFile download は記録する
  - JS / CSS / image asset は記録しない

## 実装メモ

- テストデータは FactoryBot を優先し、必要に応じて spec 内で最小限だけ上書きする
- ファイル配信や Docusaurus 公開物のテストでは、`storage/docs_sites/` 配下に一時ファイルを作って検証する
- public_id のテストでは、DB id と public_id を意図的に異なる値として扱う
- Docusaurus build を伴うテストは重くなりやすいため、request spec では必要最小限の HTML / asset ファイルを fixture として配置して確認する
- Docusaurus renderer 連携は request spec では Job enqueue までを確認し、archive / client / artifact install / Job 連携は service spec と job spec で確認する

## internal upload API の確認コマンド例

CI が使えない場合は、少なくとも次をローカルまたは Docker 上で優先実行する。

```bash
bundle exec rspec spec/requests/api/internal
bundle exec rspec spec/models/build_freshness_guard_spec.rb spec/models/admin/api_specification_page_spec.rb
```

Docker 環境だけ利用できる場合は、同じ対象を app service 経由で実行する。

```bash
docker compose run --rm app bundle exec rspec spec/requests/api/internal
docker compose run --rm app bundle exec rspec spec/models/build_freshness_guard_spec.rb spec/models/admin/api_specification_page_spec.rb
```

## 生成ファイルイベント変更時の確認コマンド例

path 正規化、イベント集約、再dispatch、管理画面 filter を触った場合は次を優先する。

```bash
bundle exec rspec \
  spec/models/generated_file_event_spec.rb \
  spec/jobs/generated_file_change_event_job_spec.rb \
  spec/jobs/generated_file_event_dispatch_job_spec.rb \
  spec/services/generated_files/change_event_notifier_spec.rb \
  spec/services/generated_files/event_buffer_spec.rb \
  spec/services/generated_files/change_event_handler_spec.rb \
  spec/requests/admin_generated_file_events_spec.rb \
  spec/requests/admin_generated_file_event_path_filter_spec.rb \
  spec/requests/admin_generated_file_event_retry_limit_spec.rb \
  spec/requests/admin_generated_file_event_retry_path_filter_spec.rb
```

## Docusaurus renderer 変更時の確認コマンド例

Rails 側の低コスト確認:

```bash
bundle exec rspec \
  spec/requests/document_uploads_spec.rb \
  spec/renderers/docusaurus_site_renderer_spec.rb \
  spec/jobs/docusaurus_preview_build_job_spec.rb \
  spec/services/line_diff_builder_spec.rb \
  spec/services/markdown_line_diff_builder_spec.rb \
  spec/services/docusaurus_preview_archive_builder_spec.rb \
  spec/services/docusaurus_preview_artifact_installer_spec.rb \
  spec/services/docusaurus_renderer_client_spec.rb \
  spec/services/document_file_viewer_plan_spec.rb \
  spec/models/document_version_path_spec.rb \
  spec/models/document_file_content_type_spec.rb \
  spec/importers/git_import/path_catalog_spec.rb \
  spec/checks/document_version_quality/markdown_reference_checks_spec.rb
```

Docker 上で renderer / Kroki も含めて確認する場合:

```bash
docker compose -f docker-compose.yml -f docker-compose.kroki.yml -f docker-compose.docusaurus.yml build docusaurus
docker compose -f docker-compose.yml -f docker-compose.kroki.yml -f docker-compose.docusaurus.yml up -d
docker compose exec app bundle exec rspec spec/requests/document_uploads_spec.rb
```

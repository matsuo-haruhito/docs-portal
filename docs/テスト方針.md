# テスト方針

## 基本方針

- 新規機能や不具合修正では、可能な限りテストも同一ターンで追加する
- `shiroyama-server` に寄せて RSpec を標準とし、まずは request spec を優先する
- controller 単体の内部実装より、利用者が踏む画面導線と権限制御を優先して守る
- ローカルに `ruby` や `node` が無くても確認を諦めず、まず Docker 上での実行確認を試みる
- 変更内容に対応する実行確認手段がある場合は、`ruby -c` のような構文確認だけで終えず、`docker compose exec app ...` または `docker compose run --rm app ...` で意味のある確認まで行う

## 優先度

### current `main` で継続監視している主要 request spec

- `internal` は管理画面を利用でき、`company_master_admin` は `/admin` から `会社` 管理へ redirect され、`external` は管理画面を利用できないこと
- `company_master_admin` は自社の `会社` / `ユーザー` のみ管理でき、他 admin surface へは入れないこと
- 公開側では `Project.code` / `Document.slug` / `DocumentVersion.public_id` / `DocumentFile.public_id` を使い、numeric id 直指定に依存しないこと
- `external` / `company_master_admin` の文書表示境界、`public_with_login` の本文閲覧と添付 download の分離、archived restore 後の再公開境界が current behavior と一致すること
- Docusaurus HTML view は `AccessLog` を記録し、JS / CSS / image asset 配信では記録しないこと

### まず追加するべきテスト

- 認証必須画面への未ログインアクセス
- ログイン / ログアウト
- ログイン成功時に `users.last_login_at` が更新されること
- 文書表示ルート
- 版番号やファイル名に `.` を含む path の解釈
- seed 投入の整合
- seed で `public_id` が設定されること
- seed Docusaurus build で、Docusaurus が同一 id とみなす Markdown が衝突しないこと
- import 処理
- 同一 Document + version_label の再 import が失敗すること
- latest_version が現行ルールどおり更新されること
- file_uploads / zip_uploads / artifact_imports の internal API
  - 旧URLを公開しないこと
  - dry-run 作成と本実行を分けて確認すること
  - file_uploads は `file` があれば `validate_only=true` なしでも dry-run を作ること
  - file_uploads は `file` がある場合、`validate_only=true` を付けても互換的に dry-run を作ること
  - file_uploads は `file` なし・`import_dry_run_id` なしを本実行リクエストとして扱い、dry-run 作成とは誤認しないこと
  - file_uploads は relative_path / original_filename fallback / Windows区切り / traversal拒否を確認すること
  - file_uploads は content_hash の形式、照合、不一致拒否、preview の content_hash / source_commit_hash を確認すること
  - zip_uploads と file_uploads の dry-run mode 取り違えを拒否すること
- 生成ファイルイベント
  - Notifier / Job / EventBuffer / Handler / Model の各入口で safe relative path だけを扱うこと
  - `\\` 区切りを `/` に正規化し、`../`、絶対パス、Windows drive path を拒否または無視すること
  - 管理画面の path filter でも `\\` 入力を `/` として検索できること
- 各マスタ CRUD のバリデーション

## README / docs 索引 link drift の確認

`README.md` と `docs/README.md` の索引 link を更新したときは、README / docs index 専用の軽量 spec を優先して確認します。

```bash
bundle exec rspec spec/docs/readme_link_drift_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec spec/docs/readme_link_drift_spec.rb
```

この check が守る範囲:

- 対象は `README.md` と `docs/README.md` に書かれた repo 内 relative Markdown file link
- URL encoded path、スペースを含む path、日本語ファイル名は decode して実ファイルの存在を確認する
- 外部 URL、anchor-only link、scheme 付き link、画像 link は対象外
- Docusaurus anchor、外部 URL の疎通、docs 全体の link validation はこの spec では保証しない

README や docs index の broken link を直すときは、link 修正だけに閉じることを優先し、runtime code、runbook 再編、Docusaurus build pipeline の変更と混ぜすぎないようにします。

## 管理画面 route identifier contract の確認

管理画面の member route、controller lookup、view link helper、request / routing spec を追加・変更したときは、admin route identifier contract の軽量 spec を優先して確認します。

```bash
bundle exec rspec spec/routing/admin_route_identifier_contract_spec.rb
```

Docker 環境だけ利用できる場合は、app service 経由で同じ spec を実行します。

```bash
docker compose run --rm app bundle exec rspec spec/routing/admin_route_identifier_contract_spec.rb
```

この check が守る範囲:

- 主要 admin member resource は `:public_id` を URL 識別子として使うこと
- `admin/projects` と project-owned member action controller は `:code` を使うこと
- collection-only controller は dynamic segment を持たないこと
- `admin/model_browser/:model_key` などの個別 dynamic segment が意図的に分類されていること

`PUBLIC_ID_CONTROLLERS` / `CODE_CONTROLLERS` / `COLLECTION_ONLY_CONTROLLERS` の分類を変える場合は、route 定義、controller lookup、view link helper、request / routing spec、`docs/開発・保守ガイド.md` の説明を同じ識別子で見直します。未確認の numeric `:id` 導線が見つかった場合は、広い route 移行にせず、対象 resource と URL を確認して concrete issue に切ります。

この spec は route 変更時の確認用です。route 変更と無関係な docs-only PR のために docs-quality workflow へ常時重い gate を追加するものではありません。

## 関連 gem smoke の代表確認

関連 gem の更新や screen adoption を進めるときは、まず `docs/関連gem連携調査runbook.md` の「3 gem 共通 verification matrix」と「代表 smoke の早見表」を正本にします。`#1026` の採用マトリクスは責務整理、ここでの smoke は docs-portal 側で壊していないことを確認する quality gate として分けて扱います。

- `tree_view-rails`
  - 代表 surface は `app/views/documents/_tree.html.erb` と `app/views/projects/_document_detail_tree.html.erb`
  - まず `spec/requests/document_tree_regressions_spec.rb` で sidebar tree、detail tree、persisted state のどこを確認したかを残す
  - request spec で確認できないブラウザ操作差分がある場合だけ、manual QA または小さな system spec へ広げる
- `rails_table_preferences`
  - 代表 surface は `app/views/admin/document_sets/index.html.slim` の editor + table
  - まず `spec/requests/admin_document_sets_index_spec.rb` と `spec/requests/admin_document_sets_spec.rb` で stable column key、filter / preset、mounted engine save のどこを確認したかを残す
  - embedded table や preview fallback は同じ PR に混ぜず、対象 issue の update log へ未確認または別確認として分ける
- `rails_fields_kit`
  - 代表 surface は `app/views/admin/document_sets/_form.html.slim` の `rfk_select` 群
  - まず `spec/requests/admin_document_sets_spec.rb` で initial load、invalid rerender、selected value / placeholder のどこを確認したかを残す
  - Turbo 再訪や remote search のように request spec だけでは足りない差分がある場合だけ、manual QA または小さな system spec を追加する

update log には、少なくとも `gem 名`、`from SHA / tag`、`to SHA / tag`、`使った代表 smoke`、`確認結果`、`rollback target` を 1 セットで残します。target SHA の確定や bump 実施は各 child issue / PR の責務であり、この方針 docs では runtime code、DB schema、認可、UI redesign は変更しません。

## 実装メモ

- テストデータは FactoryBot を優先し、必要に応じて spec 内で最小限だけ上書きする
- ファイル配信や Docusaurus 公開物のテストでは、`storage/docs_sites/` 配下に一時ファイルを作って検証する
- public_id のテストでは、DB id と public_id を意図的に異なる値として扱う
- Docusaurus build を伴うテストは重くなりやすいため、request spec では必要最小限の HTML / asset ファイルを fixture として配置して確認する
- Docusaurus renderer 連携は request spec では Job enqueue までを確認し、archive / client / artifact install / Job 連携は service spec と job spec で確認する

## internal upload API の確認コマンド例

CI が使えない場合は、少なくとも次をローカルまたは Docker 上で優先実行する。

```bash
bundle exec rspec spec/requests/api/internal
bundle exec rspec spec/models/build_freshness_guard_spec.rb spec/models/admin/api_specification_page_spec.rb
```

Docker 環境だけ利用できる場合は、同じ対象を app service 経由で実行する。

```bash
docker compose run --rm app bundle exec rspec spec/requests/api/internal
docker compose run --rm app bundle exec rspec spec/models/build_freshness_guard_spec.rb spec/models/admin/api_specification_page_spec.rb
```

```bash
  spec/requests/admin_generated_file_event_retry_path_filter_spec.rb
```

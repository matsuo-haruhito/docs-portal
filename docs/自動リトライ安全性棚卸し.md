# 自動リトライ安全性棚卸し

この文書は issue `#1055` に対応する、import / build / mail / webhook の自動リトライ前に見る判断材料です。runtime code、queue adapter、mail / webhook delivery 仕様は変更せず、現行の監視・runbook から次の実装 issue を切るための棚卸しに留めます。

## 目的

自動リトライは「失敗した処理をもう一度流せるか」だけでなく、二重実行で外部副作用や通知重複が起きても安全かを確認してから対象ごとに判断します。

この repo では、まず次を分けます。

- 手動再実行で十分な処理
- 自動リトライしても副作用が小さい処理
- retry 前に locking / idempotency key / unique constraint / external id が必要な処理
- 自動化より先に通知や失敗理由の見え方を整える処理

## 共通判定基準

| 判定 | 使う場面 | 先に必要な guard |
| --- | --- | --- |
| 手動再実行優先 | 失敗原因の確認や対象の絞り込みが必要な処理 | runbook、status filter、失敗理由、対象件数の確認 |
| 条件付きで自動リトライ可 | 同じ入力を再処理しても最終状態が同じになる処理 | concurrency key、入力の正規化、retry 親子 metadata、上限回数 |
| guard 追加後に再評価 | 外部送信、通知、永続ファイル更新など二重副作用があり得る処理 | idempotency key、unique constraint、外部ID、dedupe window、locking |
| 通知・可観測性を先行 | まだ delivery 仕様や失敗傾向が見えていない処理 | alert、failure count、失敗 payload の安全な要約、runbook 導線 |

## 処理別マトリクス

| 処理 | current の観測面 | 二重実行で起きうる副作用 | current guard / 手動導線 | retry 方針 | 次に切るなら |
| --- | --- | --- | --- | --- | --- |
| import | `PublishJob` の `pending` / `imported` / `published` / `failed`、`log_message`、manifest の `source_repo` / `source_branch` / `source_commit_hash` | 同じ artifact を再 import したときの `latest_version` 上書き、添付ファイル差し替え、生成ファイル event の再発行 | `DocumentImporter` は import root 配下の path 検証、version label 重複拒否、失敗時の `PublishJob.failed` 記録を持つ | 自動リトライは保留。まず失敗原因と manifest / artifact の同一性を見て手動再実行する | import 対象を 1 種に絞り、source commit hash と version label を使った idempotency / replay 方針を決める issue |
| Docusaurus build / generated file build | `GeneratedFileEvent`、`GeneratedFileRun`、`GeneratedFileJob` の retry metadata、generator / output writer / event source filter | build output の上書き、同じ入力から複数 run が並ぶこと、bulk retry で対象が広がること | `GeneratedFileJob` / `GeneratedFileChangeEventJob` は concurrency key を持ち、run retry は親 run public_id と requester metadata を残す。admin 画面には event 再dispatch / run 再実行の手動導線がある | 条件付きで自動リトライ候補。ただし first slice は failed run の原因分類と上限回数を決めてから | generator / output writer の 1 種に絞り、retry 回数上限・対象 status・metadata の保持を固定する issue |
| mail delivery | `docs/監視・アラート設計.md` では将来の外部依存として扱う。現時点では具体 delivery 仕様をこの文書で確定しない | 同じメールの重複送信、送信先の誤通知、unsubscribe / bounce handling との競合 | current repo では自動 retry 前提の delivery runbook を持たない | 自動リトライは不可。まず delivery 仕様、送信単位、外部 provider の idempotency / message id を決める | mail 機能本体が固まった後、通知種別 1 つ単位で message id / dedupe / retry policy を扱う issue |
| webhook delivery | `docs/監視・アラート設計.md` では将来の外部依存として扱う。admin route inventory には `webhook_endpoints` があるが delivery 契約はこの文書で新設しない | 外部 endpoint への重複 POST、相手側処理の二重実行、署名 timestamp / replay protection との競合 | current repo では delivery 成功・失敗履歴や retry policy をこの文書で確定しない | 自動リトライは不可。まず endpoint ごとの idempotency key、event id、署名、delivery log を決める | webhook event 1 種単位で delivery log / idempotency key / retry window を扱う issue |

## 自動リトライ issue に切る前のチェックリスト

- retry 対象は 1 処理または 1 delivery 種別に絞れている
- 同じ入力を 2 回処理したときの最終状態が説明できる
- 外部送信がある場合、相手側で dedupe できる id がある
- status / run / event / log から、手動確認すべき失敗と機械的に戻せる失敗を分けられる
- retry 回数、対象 status、対象期間、bulk 対象件数の上限がある
- 失敗 payload や metadata に機密情報を過剰に出さない
- 既存の `定義を同期`、`即時実行を要求`、event 再dispatch、run 再実行の導線と競合しない

## 現時点の結論

- import は manifest / artifact の同一性と `latest_version` 影響を先に固定するまで、手動再実行を優先する
- build / generated file は既に手動 retry 導線と retry metadata があるため、限定された対象なら自動リトライ候補にできる
- mail / webhook は delivery 仕様と外部 idempotency が未確定なため、自動リトライではなく通知・delivery 契約の issue を先に切る
- 次の実装 issue は broad な自動リトライ基盤ではなく、対象処理 1 つ単位に分ける

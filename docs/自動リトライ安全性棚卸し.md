# 自動リトライ安全性棚卸し

この文書は issue `#1055` に対応する、import / build / mail / webhook の自動リトライ前に見る判断材料です。runtime code、queue adapter、mail / webhook delivery 仕様は変更せず、現行の監視・runbook から次の実装 issue を切るための棚卸しに留めます。

## 目的

自動リトライは「失敗した処理をもう一度流せるか」だけでなく、二重実行で外部副作用や通知重複が起きても安全かを確認してから対象ごとに判断します。

この repo では、まず次を分けます。

- 手動再実行で十分な処理
- 自動リトライしても副作用が小さい処理
- retry 前に locking / idempotency key / unique constraint / external id が必要な処理
- 自動化より先に通知や失敗理由の見え方を整える処理

## 共通判定基準

| 判定 | 使う場面 | 先に必要な guard |
| --- | --- | --- |
| 手動再実行優先 | 失敗原因の確認や対象の絞り込みが必要な処理 | runbook、status filter、失敗理由、対象件数の確認 |
| 条件付きで自動リトライ可 | 同じ入力を再処理しても最終状態が同じになる処理 | concurrency key、入力の正規化、retry 親子 metadata、上限回数 |
| guard 追加後に再評価 | 外部送信、通知、永続ファイル更新など二重副作用があり得る処理 | idempotency key、unique constraint、外部ID、dedupe window、locking |
| 通知・可観測性を先行 | まだ delivery 仕様や失敗傾向が見えていない処理 | alert、failure count、失敗 payload の安全な要約、runbook 導線 |

## 処理別マトリクス

| 処理 | current の観測面 | 二重実行で起きうる副作用 | current guard / 手動導線 | retry 方針 | 次に切るなら |
| --- | --- | --- | --- | --- | --- |
| import | `PublishJob` の `pending` / `imported` / `published` / `failed`、`log_message`、manifest の `source_repo` / `source_branch` / `source_commit_hash` | 同じ artifact を再 import したときの `latest_version` 上書き、添付ファイル差し替え、生成ファイル event の再発行 | `DocumentImporter` は import root 配下の path 検証、version label 重複拒否、失敗時の `PublishJob.failed` 記録を持つ | 自動リトライは保留。まず失敗原因と manifest / artifact の同一性を見て手動再実行する | import 対象を 1 種に絞り、source commit hash と version label を使った idempotency / replay 方針を決める issue |
| Docusaurus build / generated file build | `GeneratedFileEvent`、`GeneratedFileRun`、`GeneratedFileJob` の retry metadata、generator / output writer / event source filter | build output の上書き、同じ入力から複数 run が並ぶこと、bulk retry で対象が広がること | `GeneratedFileJob` / `GeneratedFileChangeEventJob` は concurrency key を持ち、run retry は親 run public_id と requester metadata を残す。admin 画面には event 再dispatch / run 再実行の手動導線がある。自動 retry first slice は `ai_usecase_decision_flow` + `filesystem` writer の failed parent run に限定し、`auto_retry: true` child がある parent は再 enqueue しない。手動 retry / bulk retry child は自動 retry 済みとは扱わない | 条件付きで自動リトライ候補。ただし first slice は対象 generator / writer、1 parent 1 auto retry、metadata の読み分けを固定し、runtime 自動化の拡大には進まない | generator / output writer の 1 種に絞り、retry 回数上限・対象 status・metadata の保持を固定する issue |
| mail delivery | `docs/監視・アラート設計.md` では将来の外部依存として扱う。現時点では具体 delivery 仕様をこの文書で確定しない | 同じメールの重複送信、送信先の誤通知、unsubscribe / bounce handling との競合 | current repo では自動 retry 前提の delivery runbook を持たない | 自動リトライは不可。まず delivery 仕様、送信単位、外部 provider の idempotency / message id を決める | mail 機能本体が固まった後、通知種別 1 つ単位で message id / dedupe / retry policy を扱う issue |
| webhook delivery | `WebhookDelivery` の送信履歴、詳細、送信履歴検索、1 件手動再送、`失敗` 表示中の表示範囲まとめて再送、継続失敗 handoff JSON | 外部 endpoint への重複 POST、相手側処理の二重実行、署名 timestamp / replay protection との競合 | 再送は failed かつ endpoint が有効な delivery に限定する。まとめて再送は `Webhook` 画面の `失敗` 表示中、最近 50 件のうち再送可能な delivery だけを対象にする。継続失敗 handoff JSON は threshold 3、lookback 200、limit 20 の read-only evidence で、通知・ack・自動 retry は実行しない。`X-Docs-Portal-Delivery` は将来の冪等キー候補として扱うが、受信側 dedupe を保証するものではない | 自動リトライは未実装のまま保留。current support は手動再送と handoff に閉じ、scheduled retry / exponential backoff / retry metadata / 親子 delivery relation は先取りしない | webhook event 1 種単位で、対象 event、retry 回数 / 期間、重複送信時の受信側挙動、failure handoff との役割差を決める issue |

## 自動リトライ issue に切る前のチェックリスト

- retry 対象は 1 処理または 1 delivery 種別に絞れている
- 同じ入力を 2 回処理したときの最終状態が説明できる
- 外部送信がある場合、相手側で dedupe できる id がある
- status / run / event / log から、手動確認すべき失敗と機械的に戻せる失敗を分けられる
- retry 回数、対象 status、対象期間、bulk 対象件数の上限がある
- 失敗 payload や metadata に機密情報を過剰に出さない
- Webhook delivery を対象にする場合、手動再送、表示範囲まとめて再送、継続失敗 handoff JSON、自動 retry queue の役割差を説明できる
- 既存の `定義を同期`、`即時実行を要求`、event 再dispatch、run 再実行の導線と競合しない

## 現時点の結論

- import は manifest / artifact の同一性と `latest_version` 影響を先に固定するまで、手動再実行を優先する
- build / generated file は既に手動 retry 導線と retry metadata があり、first slice では `ai_usecase_decision_flow` + `filesystem` writer の failed parent run だけを自動リトライ候補として扱う。`document_version` writer、import、mail、webhook、手動 retry / bulk retry child run は対象外のままにする
- auto retry metadata は `retry_requested_by_user_id: nil`、`auto_retry: true`、`retry_reason` で手動 retry / bulk retry と読み分ける。parent run 1 件につき auto retry child は 1 回までとし、手動 / bulk retry child の存在だけでは auto retry 済みとは扱わない
- mail は delivery 仕様と外部 idempotency が未確定なため、自動リトライではなく通知・delivery 契約の issue を先に切る
- webhook は送信履歴、1 件手動再送、失敗表示中の表示範囲まとめて再送、継続失敗 handoff JSON までが current support。自動 retry queue、scheduled retry、exponential backoff、retry metadata、親子 delivery relation は未実装のままなので、次に切る場合は event 1 種単位で `X-Docs-Portal-Delivery` の冪等キー候補としての扱い、受信側 dedupe の確認、retry 回数 / 期間、handoff との役割差を決める
- 次の実装 issue は broad な自動リトライ基盤ではなく、対象処理 1 つ単位に分ける

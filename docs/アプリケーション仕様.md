# アプリケーション仕様

## 概要

- 本アプリは、案件ごとの文書公開、版管理、閲覧権限制御、添付ファイル配布を行う Rails ポータルである
- Markdown 文書は Docusaurus build を公開物として扱い、Rails 側では閲覧権限を付けたうえで配信する
- DB は単一構成とし、案件・会社・文書権限による論理分離を行う
- 社内ユーザーは案件横断で文書を閲覧できる

## 識別子

- DB 内部の主キー `id` は、外部公開 URL には原則使用しない
- アプリケーション管理テーブルには `public_id` を持たせる
- `public_id` は作成時に自動生成し、原則変更しない
- `public_id` は URL-safe な文字列とし、テーブルごとに prefix を付ける
- 通常の `create!` / `save!` 経路では model concern で自動採番する
- seed の `upsert_all` では callback が走らないため、`db/seeds.rb` 側で明示投入する
- 外部 URL では次を使う
  - Project: `code`
  - Document: Project 内の `slug`
  - DocumentVersion: `public_id`
  - DocumentFile: `public_id`
- 管理画面でも、外部公開 URL と同じく `public_id` / `code` / `slug` を優先して使う

## ユーザー種別

### internal

- 社内ユーザー
- 全案件・全文書を閲覧可能
- 管理画面の操作権限は持たない
- draft / published / archived の各 DocumentVersion を閲覧可能

### admin

- システム管理者
- 管理画面へアクセス可能
- 以下を CRUD 可能とする
  - 会社
  - ユーザー
  - 案件
  - 案件所属
  - 文書
  - 文書権限
- 全案件・全文書を閲覧可能

### company_master_admin

- 会社マスタ管理者
- 自社のユーザーのメンテナンスを行える
- 自社の会社マスタのメンテナンスを行える
- 他社の会社マスタ、他社ユーザー、案件、文書、文書権限は管理できない
- 文書閲覧権限は `external` と同じ制約に従う

### external

- 社外ユーザー
- 自身が所属する会社・案件・文書権限に基づき、許可された文書のみ閲覧可能
- 管理者相当の概念を追加しても、他社の会社マスタにはアクセスさせない

## 認証

- セッション認証を使う
- 未ログイン時はログイン画面へリダイレクトする
- ナビゲーション右上からログアウトできる
- ログイン成功時に `users.last_login_at` を更新する
- ログイン済みユーザーは自身の access request 一覧を参照できる

## 画面

- ブランド名クリックで TOP に戻る
- 管理画面は `admin` または `company_master_admin` がアクセス可能とする
- `admin` は全管理機能を利用できる
- `admin` は Project 編集画面から標準文書テンプレートの preview と適用を実行できる
- `admin` は案件単位の文書利用状況レポートを参照できる
- `company_master_admin` は自社の会社マスタ・自社ユーザー管理のみ利用できる
- 文書詳細では `tree_view` gem を使って「案件 > 文書」の左ペインツリーを表示する
- Project 配下で文書カタログ一覧・詳細を表示できる
- 管理画面には、主要 model の件数・最近の record・既存 CRUD への入口を横断表示する `model browser` を持つ
- 403 / 404 / 400 は利用者向けエラー画面で表示し、平文レスポンスにはしない
- `external` / `company_master_admin` が参照できる Project は、`project_memberships` がある案件だけとする
- 案件詳細・文書一覧でも同じ Project 参照制約を適用する

## 利用者ダッシュボード

- ログイン後に `GET /dashboard` で利用者向けダッシュボードを表示できる
- ダッシュボードには、自分が閲覧可能な案件、お気に入り、後で読む、最近見た文書、最近更新された文書を表示する
- ダッシュボードには、案件数・文書数・保存ショートカット数・保留中申請数などの workspace summary を表示する
- お気に入りと後で読むは `DocumentBookmark` を利用する
- 最近見た文書は、自分の `AccessLog` の document view 履歴から集計する
- 最近更新された文書は、viewer が参照可能な `Document` だけを表示する
- 権限外文書は bookmark や access log に存在しても表示しない

## 文書 visibility_policy

### internal_only

- 社内ユーザーおよび admin のみ閲覧可能
- external / company_master_admin は DocumentPermission があっても閲覧不可

### restricted_external

- 社内ユーザーおよび admin は閲覧可能
- external / company_master_admin は、ProjectMembership と DocumentPermission がある場合のみ閲覧可能

### public_with_login

- 社内ユーザーおよび admin は閲覧可能
- external / company_master_admin は、ProjectMembership がある場合に閲覧可能
- DocumentPermission は不要
- ただし添付ファイル download は別判定とし、`DocumentPermission.download` がない限り取得できない

## Data Classification

- `visibility_policy` と `data_classification_tags` は分けて扱う
- 閲覧可否・download 可否・社外露出可否は `visibility_policy` / ProjectMembership / DocumentPermission で判定する
- `data_classification_tags` は権限判定そのものには使わず、取り扱い補助のためのタグとして使う
- 初期対象は Document 単位のみとし、DocumentVersion / DocumentFile / DocumentSet / Catalog 単位のタグは後続とする
- 初期のタグ用途は badge 表示、注意表示、出力前確認、送信前確認、透かし、監査ログ補助、AI export 前確認とする
- タグは固定 enum ではなく tag master で管理し、Document と多対多で紐づける前提とする
- 代表的なタグ候補は `external_shareable`, `confidential`, `personal_data`, `contract_related`, `customer_provided`, `submitted` とする
- `public` や `internal_only` のような公開範囲概念は分類タグではなく `visibility_policy` 側で表現する
- import dry-run では `DocumentClassificationSuggester` により source path / file extension / frontmatter から category, document_kind, visibility_policy, snapshot_kind を推定できる
- 初期の自動判定ルールは `config/document_classification_rules.yml` で管理する
- frontmatter がある場合は、rule / extension 推定より frontmatter を優先する
- import dry-run の結果には `matched_rules` を含め、どの rule が推定に使われたかを確認できる

## DocumentPermission

### view

- Docusaurus HTML の閲覧が可能
- 添付ファイルは取得不可

### download

- Docusaurus HTML の閲覧が可能
- 添付ファイルを取得可能
- `download` は `view` を包含する

## Docusaurus viewer

- `GET /projects/:project_code/site/*site_path` および `GET /document_versions/:public_id/site/*site_path` の HTML 応答は、初期表示では viewer shell を返す
- viewer shell は Rails 側の header / breadcrumb / action を持ち、本文部分は same-origin iframe で読み込む
- iframe 側の本文は `embedded=1` 付き同 route を使って取得する
- iframe 側では Docusaurus navbar / footer / toc / sidebar を除去し、本文を中央寄せで表示する
- iframe 側で rewrite される内部 link / asset URL も `embedded=1` を維持し、本文遷移が shell を壊さないようにする

## AccessRequest

- `AccessRequest` は `Project` `Document` `DocumentFile` を対象にできる
- 利用者は自分の申請一覧を参照できる
- 利用者は pending な自分の申請を取り消せる
- 初回の利用者向け導線は、download 不可の文書/添付に対する download 申請を対象とする
- admin は管理画面から申請一覧を参照し、pending 申請を承認または却下できる

## DocumentApprovalRequest

- `DocumentApprovalRequest` は `Document` 単位の最小確認依頼である
- requester は対象 `Document` を閲覧できる active user とする
- approver は任意指定とし、初期実装では internal user を想定する
- status は `pending`, `approved`, `cancelled` を持つ
- `approved` / `cancelled` 時は `acted_by` と時刻を記録する
- 文書詳細では確認依頼の作成導線を表示し、internal user には文書単位一覧への導線も表示する
- internal user は確認依頼一覧・詳細を参照でき、OK / Cancel を実行できる
- requester は自分が作成した確認依頼の詳細を参照できる

## 危険操作の安全装置

- 初期実装では、既存の危険操作に対して confirm と影響表示を優先する
- admin 画面の delete 操作は `turbo_confirm` 付きで実行し、対象名と影響概要を確認文言に含める
- 標準テンプレート適用のような複数文書へ影響する操作では、実行前に作成予定件数 / スキップ予定件数を表示し、confirm を要求する
- Project 権限 preview や external preview のような変更前確認導線は、適用処理から分離した dry-run / preview として扱う
- 未実装の mail / shared link / bulk edit / archive などは、将来追加時も同じく confirm と影響表示を必須にする

## DocumentReviewComment

- `DocumentReviewComment` は internal review と公開 Q&A の共通 comment thread として使う
- コメント対象は `Document` と任意の `DocumentVersion` とする
- comment type は `note` `issue` `request_change` `question` を持つ
- status は `open` `resolved` `rejected` を持つ
- `parent_id` による親コメント参照を持ち、将来の thread/Q&A 拡張余地を残す
- text 系文書向けの補助情報として `text_line_start`, `text_line_end`, `text_anchor_type`, `text_anchor_path`, `text_anchor_label`, `source_path` を持てる
- `internal_only = true` は社内レビュー、`false` は公開 Q&A thread として扱う
- internal user は文書詳細または版詳細から、Document 全体コメントと DocumentVersion コメントを登録できる
- admin は解決・却下の状態変更ができる
- external user には `internal_only = true` コメントを表示しない

## 社外向けQ&A

- 初期実装では Q&A 専用 model を作らず、`DocumentReviewComment` の public thread を再利用する
- root comment が `comment_type = question` かつ `internal_only = false` のものを Q&A thread として扱う
- reply は同じ `DocumentReviewComment` の `parent_id` で表現し、public thread への返信は `internal_only = false` に揃える
- external / company_master_admin / internal は、閲覧可能な文書または版の detail 画面から Q&A を投稿できる
- internal user / admin は public Q&A thread に返信できる
- 文書詳細と版詳細では、回答済み表示を `resolved => 回答済み`, `rejected => クローズ`, `open => 受付中` として扱う
- internal review 用の `internal_only = true` コメントは、従来どおり external user には表示しない
- 初期実装の公開範囲は thread 単位の `internal_only` だけで制御し、`company_visible` / `project_visible` などの細分化は後続とする

## DocumentVersion 品質チェック

- `DocumentVersionQualityChecker` により、版単位の品質チェックを都度計算できる
- 初期導線は `GET /document_versions/:public_id/quality_check` とし、`DocumentVersion` 詳細から辿れる
- 品質チェック詳細は internal user / admin だけ参照できる
- 初期チェック結果は `error` `warning` `info` を持つ
- 初期対象には title/category/document kind/visibility/source path/rendered site/document file 存在と scan 状態/internal-only wording を含める
- Markdown 原本が添付されている版では、相対 Markdown link / image / attachment 参照切れも error として検出する
- Docusaurus build 成果物の entry が見つからない場合は、build failure 相当の error として扱う
- 同じ結果は HTML に加えて JSON と Markdown でも取得できる

## 文書重要度

- `Document` は `importance_level` と `recommended_sort_order` を持つ
- `importance_level` は `critical`, `important`, `normal`, `reference`, `deprecated` を持つ
- Project 詳細と文書一覧では、文書を `importance_level`, `recommended_sort_order`, `title`, `id` の順で優先表示する
- Project 詳細では `critical` / `important` な文書を「優先して確認したい資料」として先頭導線に出す
- 文書詳細では現在の重要度を表示する
- Catalog / DocumentSet 内の表示順は、それぞれ item の `sort_order` で制御する

## 文書一括編集 dry-run

- 初期実装では専用 UI より先に、文書一括編集の logic 層と保存付き dry-run を提供する
- 保存モデルは `BulkEditDryRun` とし、`operation_type`, `target_document_ids`, `params_json`, `summary_json`, `result_json`, `warnings_json`, `errors_json`, `created_by`, `confirmed_by`, `confirmed_at`, `expires_at` を持つ
- 初期の操作種別は `document_metadata` のみとする
- dry-run / 実行ロジックは `DocumentBulkEditPlan`, `DocumentBulkEditPreview`, `DocumentBulkEditExecutor` で扱う
- 初期対象は、現行 schema にある項目に限定する
  - `Document`: `category`, `document_kind`, `visibility_policy`, `importance_level`, `recommended_sort_order`, `retention_until`, `discard_candidate_at`
  - `Document.latest_version`: `snapshot_kind`, `published_from`, `published_until`
  - `document_tags` の追加 / 削除
  - archive / restore
- 最新版を持たない `Document` に対して version 属性変更を要求した場合は dry-run error とする
- `published_from` / `published_until` は dry-run 時点で整合性を検証し、逆転している場合は error とする
- archive / restore は dry-run 上で通常一覧や社外導線への影響を確認できる前提とする
- 初期実装では `data_classification_tags` 専用の物理モデルが未導入のため、分類タグの一括変更は対象外とする
- 初期実装では dry-run と execute の service / spec を先に整備し、管理画面や検索結果からの選択 UI は後続 issue で接続する

## Import dry-run

- `ImportDryRunValidator` は source path 妥当性、既存 Document 更新候補、分類推定、重複候補を dry-run 結果として返す
- dry-run の出力形式は hash / markdown presenter で共通利用できる
- `ImportManifestDryRun` は manifest 内の document 群を project ごとに束ね、internal import API の validation mode で共通 dry-run 結果を返す
- internal import API `POST /api/internal/doc_imports` は `validate_only=true` で dry-run を保存し、`dry_run_id`, summary, items, warnings, errors を JSON で返す
- internal ZIP import API `POST /api/internal/zip_imports` は `zip_file`, `project_code` を受け取り、ZIP を `storage/imports/zip_uploads/...` に安全に展開したうえで manifest を自動生成し、`validate_only=true` で dry-run を保存できる
- ZIP upload の preview には `zip_import_preview` として `orphan_files`, `skipped_files`, `warnings` を含める
- ZIP upload の初期実装では、Markdown / MDX / markdown 拡張子の文書、および standalone diagram (`.puml`, `.plantuml`, `.d2`, `.mmd`, `.mermaid`) を import 対象にする
- ZIP upload の本実行は保存済み `ImportDryRun(import_mode=zip)` の `dry_run_id` を指定して行い、validate 時に生成した staged artifact / manifest をそのまま再利用する
- 保存結果は `ImportDryRun` として保持し、初期実装では `import_mode`, `project`, `source_commit_hash`, summary/result/warnings/errors, `created_by`, `confirmed_by`, `confirmed_at` を持つ
- 同 API は `import_dry_run_id` を指定した本実行も受け付け、保存済み dry-run の `source_commit_hash` と今回 manifest の commit hash が一致する場合に `confirmed` として import 実行に進める
- 既存 Document の新 Version 候補は、同一 source path または十分に狭いコンテキストの既存文書一致で判定する
- 類似候補は `same_title`, `same_source_relative_path`, `same_source_basename` の理由で表示できる
- 類似候補は preview に表示するが、自動統合しない
- dry-run では `matched_rules` に加えて重複候補の理由と既存文書一覧を確認できる
- 初期実装で保存付き dry-run が有効なのは internal push/import API と internal ZIP upload API であり、Git pull / manual upload の UI 導線は後続 issue で同じ result 形式に寄せる

## 文書カタログ

- `DocumentCatalog` は Project 配下の curated な文書一覧である
- route は `projects/:project_code/document_catalogs/:public_id` を使う
- 一覧では viewer が参照可能な catalog だけ表示する
- 詳細では viewer が参照可能な item だけ表示する
- `internal_only` catalog は社外ユーザーへ表示しない

## 文書セット

- `DocumentSet` は Project 配下の用途別 document grouping である
- route は `projects/:project_code/document_sets/:public_id` を使う
- `admin` は管理画面から `DocumentSet` を作成・編集・削除できる
- `DocumentSetItem` には `Document` と任意の固定 `DocumentVersion` を設定できる
- fixed version を指定しない item は、その `Document` の `latest_version` を使う
- 文書セット詳細では item の `sort_order` 順に文書を表示する
- external / company_master_admin には、set 自体が見えても viewer が参照可能な item だけを表示する
- `internal_only` set は社外ユーザーへ表示しない

## 権限変更 dry-run

- 管理画面では、Project 単位の権限変更を適用する前に preview を取得できる
- preview は admin 限定の `GET /admin/projects/:code/permission_preview` とする
- 対象 viewer は `user_ids[]` と `company_ids[]` で指定できる
- preview は次の差分を返す
  - viewer ごとの閲覧可能文書の増減
  - viewer ごとのダウンロード可能文書の増減
  - company ごとの影響人数集計
- ProjectMembership 付与/剥奪と DocumentPermission 相当の付与/剥奪を、変更前に dry-run で確認できる
- `internal_only` 文書は ProjectMembership を付与しても社外ユーザーの gained document に含めない

## 外部表示プレビュー

- 管理画面では、Project 単位で社外ユーザー視点の表示 preview を確認できる
- 初期導線は `admin/projects/:code/external_preview` とし、案件編集画面から遷移できる
- user 指定 preview は、指定した active な `external` / `company_master_admin` を viewer として既存 `viewable_by?` / `downloadable_by?` 判定を流用する
- company 指定 preview は、その会社に所属する active な `external` / `company_master_admin` を一覧し、viewer ごとの差分を同じ画面で確認する
- preview では文書の表示/非表示、添付ファイルの download 可否を確認できる
- preview 実行時は `AccessLog` に `target_type = external_preview` として監査ログを残す
- external user 自身は preview 機能を利用できない

## AI向けコンテキスト生成

- Project 単位で AI 向けコンテキストを生成できる
- 初期導線は `GET /projects/:code/ai_context` とし、案件詳細から辿れる
- 出力は viewer の既存閲覧権限に従い、権限外文書は export 本文に含めない
- 初期 mode は `compact` と `full` を持つ
- 同じ内容を HTML preview / JSON / Markdown で取得できる
- AI 向けコンテキスト表示・export は `AccessLog` に `target_type = ai_context` として残す

## 外部送付履歴

- 初期実装では `DocumentDeliveryLog` を portal link 送付の下書き・監査ログとして使う
- 対象は `Document` または `DocumentSet` とし、sender はその対象を閲覧できる user に限る
- 初期 `delivery_type` は `portal_link` を主対象とし、shared link / attachment / zip attachment は後続とする
- 文書詳細と文書セット詳細から送付下書きを作成できる
- 下書きでは `to/cc/bcc`, `subject`, `body` を保存し、本文初期値には portal URL を含める
- 保存後は `mailto:` で既定メーラーを開ける
- 送付履歴は `draft`, `sent`, `failed` を持ち、初期実装では利用者が手動で `sent` / `failed` を記録する
- internal user は全送付履歴を参照でき、external / company_master_admin は自分が sender の履歴だけ参照できる
- 宛先文字列は保存時に正規化し、重複除去したうえで監査用に保持する

## DocumentVersion の状態

### draft

- 社内ユーザーおよび admin のみ閲覧可能
- external / company_master_admin には表示しない
- `latest_version` にはしない

### published

- 社外公開対象
- 権限条件を満たす external / company_master_admin に表示する
- `published_from` / `published_until` がある場合、社外ユーザーはその期間内だけ表示する
- `latest_version` 候補になる

### archived

- `DocumentVersion.status = archived` は版単位の履歴状態として扱う
- `Document` 単位の archive は `documents.archived_at` を正本とする
- archived Document は通常一覧・案件詳細・文書一覧・左ツリー・検索結果から除外する
- external / company_master_admin だけでなく internal user の通常導線からも除外する
- archived Document の復元と状態確認は admin 管理画面から行う
- archived Document には `archived_by_user`, `retention_until`, `discard_candidate_at` を保持できる
- archive / restore 時は `NotificationEvent` に監査用イベントを残す
- archive 操作時には、操作画面で「通常一覧・社外導線から非表示になる」ことを確認させる

## latest_version

- バージョン管理ありの Document では、`published` な DocumentVersion のうち、`version_label` の semantic version 比較で最も新しいものを `latest_version` とする
- `draft` / `archived` は `latest_version` にしない
- `version_label` は semantic version として比較可能な形式を基本とする
- バージョン管理しない Document では、同一 Document を常に上書きして最新として扱う
- external sample seed では `current` 版を優先して `latest_version` にし、`current` がなければ取り込まれた公開版の中から 1 つを採用する
- 公開予約・公開期限の初期実装は `DocumentPublication` 別モデルではなく `DocumentVersion.published_from / published_until` を正本として扱う
- 公開可否は `published?` と publication window の動的判定で制御し、期限切れ時に status を書き換える job は持たない
- external / company_master_admin の案件一覧では、現在表示可能な文書が 1 件以上ある Project だけ表示する
- internal / admin は期間外の版も閲覧でき、版詳細で `not_started` / `active` / `expired` の公開状態を確認できる

## 文書表示

- 文書管理・権限・検索・コメント・送付・監査ログの正本は Rails 側に置く
- 表示方法は文書種別ごとの viewer adapter として扱い、Docusaurus は Markdown / MDX 向け adapter と位置づける
- Markdown 公開物は Docusaurus build を版単位で保存する
- Rails 側では Docusaurus HTML の内部リンクと静的 asset の参照先を Project 単位の `/projects/:id/site/...` へ張り替えて配信する
- Document 詳細の「HTMLで表示」導線は残すが、遷移先は Project 単位の site route とする
- Rails 画面内に Docusaurus 成果物を組み込む場合は iframe viewer を優先し、Rails 側の metadata / コメント / 版履歴 / download 導線を正とする
- Docusaurus viewer では本文レンダリングを主目的とし、navbar / sidebar / breadcrumbs / footer / 前後ナビは最小化する方針とする
- Project 詳細・文書一覧・文書詳細では、共通の TreeView で「案件 > 文書」の階層をたどれるようにする
- 左ペイン、パンくず、標準ドキュメントレイアウトは Docusaurus 標準の見た目を利用する
- Docusaurus HTML には Rails 側の Project 詳細へ戻る導線を入れ、対応する Document を解決できるページでは Document 詳細へ戻る導線も入れる
- portal 経由で配信する Docusaurus HTML には viewer 用 CSS を追加し、本文幅、見出し、段落、表、コードブロック、blockquote、画像の見た目を整える
- portal 経由では navbar / sidebar / breadcrumbs / footer / 前後ナビ / edit this page 等の周辺UIは非表示または最小化する
- text / markdown / json / yaml などの text 系添付は、`disposition=inline` 時に Rails 側の行番号付き preview 画面で表示する
- Rails 側 text preview は固定幅フォント、横スクロール、行番号、download 導線を持つ
- `DOCUSAURUS_EDIT_URL` を build 時に与えた場合は、`Edit this page` を表示できる
- 文書詳細の「表示」リンクは、対応する Docusaurus 生成 HTML が実在する版にだけ表示する
- external / company_master_admin の文書一覧、文書詳細、AI export、添付 download では、`latest_version` が公開期間外の文書/版を表示しない
- 同じ site build を共有する複数の DocumentVersion がある場合は、各版の `storage/docs_sites/<version.id>/...` に同じ build 成果物を配置する
- Markdown 添付は Docusaurus 画面とは別系統の生ファイル配信とする
- JSON / YAML / plain text などのテキスト系ファイルは Rails 側 preview で扱い、行番号や将来のコメント位置情報と連携できる余地を残す
- PDF / 画像は Rails 側 preview を基本とし、Office 等は必要に応じて外部 viewer adapter を追加できる設計にする
- text 系の添付ファイルは UTF-8 charset 付きで返す
- 添付ファイルおよび zip download の `Content-Disposition` は UTF-8 file name を返せる形式にする
- 添付ファイル配信は `download` 権限を要求し、`view` 権限だけでは取得できない
- Project/Version の zip 出力では、`zip_path_mode`, `include_markdown_sources`, `include_attachments`, `pdf_only` の出力オプションを受け付ける
- `zip_path_mode = source_path` の場合は source path 準拠の zip 内パスを使い、それ以外は文書 slug / version をベースにした zip 内パスを使う
- zip 出力には README を同梱し、適用された出力オプションと、PDF が含まれる場合の watermark metadata を記録する
- PDF 系出力の watermark metadata は `ExportOutputPlan` を正本とし、初期値には classification label, company name, user email, project code, document public_id, exported_at を含める
- 文書詳細 / 版詳細の HTML 画面では透かし自体は入れず、タグと「出力時の扱い」説明、および PDF がある場合の watermark metadata 例を表示する
- Docusaurus build では Mermaid fenced code block を図として扱えるようにする
- PlantUML / D2 fenced code block は Kroki 連携で build 時に SVG 化する
- Kroki 連携先は `KROKI_ENDPOINT` で指定し、未設定のまま PlantUML / D2 図を含む build を実行した場合は失敗させる
- 表示 adapter の候補としては `docusaurus_iframe`, `rails_text`, `rails_pdf`, `rails_image`, `external_iframe`, `download_only` を想定し、viewer 選択は拡張子・mime type・変換可否で判定する

## ドキュメントサイト権限フィルタ

- 主目的は、権限のない利用者へ本文 HTML を返さないこととする
- Project site への入口は ProjectMembership または社内権限で判定する
- HTML page 配信時は、site path から対応する DocumentVersion / Document を解決できる場合、その文書閲覧権限も確認する
- `public_with_login` の本文 HTML は、DocumentPermission がなくても ProjectMembership があれば閲覧できる
- 権限のない Document の URL を直打ちされた場合は 403 または 404 とする
- Docusaurus HTML の sidebar / nav / breadcrumbs など通常 UI のリンクから、権限のない Document への導線を取り除く
- JS / CSS / image asset は Document 単位では止めず、Project site へ入れる利用者には配信する
- DevTools 等から JS bundle / client data / metadata を直接確認した場合に、ページタイトルや slug 程度が見える可能性は当面許容する
- metadata まで厳密に秘匿する必要が出た場合は、利用者権限ごとの site build 分離を別途検討する

## import

- internal import API が読み取るローカルパスは `storage/imports/` 配下に限定する
- import manifest の添付ファイル `storage_key` は相対パスのみ許可し、絶対パスや path traversal は拒否する
- 同一 Document の同一 `version_label` は immutable とし、再 import は失敗させる
- 修正する場合は新しい `version_label` を発行する
- バージョン管理しない Document では、同一 Document を上書きできる

## seed データ

- seed 用 CSV は `db/seeds/data/` に置く
- seed 補助クラスは `db/seeds/support/` に置く
- `db/seeds.rb` は `upsert_all` ベースで投入し、初期化前提の `delete_all` は行わない
- `upsert_all` は callback を通さないため、seed では `public_id` を明示的に投入する
- sample 文書ファイルは `storage/document_files/external_samples/` 配下を正本として扱う
- 配置規約は `storage/document_files/external_samples/<sample-set>/<site-dir>/...` とする
- `sample-set` と `site-dir` の命名は利用者に委ねる
- seed 時には `site-dir` ごとに 1 Project を生成する
- `site-dir` 直下およびその配下の Markdown から、ページ単位の Document を生成する
- ファイル名または論理パスが異なる Markdown / 資料ファイルは、同じ資料の別版らしく見える場合でも原則として別 Document として扱う
- 同じ Document の別 DocumentVersion として扱うのは、同じ論理資料として明示的に版追加される場合に限る
- `README.md` はそのディレクトリのトップページとして扱い、非 `README` Markdown はファイル単位で別 Document として扱う
- `編集正本` `編集正本PDF化済` `提出済` などのスナップショット用ディレクトリは、同じ Document の別 `DocumentVersion` として扱う
- seed 上の外部サンプル走査ロジックは、将来の import / controller / UI 経由追加でも共有するべき Document / DocumentVersion 判定ルールのプロトタイプとして扱う
- Markdown がある場合は seed 時に Docusaurus build を行い、対応する `DocumentVersion.site_build_path` と `storage/docs_sites/<version.id>/...` を生成する
- seed 時の `DocumentVersion.markdown_entry_path` は source Markdown path の推測値ではなく、Docusaurus build 後の実 HTML 出力パスから確定する
- seed 用 Markdown は build 成功を必須とし、失敗した場合は seed 全体を失敗させる
- seed 時に external sample の Markdown を Docusaurus build する場合、元 Markdown は一時 workspace にコピーして build する
- Docusaurus の doc id 衝突を避けるため、一時コピーには source relative path から生成した一意な `id` front matter を付与する
- 元ファイルは変更しない

## AccessLog

- Docusaurus HTML ページ閲覧を `view` として記録する
- DocumentFile の取得を `download` として記録する
- JS / CSS / image などの Docusaurus asset 配信は記録しない
- 案件一覧・文書詳細画面・管理画面の閲覧は現時点では記録しない
- ログイン日時は AccessLog ではなく `users.last_login_at` で管理する

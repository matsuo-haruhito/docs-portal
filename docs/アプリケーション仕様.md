# アプリケーション仕様

## 概要

- 本アプリは、案件ごとの文書公開、版管理、閲覧権限制御、添付ファイル配布を行う Rails ポータルである
- Markdown 文書は Docusaurus build を公開物として扱い、Rails 側では閲覧権限を付けたうえで配信する
- DB は単一構成とし、案件・会社・文書権限による論理分離を行う
- 社内ユーザーは案件横断で文書を閲覧できる

## 識別子

- DB 内部の主キー `id` は、外部公開 URL には原則使用しない
- アプリケーション管理テーブルには `public_id` を持たせる
- `public_id` は作成時に自動生成し、原則変更しない
- `public_id` は URL-safe な文字列とし、テーブルごとに prefix を付ける
- 通常の `create!` / `save!` 経路では model concern で自動採番する
- seed の `upsert_all` では callback が走らないため、`db/seeds.rb` 側で明示投入する
- 外部 URL では次を使う
  - Project: `code`
  - Document: Project 内の `slug`
  - DocumentVersion: `public_id`
  - DocumentFile: `public_id`
- 管理画面でも、外部公開 URL と同じく `public_id` / `code` / `slug` を優先して使う

## ユーザー種別

### internal

- 社内ユーザー
- 全案件・全文書を閲覧可能
- 管理画面の操作権限は持たない
- draft / published / archived の各 DocumentVersion を閲覧可能

### admin

- システム管理者
- 管理画面へアクセス可能
- 以下を CRUD 可能とする
  - 会社
  - ユーザー
  - 案件
  - 案件所属
  - 文書
  - 文書権限
- 全案件・全文書を閲覧可能

### company_master_admin

- 会社マスタ管理者
- 自社のユーザーのメンテナンスを行える
- 自社の会社マスタのメンテナンスを行える
- 他社の会社マスタ、他社ユーザー、案件、文書、文書権限は管理できない
- 文書閲覧権限は `external` と同じ制約に従う

### external

- 社外ユーザー
- 自身が所属する会社・案件・文書権限に基づき、許可された文書のみ閲覧可能
- 管理者相当の概念を追加しても、他社の会社マスタにはアクセスさせない

## 認証

- セッション認証を使う
- 未ログイン時はログイン画面へリダイレクトする
- ナビゲーション右上からログアウトできる
- ログイン成功時に `users.last_login_at` を更新する

## 画面

- ブランド名クリックで TOP に戻る
- 管理画面は `admin` または `company_master_admin` がアクセス可能とする
- `admin` は全管理機能を利用できる
- `company_master_admin` は自社の会社マスタ・自社ユーザー管理のみ利用できる
- 文書詳細では `tree_view` gem を使って「案件 > 文書」の左ペインツリーを表示する
- `external` / `company_master_admin` が参照できる Project は、`project_memberships` がある案件だけとする
- 案件詳細・文書一覧でも同じ Project 参照制約を適用する

## 文書 visibility_policy

### internal_only

- 社内ユーザーおよび admin のみ閲覧可能
- external / company_master_admin は DocumentPermission があっても閲覧不可

### restricted_external

- 社内ユーザーおよび admin は閲覧可能
- external / company_master_admin は、ProjectMembership と DocumentPermission がある場合のみ閲覧可能

### public_with_login

- 社内ユーザーおよび admin は閲覧可能
- external / company_master_admin は、ProjectMembership がある場合に閲覧可能
- DocumentPermission は不要
- ただし添付ファイル download は別判定とし、`DocumentPermission.download` がない限り取得できない

## DocumentPermission

### view

- Docusaurus HTML の閲覧が可能
- 添付ファイルは取得不可

### download

- Docusaurus HTML の閲覧が可能
- 添付ファイルを取得可能
- `download` は `view` を包含する

## DocumentVersion の状態

### draft

- 社内ユーザーおよび admin のみ閲覧可能
- external / company_master_admin には表示しない
- `latest_version` にはしない

### published

- 社外公開対象
- 権限条件を満たす external / company_master_admin に表示する
- `latest_version` 候補になる

### archived

- 論理削除と同じ扱いとする
- external / company_master_admin には表示しない
- 社内ユーザーおよび admin は閲覧可能
- `latest_version` にはしない
- archive 操作時には、操作画面に次の警告を表示する
  - 「アーカイブ操作をすると、本システムの提供元に依頼しなければ復元できません」
- 復元操作は社内から実行できる必要がある

## latest_version

- バージョン管理ありの Document では、`published` な DocumentVersion のうち、`version_label` の semantic version 比較で最も新しいものを `latest_version` とする
- `draft` / `archived` は `latest_version` にしない
- `version_label` は semantic version として比較可能な形式を基本とする
- バージョン管理しない Document では、同一 Document を常に上書きして最新として扱う
- external sample seed では `current` 版を優先して `latest_version` にし、`current` がなければ取り込まれた公開版の中から 1 つを採用する

## 文書表示

- Markdown 公開物は Docusaurus build を版単位で保存する
- Rails 側では Docusaurus HTML の内部リンクと静的 asset の参照先を Project 単位の `/projects/:id/site/...` へ張り替えて配信する
- Document 詳細の「HTMLで表示」導線は残すが、遷移先は Project 単位の site route とする
- Project 詳細・文書一覧・文書詳細では、共通の TreeView で「案件 > 文書」の階層をたどれるようにする
- 左ペイン、パンくず、標準ドキュメントレイアウトは Docusaurus 標準の見た目を利用する
- Docusaurus HTML には Rails 側の Project 詳細へ戻る導線を入れ、対応する Document を解決できるページでは Document 詳細へ戻る導線も入れる
- `DOCUSAURUS_EDIT_URL` を build 時に与えた場合は、`Edit this page` を表示できる
- 文書詳細の「表示」リンクは、対応する Docusaurus 生成 HTML が実在する版にだけ表示する
- 同じ site build を共有する複数の DocumentVersion がある場合は、各版の `storage/docs_sites/<version.id>/...` に同じ build 成果物を配置する
- Markdown 添付は Docusaurus 画面とは別系統の生ファイル配信とする
- text 系の添付ファイルは UTF-8 charset 付きで返す
- 添付ファイル配信は `download` 権限を要求し、`view` 権限だけでは取得できない
- Docusaurus build では Mermaid fenced code block を図として扱えるようにする
- PlantUML / D2 fenced code block は Kroki 連携で build 時に SVG 化する
- Kroki 連携先は `KROKI_ENDPOINT` で指定し、未設定のまま PlantUML / D2 図を含む build を実行した場合は失敗させる

## ドキュメントサイト権限フィルタ

- 主目的は、権限のない利用者へ本文 HTML を返さないこととする
- Project site への入口は ProjectMembership または社内権限で判定する
- HTML page 配信時は、site path から対応する DocumentVersion / Document を解決できる場合、その文書閲覧権限も確認する
- `public_with_login` の本文 HTML は、DocumentPermission がなくても ProjectMembership があれば閲覧できる
- 権限のない Document の URL を直打ちされた場合は 403 または 404 とする
- Docusaurus HTML の sidebar / nav / breadcrumbs など通常 UI のリンクから、権限のない Document への導線を取り除く
- JS / CSS / image asset は Document 単位では止めず、Project site へ入れる利用者には配信する
- DevTools 等から JS bundle / client data / metadata を直接確認した場合に、ページタイトルや slug 程度が見える可能性は当面許容する
- metadata まで厳密に秘匿する必要が出た場合は、利用者権限ごとの site build 分離を別途検討する

## import

- internal import API が読み取るローカルパスは `storage/imports/` 配下に限定する
- import manifest の添付ファイル `storage_key` は相対パスのみ許可し、絶対パスや path traversal は拒否する
- 同一 Document の同一 `version_label` は immutable とし、再 import は失敗させる
- 修正する場合は新しい `version_label` を発行する
- バージョン管理しない Document では、同一 Document を上書きできる

## seed データ

- seed 用 CSV は `db/seeds/data/` に置く
- seed 補助クラスは `db/seeds/support/` に置く
- `db/seeds.rb` は `upsert_all` ベースで投入し、初期化前提の `delete_all` は行わない
- `upsert_all` は callback を通さないため、seed では `public_id` を明示的に投入する
- sample 文書ファイルは `storage/document_files/external_samples/` 配下を正本として扱う
- 配置規約は `storage/document_files/external_samples/<sample-set>/<site-dir>/...` とする
- `sample-set` と `site-dir` の命名は利用者に委ねる
- seed 時には `site-dir` ごとに 1 Project を生成する
- `site-dir` 直下およびその配下の Markdown から、ページ単位の Document を生成する
- ファイル名または論理パスが異なる Markdown / 資料ファイルは、同じ資料の別版らしく見える場合でも原則として別 Document として扱う
- 同じ Document の別 DocumentVersion として扱うのは、同じ論理資料として明示的に版追加される場合に限る
- `README.md` はそのディレクトリのトップページとして扱い、非 `README` Markdown はファイル単位で別 Document として扱う
- `編集正本` `編集正本PDF化済` `提出済` などのスナップショット用ディレクトリは、同じ Document の別 `DocumentVersion` として扱う
- seed 上の外部サンプル走査ロジックは、将来の import / controller / UI 経由追加でも共有するべき Document / DocumentVersion 判定ルールのプロトタイプとして扱う
- Markdown がある場合は seed 時に Docusaurus build を行い、対応する `DocumentVersion.site_build_path` と `storage/docs_sites/<version.id>/...` を生成する
- seed 時の `DocumentVersion.markdown_entry_path` は source Markdown path の推測値ではなく、Docusaurus build 後の実 HTML 出力パスから確定する
- seed 用 Markdown は build 成功を必須とし、失敗した場合は seed 全体を失敗させる
- seed 時に external sample の Markdown を Docusaurus build する場合、元 Markdown は一時 workspace にコピーして build する
- Docusaurus の doc id 衝突を避けるため、一時コピーには source relative path から生成した一意な `id` front matter を付与する
- 元ファイルは変更しない

## AccessLog

- Docusaurus HTML ページ閲覧を `view` として記録する
- DocumentFile の取得を `download` として記録する
- JS / CSS / image などの Docusaurus asset 配信は記録しない
- 案件一覧・文書詳細画面・管理画面の閲覧は現時点では記録しない
- ログイン日時は AccessLog ではなく `users.last_login_at` で管理する

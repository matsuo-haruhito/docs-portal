# アクセス申請・同意管理・Webhook運用runbook

この runbook は、管理画面の `アクセス申請` `同意文面` `案件同意設定` `Webhook` を日常運用で見直すときの入口をまとめる。

新しい運用ポリシーはここでは定義しない。current 実装と既存仕様 docs を前提に、どの画面で何を確認するか、変更時にどこへ戻るかを整理する。

## 先に見るもの

1. 利用者向けの同意仕様を確認したいときは [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)
2. 外部通知の payload や署名仕様を確認したいときは [Webhook・外部API連携方針](./Webhook・外部API連携方針.md)
3. 管理画面の位置関係は `app/views/admin/_nav.html.slim` の `アクセス申請` `同意文面` `案件同意設定` `Webhook` を起点に確認する

## 画面ごとの役割

### 1. アクセス申請

`admin/access_requests` は、利用者から届いた閲覧・ダウンロード・管理権限の申請を internal 管理者が裁く画面。

- 一覧には `日時 / 申請者 / 対象 / 要求権限 / 状態 / 理由 / 承認者` が出る
- 対象は `Project` `Document` `DocumentFile` のいずれかで、承認前は `承認` / `却下` ボタンが出る
- 申請の状態は `pending / approved / rejected / cancelled`
- 要求権限は `view / download / manage`
- 一覧上部の `状態` では `すべて / 承認待ち / 承認済み / 却下 / 取消済み` を切り替えられる
- `申請者・対象` 検索では、申請者名・メールアドレス・理由・対象名・案件コード・文書名・ファイル名・公開IDをまとめて検索する

#### requested_access_level と実際の付与先

`要求権限` は利用者が申請した `requested_access_level` であり、承認時に作られる role / permission と 1:1 ではない。current `AccessRequestResolver` の付与先は次の通り。

| 対象 | 承認時の付与先 | current mapping |
| --- | --- | --- |
| `Project` | `ProjectMembership` | 要求権限にかかわらず requester を `viewer` role として所属させる |
| `Document` | `DocumentPermission` | `download` 申請は `download`、それ以外は `view` |
| `DocumentFile` | 親 `Document` の `DocumentPermission` | file 単位の permission は作らず、親 Document へ `download` または `view` を付与する |

- `manage` は enum と一覧表示上の要求値として存在するが、current resolver は明示的な manage grant を持たない
- `manage` 申請を承認しても、Project 管理者 role や Document 管理権限を付与したものとして扱わない
- `manage` の扱いを変えるかどうかは `#1124` / `#1126` の design / feature lane、current mapping の regression guard は `#1127` の quality lane で扱う
- この runbook は current behavior の読み分けを補うだけで、承認基準や新しい権限仕様は定義しない

#### 結果サマリと empty state の見方

- `表示中` は、現在の `状態` filter と `申請者・対象` 検索を適用したあとに一覧へ残っている申請件数
- `状態` は今どの filter を適用しているかを表示する。未選択なら `すべて`
- `検索` は free-text 条件を入れているときだけ表示される
- `承認待ち / 承認済み / 却下 / 取消済み` の件数は、現在表示中の一覧に対する状態別の内訳
- `取消済み` は 1 件以上あるときだけ表示される
- `条件に一致する申請はありません。` は、申請自体は存在するが現在の条件では 0 件になった状態を示す
- `申請はありません。` は、filter を掛けていない状態で申請レコード自体がまだ無いことを示す

日常確認ポイント:

- `承認待ち` で絞り、処理待ちが溜まっていないかを最初に確認する
- `申請者・対象` 検索で、特定案件や文書に申請が集中していないかを見返す
- 申請対象と `requested_access_level` が利用者の想定と合っているか
- すでに処理済みの申請で、承認者が想定どおり記録されているか
- 条件に一致しないときは、`状態` の残り値や検索語の表記揺れを先に疑う

#### 却下理由の定型候補と補足

pending 申請を却下するときは、`却下理由の定型候補` と `補足（任意）` を使う。

- 定型候補は `権限不足 / 対象誤り / 情報不足 / 承認条件不一致` から選ぶ
- `補足（任意）` には、申請者へ伝える追加説明が必要なときだけ短く書く
- 保存される却下理由は既存の `AccessRequest#rejection_reason` で、定型候補と補足が 1 つの理由文字列として残る
- 定型候補は入力補助であり、承認基準、権限付与仕様、却下理由 taxonomy の恒久運用をこの画面で決めるものではない
- 空理由や、未対応の候補だけで補足も空の却下は current controller で受け付けない
- 既存 request spec 用の直接 `rejection_reason` param 互換は残っているが、管理画面の日常操作では定型候補と任意補足を正規の入力導線として読む
- 却下後も current filter は維持されるため、`承認待ち` や検索条件で絞っている場合は、戻った一覧の条件を見直す

変更時の注意:

- 権限モデル自体の正本はアプリ仕様側にあり、この runbook では承認基準を新設しない
- 定型候補を増やす、通知や SLA と結び付ける、回答期限や自動エスカレーションへ広げる場合は、この runbook ではなく別の feature / operations 判断に戻す
- current filter と件数サマリは in-memory の一覧結果を前提にしているため、DB 集計や別 KPI の説明へ広げない

戻り先:

- 権限や公開範囲の意味は [アプリケーション仕様](./アプリケーション仕様.md)
- 同意前提を含む利用者導線は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 2. 同意文面

`admin/consent_terms` は、利用規約・秘密保持・注意事項などの文面そのものを管理する画面。

- `タイトル / 版 / 種別 / 再同意方針 / 状態` を一覧で確認できる
- 新規登録と編集では `title` `version_label` `consent_scope` `requirement_timing` `active` `body` を扱う
- `title` と `version_label` の組み合わせは重複できない
- 利用履歴や設定が残っている文面は削除できないため、通常は `無効` へ切り替えて運用する

#### 0 件から始めるときの最小手順

初回状態では、一覧 table や表示設定ではなく `まだ同意文面はありません。` の empty state が出る。最初の 1 件は上段の `新規登録` フォームから作る。

1. `タイトル` と `版` を決める
2. `種別` と `再同意方針` を current 運用に合わせて選ぶ
3. `本文` を入力し、必要なら `有効` を切り替えて保存する
4. 保存したら empty state は消え、`同意文面一覧` に最初の 1 行が出る。続けて `版` `種別` `状態` が想定どおりかを見る

一覧が 1 件以上あるときだけ `同意文面一覧の表示設定` と table が出るので、初回状態では「最初の文面を作る画面」と読めばよい。

日常確認ポイント:

- 利用中の文面が `有効` のまま残っているか
- 改訂時に `version_label` を上げているか
- `consent_scope` と `requirement_timing` が intended な利用箇所と合っているか

変更時の注意:

- 文面本文の変更を既存版へ上書きすると、どの版に同意したかの追跡が曖昧になりやすい。改訂時は新しい `version_label` を使う
- 法務レビューや文面の妥当性そのものはアプリ外運用で担保する

戻り先:

- モデル責務と enforcement 範囲は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 3. 案件同意設定

`admin/project_consent_settings` は、どの案件でどの同意文面を必須にするかを結び付ける画面。

- `案件 / 同意文面 / 必須タイミング / 状態` を一覧で確認できる
- フォームで選べる文面は `active` な `ConsentTerm` のみ
- 新規登録フォームの `案件` と `同意文面` は検索可能な select で、多件数でも候補名を絞り込みながら選べる
- 一覧上部の filter でも `案件` と `同意文面` は検索可能な select になっており、`状態` filter と合わせて現在の一覧だけを絞り込む
- 一覧 filter の `案件` 候補は `案件名 (案件コード)`、`同意文面` 候補は `タイトル / 版` として読む
- `required_on` は `first_access` `download` `shared_link_access` `shared_link_download`
- current 実装で実際に enforcement 済みなのは `first_access` と `download`

#### 0 件から始めるときの最小手順

初回状態では、一覧 table や表示設定ではなく `まだ案件同意設定はありません。` の empty state が出る。最初の 1 件を作る前に、先に `同意文面` で有効な文面を用意する。

1. `同意文面` で使いたい文面が `有効` になっていることを確認する
2. 上段の `新規登録` フォームで `案件` `同意文面` `必須タイミング` を選ぶ。候補が多い場合は、`案件を選択` / `同意文面を選択` の select 内で名称を検索してから選ぶ
3. 必要なら `状態` を確認して保存する
4. 保存したら empty state は消え、`案件同意設定一覧` に最初の 1 行が出る。続けて `案件` `版` `必須タイミング` `状態` が想定どおりかを見る

一覧が 1 件以上あるときだけ `案件同意設定一覧の表示設定` と table が出るので、初回状態では「有効な文面を案件へ結び付ける画面」と読めばよい。

#### 一覧 filter と表示設定の読み分け

- 一覧 filter の `案件` / `同意文面` は、既存設定を探すための条件。`すべて` のままならその条件では絞り込まない
- filter で選んだ `案件` / `同意文面` は、絞り込み後も selected value として保持される。意図しない 0 件に見えるときは、まず選択中の filter と `状態` を確認する
- `条件に一致する案件同意設定はありません。` は、設定自体は存在しうるが現在の filter 条件では 0 件になった状態を示す
- `案件同意設定一覧の表示設定` は列の表示・幅・固定列を調整する table preferences であり、filter 条件や enforcement を変えるものではない

日常確認ポイント:

- 案件ごとに `enabled` な設定だけが残っているか
- `consent_term` の版が current 運用で要求したいものと一致しているか
- `required_on` が intended な入口と一致しているか
- 大量の案件や文面から対象を探すときは、一覧 filter の検索可能 select で `案件名 (案件コード)` や `タイトル / 版` を絞ってから見る

変更時の注意:

- `shared_link_access` と `shared_link_download` は将来拡張用の予約値で、現時点の shared link 実装を意味しない
- 同じ案件・同じ必須タイミングには 1 つの文面しか結び付けられない

戻り先:

- 利用者画面での同意 enforcement は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 4. Webhook

`admin/webhook_endpoints` は、外部通知先 endpoint の設定と、最近の送信結果を確認する画面。

- 設定一覧では `名称 / 送信先URL / イベント / 有効` を確認できる
- フォームでは `name` `target_url` `secret_token` `active` `event_types[]` を更新できる
- 新規作成時の既定イベントは `document_updated` と `document_published`
- 送信履歴には `作成日時 / 設定 / イベント / ステータス / HTTP / エラー` が出て、current filter に一致する recent 50 件を確認できる

#### empty state の読み分け

- `Webhook設定` が 0 件のときは、まだ通知先 endpoint 自体が未登録の状態を示す。上段の `新規登録` を入口にして、`名称` `送信先URL` `通知対象イベント` を埋めて最初の 1 件を保存する
- `最近の送信履歴` が 0 件のときは、送信失敗とは限らない。Webhook を保存した直後や、まだ通知対象イベントが発生していない間は、この欄が空のままでも current behavior として自然
- `failed / pending / succeeded` で絞った結果が 0 件のときは、送信履歴全体が空とは限らない。表示を `すべて` に戻し、別ステータスの履歴が recent 50 件内に残っているかを確認する
- 初回セットアップでは `Webhook設定` と `最近の送信履歴` の両方が 0 件になりうるが、前者は「登録前」、後者は「まだ送っていないか、送る機会がない」を表す別文脈として読む

#### status filter と recent 50 件の読み方

- `最近の送信履歴` の `表示` は `すべて / 失敗 / 送信待ち / 成功` を切り替える current filter。URL 上は `delivery_status=failed / pending / succeeded` で、未指定または範囲外の値は `すべて` として扱う
- 件数表示の `表示範囲` は、current filter に一致する送信履歴の総数と、そのうち画面に出ている件数を示す
- 一覧に出るのは recent order の先頭 50 件まで。50 件外の履歴は current first slice では探索導線がなく、必要なら後続 slice で詳細条件を決めて追加する
- ステータス別の件数サマリは全送信履歴の `status` 集計であり、表示中 50 件だけの内訳ではない
- status filter は送信履歴の調査入口であり、endpoint 設定の有効 / 無効や通知対象イベントの設定を絞り込むものではない

#### 失敗履歴の再送範囲

- 1 件ずつの `再送` は、失敗した送信で、現在の Webhook 設定が有効な場合だけ表示される
- `失敗` filter のときだけ、表示中の failed delivery のうち再送可能なものをまとめて再送できる
- まとめて再送の対象は、画面に表示されている recent 50 件内の再送可能な failed delivery に限られる。50 件外の failed delivery や停止中 endpoint の履歴は対象外
- まとめて再送の説明には対象件数、設定名、イベント名が出るので、受信先側で重複処理できるかを確認してから押す
- `pending` や `succeeded` は、まとめて再送の対象に含めない

#### 送信履歴詳細で見える情報

`Webhook送信履歴詳細` では、一覧の `詳細` から 1 件分の調査に進む。

- 概要では `設定 / イベント / ステータス / HTTPステータス / 送信日時 / 作成日時 / 送信先URL` を確認できる
- 失敗調査では `エラー` と `レスポンス本文` を確認できる
- `リクエスト本文` は first slice では非表示。secret や個人情報を含む可能性があるため、表示範囲とマスク方針が決まるまでは runbook 上も見える情報として扱わない
- 詳細画面の戻り先は current UI では `Webhook一覧へ戻る` で、元の status filter を保持する導線は別 issue の範囲として扱う

日常確認ポイント:

- `active` な endpoint が intended な通知先だけに絞られているか
- `Webhook設定` が 0 件なら、まず上段 `新規登録` から最初の endpoint を作る
- `最近の送信履歴` が 0 件なら、設定漏れだけでなく通知対象イベント未発生の可能性も含めて判断する
- `failed` な配信が増えていないか。増えている場合は `失敗` filter で再送可能な対象と停止中 endpoint の履歴を分けて見る
- `response_status` や `error_message`、詳細画面の `レスポンス本文` から、受信側障害か設定誤りかを切り分けられるか
- 50 件外の履歴を追いたい場合は、この runbook だけで探索できると扱わず、後続 slice の条件設計へ戻す

変更時の注意:

- current 実装には自動再送キューがないため、失敗配信の再送要否は送信履歴を見て個別または表示中 failed filter の範囲で判断する
- `event_types` は `WebhookEndpoint::EVENT_TYPES` に含まれる値だけが有効
- 署名シークレットを設定した場合は `X-Docs-Portal-Signature-256` を付与する
- request body の表示、50 件外の探索、詳細から元 filter へ戻る導線は current UI の外側なので、実装済みとして運用しない

戻り先:

- payload とヘッダー仕様は [Webhook・外部API連携方針](./Webhook・外部API連携方針.md)

## 迷ったときの切り分け

- 利用者に何を同意させるか、どの入口で止めるかを見直したい: 同意仕様 docs を先に確認する
- どの案件にどの文面を適用するか迷う: `同意文面` と `案件同意設定` を対で確認する
- 外部通知の失敗原因を追いたい: `Webhook` の recent deliveries と通知仕様 docs を対で確認する
- 権限付与の是非そのものが曖昧: この runbook では判断せず、案件運用や権限仕様の人間確認へ戻す

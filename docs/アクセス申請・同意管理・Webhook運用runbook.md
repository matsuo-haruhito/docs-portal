# アクセス申請・同意管理・Webhook運用runbook

この runbook は、管理画面の `アクセス申請` `同意文面` `案件同意設定` `Webhook` を日常運用で見直すときの入口をまとめる。

新しい運用ポリシーはここでは定義しない。current 実装と既存仕様 docs を前提に、どの画面で何を確認するか、変更時にどこへ戻るかを整理する。

## 先に見るもの

1. 利用者向けの同意仕様を確認したいときは [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)
2. 外部通知の payload や署名仕様を確認したいときは [Webhook・外部API連携方針](./Webhook・外部API連携方針.md)
3. 管理画面の位置関係は `app/views/admin/_nav.html.slim` の `アクセス申請` `同意文面` `案件同意設定` `Webhook` を起点に確認する

## 画面ごとの役割

### 1. アクセス申請

`admin/access_requests` は、利用者から届いた閲覧・ダウンロード・管理権限の申請を internal 管理者が裁く画面。

- 一覧には `日時 / 申請者 / 対象 / 要求権限 / 状態 / 理由 / 承認者` が出る
- 対象は `Project` `Document` `DocumentFile` のいずれかで、承認前は `承認` / `却下` ボタンが出る
- 申請の状態は `pending / approved / rejected / cancelled`
- 要求権限は `view / download / manage`
- 一覧上部の `状態` では `すべて / 承認待ち / 承認済み / 却下 / 取消済み` を切り替えられる
- 一覧上部の `要求権限` では `すべて / 閲覧 / ダウンロード / 管理` を切り替え、利用者が申請した `requested_access_level` だけを絞り込む
- 一覧上部の `対象種別` では `すべて / 案件 / 文書 / 添付ファイル` を切り替え、`Project` / `Document` / `DocumentFile` の申請を分けて見る
- `申請者・対象` 検索では、申請者名・メールアドレス・理由・対象名・案件コード・文書名・ファイル名・公開IDをまとめて検索する
- `申請者・対象` 検索語は最大 100 文字までで、`状態`、`要求権限`、`対象種別` と組み合わせた条件として絞り込まれる。長い理由やファイル名を探すときも、まず短い断片で検索する

#### requested_access_level と実際の付与先

`要求権限` は利用者が申請した `requested_access_level` であり、承認時に作られる role / permission と 1:1 ではない。current `AccessRequestResolver` の付与先は次の通り。

| 対象 | 承認時の付与先 | current mapping |
| --- | --- | --- |
| `Project` | `ProjectMembership` | 要求権限にかかわらず requester を `viewer` role として所属させる |
| `Document` | `DocumentPermission` | `download` 申請は `download`、それ以外は `view` |
| `DocumentFile` | 親 `Document` の `DocumentPermission` | file 単位の permission は作らず、親 Document へ `download` または `view` を付与する |

- `manage` は enum と一覧表示上の要求値として存在するが、current resolver は明示的な manage grant を持たない
- `manage` 申請を承認しても、Project 管理者 role や Document 管理権限を付与したものとして扱わない
- `manage` の扱いを変えるかどうかは `#1124` / `#1126` の design / feature lane、current mapping の regression guard は `#1127` の quality lane で扱う
- この runbook は current behavior の読み分けを補うだけで、承認基準や新しい権限仕様は定義しない

#### 結果サマリと empty state の見方

- `検索結果` は、現在の `状態`、`要求権限`、`対象種別`、`申請者・対象` 検索を適用したあとの総件数
- `表示中` は、検索結果のうち現在のページに表示している範囲
- `前へ` / `次へ` は、現在の filter と `per_page` を保ったままページを移動する
- `状態` は今どの filter を適用しているかを表示する。未選択なら `すべて`
- `要求権限` は `view / download / manage` の要求値を表示する。未選択なら `すべて`
- `対象種別` は `Project / Document / DocumentFile` の対象 class を表示する。未選択なら `すべて`
- `検索` は free-text 条件を入れているときだけ表示される
- `承認待ち / 承認済み / 却下 / 取消済み` の件数は、現在の検索結果全体に対する状態別の内訳
- `取消済み` は 1 件以上あるときだけ表示される
- `条件に一致する申請はありません。` は、申請自体は存在するが現在の条件では 0 件になった状態を示す
- 条件適用中は、フォーム側の `条件をクリア` または 0 件表示下の `すべての申請を見る` から条件なし一覧へ戻れる
- 条件未指定では解除対象がないため、フォーム側の `条件をクリア` は表示されない
- `申請はありません。` は、filter を掛けていない状態で申請レコード自体がまだ無いことを示す
- その下の `利用者からアクセス申請が届くと、承認待ち・承認済み・却下の履歴がここに表示されます。` は、初回 0 件時に今後ここへ履歴が積まれることを示す補足 copy として読む。filtered 0 件時の条件解除導線や承認基準を追加するものではない

#### 詳細画面と一覧 action の使い分け

一覧の各行にある `詳細` は、申請 1 件の確認画面へ進むための read-only review 入口として読む。

- detail では `申請ID`、`申請日時`、`申請者`、`対象`、`要求権限`、`状態`、`理由`、`処理日時`、`承認者`、`却下理由` を 1 件単位で確認できる
- 一覧から detail へ進むときは、現在の filter / page を `return_to` として渡す。detail から戻ると、承認待ち filter、要求権限、対象種別、検索語、page の文脈を保った一覧へ戻る
- `return_to` は path-only の内部 path だけを採用する。外部 URL や protocol-relative URL は一覧へ fallback するため、戻り先を外部 redirect として扱わない
- pending の申請だけ、detail から既存の承認 / 却下処理を実行できる。処理すると一覧 action と同じ resolver / rejection reason flow を通る
- approved / rejected / cancelled など処理済みの申請では、detail に承認 / 却下 form は出ない。処理済み detail は判断結果の読み返し用として扱う
- 一覧上の inline `承認` / `却下` は、filter 結果を見ながら即時判断するための入口。detail は対象・理由・処理結果を 1 件ずつ確認してから判断または読み返すための入口として使い分ける
- detail 画面は、承認段階、通知、SLA、担当者割当、bulk approve / reject を追加するものではない

#### pending handoff JSON の読み方

`GET /admin/access_requests/pending_handoff.json` は、現在の一覧 filter から pending 申請候補だけを read-only JSON として引き継ぐための補助 endpoint。承認・却下、通知、担当者割当、SLA、自動 escalation は実行しない。

- `current_filter` は、一覧と同じ `状態`、`要求権限`、`対象種別`、`申請者・対象` 検索のうち、採用された条件だけを示す
- `status` は常に `pending` として読み、`status=approved` などを指定しても候補抽出は pending に閉じる
- `total_count` は current filter に一致する pending 候補の総件数、`limit` は返す候補の上限で current 実装では 50 件
- `truncated: true` のときは 50 件を超える候補がある。JSON だけで全件処理したものと扱わず、一覧 filter や検索語を絞って再確認する
- `candidates` には `public_id`、申請者 preview、対象種別、対象 label / context、要求権限、理由 preview、状態、確認用 path が入る
- `reason_preview` は短く丸めた確認用の断片で、申請理由全文や監査用 export として扱わない
- `admin_review_path` は該当申請を一覧で探し直すための path。承認・却下の API 実行や一括処理対象ではない
- `note` は、候補ありなら read-only handoff であること、0 件なら「現在条件で対象なし」であることを補足する。0 件 note は正常保証、通知正常、全期間 0 件を意味しない

日常確認ポイント:

- `承認待ち` で絞り、処理待ちが溜まっていないかを最初に確認する
- `要求権限` で `download` 申請、`manage` 要求、通常の `view` 申請を分け、申請者の意図と current mapping の読み違いがないかを見る
- `対象種別` で `Project` / `Document` / `DocumentFile` を分け、file 申請が親 Document の permission へ寄る current behavior と混同していないかを見る
- `申請者・対象` 検索で、特定案件や文書に申請が集中していないかを見返す
- 申請対象と `requested_access_level` が利用者の想定と合っているか
- 詳細画面で対象、理由、処理日時、承認者、却下理由を 1 件ずつ読み返し、一覧上の要約だけで判断していないか確認する
- すでに処理済みの申請で、承認者が想定どおり記録されているか
- 条件に一致しないときは、`状態`、`要求権限`、`対象種別` の残り値や検索語の表記揺れを先に疑う

#### 却下理由の定型候補と補足

pending 申請を却下するときは、`却下理由の定型候補` と `補足（任意）` を使う。

- 定型候補は `権限不足 / 対象誤り / 情報不足 / 承認条件不一致` から選ぶ
- `補足（任意）` には、申請者へ伝える追加説明が必要なときだけ短く書く
- 保存される却下理由は既存の `AccessRequest#rejection_reason` で、定型候補と補足が 1 つの理由文字列として残る
- 定型候補は入力補助であり、承認基準、権限付与仕様、却下理由 taxonomy の恒久運用をこの画面で決めるものではない
- 空理由や、未対応の候補だけで補足も空の却下は current controller で受け付けない
- 既存 request spec 用の直接 `rejection_reason` param 互換は残っているが、管理画面の日常操作では定型候補と任意補足を正規の入力導線として読む
- 却下後も current filter と page 文脈は維持されるため、`承認待ち`、`要求権限`、`対象種別`、検索条件で絞っている場合は、戻った一覧の条件と表示範囲を見直す

変更時の注意:

- 権限モデル自体の正本はアプリ仕様側にあり、この runbook では承認基準を新設しない
- 定型候補を増やす、通知や SLA と結び付ける、回答期限や自動エスカレーションへ広げる場合は、この runbook ではなく別の feature / operations 判断に戻す
- current filter と件数サマリは検索結果全体と現在ページを分けて読む。別 KPI や全期間集計の説明へ広げない

戻り先:

- 権限や公開範囲の意味は [アプリケーション仕様](./アプリケーション仕様.md)
- 同意前提を含む利用者導線は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 2. 同意文面

`admin/consent_terms` は、利用規約・秘密保持・注意事項などの文面そのものを管理する画面。

- `タイトル / 版 / 種別 / 再同意方針 / 状態` を一覧で確認できる
- 新規登録と編集では `title` `version_label` `consent_scope` `requirement_timing` `active` `body` を扱う
- `title` と `version_label` の組み合わせは重複できない
- 利用履歴や設定が残っている文面は削除できないため、通常は `無効` へ切り替えて運用する

#### 0 件から始めるときの最小手順

初回状態では、一覧 table や表示設定ではなく `まだ同意文面はありません。` の empty state が出る。最初の 1 件は上段の `新規登録` フォームから作る。

1. `タイトル` と `版` を決める
2. `種別` と `再同意方針` を current 運用に合わせて選ぶ
3. `本文` を入力し、必要なら `有効` を切り替えて保存する
4. 保存したら empty state は消え、`同意文面一覧` に最初の 1 行が出る。続けて `版` `種別` `状態` が想定どおりかを見る

一覧が 1 件以上あるときだけ `同意文面一覧の表示設定` と table が出るので、初回状態では「最初の文面を作る画面」と読めばよい。

#### 一覧 filter summary と表示設定の読み分け

`一覧の絞り込み` は、既存の同意文面を探すための条件。`タイトル・版ラベル`、`状態`、`種別`、`再同意方針` を組み合わせられる。

- `条件に一致する同意文面 N件中 X-Y件を表示` は、現在の filter を適用した総件数と、現在ページに表示している範囲を分けて読む
- 条件を入れている場合は `検索: ... / 状態: ... / 種別: ... / 再同意方針: ...` のように、適用中の条件だけが summary に並ぶ
- 条件がない場合も summary は全同意文面のうち現在ページの範囲を示す。表示中の行だけで全件を確認したものとして扱わない
- 既定では 1 ページ 25 件まで表示する。URL で `per_page` が渡された場合も最大 100 件までに正規化される
- `前へ` / `次へ` は、`タイトル・版ラベル`、`状態`、`種別`、`再同意方針` と正規化済み `per_page` を保ったままページを移動する
- 無効な `page` や範囲外の `page` は 1 ページ目へ戻る。ページ指定の異常を、同意文面の削除や filter 条件の不一致とは読まない
- `状態` は filter では `有効 / 無効`、一覧の badge では `利用中 / 無効化済み` と表示される。どちらも `active` の読み方で、同意 requirement や enforcement を変更する操作ではない
- `条件に一致する同意文面はありません。` は、同意文面自体は存在するが現在の filter 条件では 0 件になった状態を示す。`条件をリセット` するか、タイトル・版ラベル、状態、種別、再同意方針を見直す
- filter なしで 0 件のときは `まだ同意文面はありません。` の初期状態として読み、pagination や表示設定ではなく上段の `新規登録` から最初の文面を作る
- `同意文面一覧の表示設定` は列の表示・幅・固定列を調整する table preferences であり、filter 条件、page、同意文面の有効 / 無効、利用者への同意要求を変えるものではない

日常確認ポイント:

- 利用中の文面が `有効` のまま残っているか
- 改訂時に `version_label` を上げているか
- `consent_scope` と `requirement_timing` が intended な利用箇所と合っているか
- 条件を入れているときは summary を見て、検索語・状態・種別・再同意方針、表示範囲が意図した組み合わせになっているか
- 件数が多いときは `前へ` / `次へ` で同じ filter のまま古い文面を確認し、現在ページだけで全件を判断しない

変更時の注意:

- 文面本文の変更を既存版へ上書きすると、どの版に同意したかの追跡が曖昧になりやすい。改訂時は新しい `version_label` を使う
- 法務レビューや文面の妥当性そのものはアプリ外運用で担保する
- filter summary と page 移動は一覧の読み返し補助であり、通知、法務承認、再同意催促 workflow を定義するものではない

戻り先:

- モデル責務と enforcement 範囲は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 3. 案件同意設定

`admin/project_consent_settings` は、どの案件でどの同意文面を必須にするかを結び付ける画面。

- `案件 / 同意文面 / 必須タイミング / 状態` を一覧で確認できる
- フォームで選べる文面は `active` な `ConsentTerm` のみ
- 新規登録フォームの `案件` と `同意文面` は検索可能な select で、多件数でも候補名を絞り込みながら選べる
- 一覧上部の filter でも `案件` と `同意文面` は検索可能な select になっており、`状態` filter と合わせて現在の一覧だけを絞り込む
- 一覧 filter の `案件` 候補は `案件名 (案件コード)`、`同意文面` 候補は `タイトル / 版` として読む
- `required_on` は `first_access` `download` `shared_link_access` `shared_link_download`
- current 実装で実際に enforcement 済みなのは `first_access` と `download`

#### 0 件から始めるときの最小手順

初回状態では、一覧 table や表示設定ではなく `まだ案件同意設定はありません。` の empty state が出る。最初の 1 件を作る前に、先に `同意文面` で有効な文面を用意する。

1. `同意文面` で使いたい文面が `有効` になっていることを確認する
2. 上段の `新規登録` フォームで `案件` `同意文面` `必須タイミング` を選ぶ。候補が多い場合は、`案件を選択` / `同意文面を選択` の select 内で名称を検索してから選ぶ
3. 必要なら `状態` を確認して保存する
4. 保存したら empty state は消え、`案件同意設定一覧` に最初の 1 行が出る。続けて `案件` `版` `必須タイミング` `状態` が想定どおりかを見る

一覧が 1 件以上あるときだけ `案件同意設定一覧の表示設定` と table が出るので、初回状態では「有効な文面を案件へ結び付ける画面」と読めばよい。

#### 検索可能 select の読み方

- `案件` の候補検索は、案件コードと案件名を対象にする。候補 label は `案件名 (案件コード)` として表示される
- `同意文面` の候補検索は、有効な同意文面の title と version label を対象にする。候補 label は `タイトル / 版` として表示される
- 検索語は前後空白を除き、最大 100 文字までを条件として扱う
- 候補は最大 20 件まで返る。見つからない場合は、検索語を短くするか、同意文面が `有効` かを確認する
- filter や validation error 後の rerender では、選択済みの `案件` / `同意文面` が selected value として復元される
- 既存の案件同意設定が後から無効化された同意文面を参照していても、設定行や編集・validation rerender の selected label は確認できる。これは既存データの読み返し用であり、無効化済み文面を新規候補として再選択できることを意味しない
- filter や選択候補の `同意文面` は引き続き active-only として読む。URL に inactive な `consent_term_id` を直接渡しても、その filter は採用されず、候補 endpoint も inactive 文面を selected option として返さない
- URL に存在しない `project_id` / `consent_term_id` が渡された場合、その filter は採用されず、未指定として読む。500 や権限外候補の表示ではない

#### 一覧 filter と bounded 表示の読み分け

- 一覧 filter の `案件` / `同意文面` は、既存設定を探すための条件。`すべて` のままならその条件では絞り込まない
- filter で選んだ `案件` / `同意文面` は、絞り込み後も selected value として保持される。意図しない 0 件に見えるときは、まず選択中の filter と `状態` を確認する
- `検索結果: N件` は、案件 / 同意文面 / 状態 filter を適用したあとの総件数
- `表示中: X-Y件 / N件` は、検索結果のうち現在ページに表示している範囲。0 件のときは `表示中: 0件` と読む
- 既定では 1 ページ 25 件ずつ表示する。件数が多い場合は `前へ` / `次へ` で移動する
- page 移動では、`案件`、`同意文面`、`状態`、必要な場合の `per_page` が維持される
- invalid page は共有 admin pagination rules に従って有効範囲へ寄せられる。ページ指定の異常を案件同意設定そのものの欠落とは読まない
- `条件に一致する案件同意設定はありません。` は、設定自体は存在しうるが現在の filter 条件では 0 件になった状態を示す
- filter あり 0 件時は、一覧 table の 0 件行に出る `すべての案件同意設定を見る` から条件なし一覧へ戻れる。フォーム側の `絞り込み解除` と同じく、設定の削除や enforcement 変更ではなく表示条件の解除として読む
- `まだ案件同意設定はありません。` は、filter なしで設定レコード自体がまだ無い初期状態を示す。この初期状態では `すべての案件同意設定を見る` は出ない
- `案件同意設定一覧の表示設定` は列の表示・幅・固定列を調整する table preferences であり、filter 条件、page、同意 enforcement を変えるものではない

#### 引き継ぎ用 summary の読み方

`引き継ぎ用 summary` は、現在の filter 条件と表示中ページだけを対象にした read-only の引き継ぎメモ。設定変更、同意本文 export、利用者同意履歴 export ではない。

- summary の `条件` には、選択中の `案件`、`同意文面`、`状態`、検索結果件数、表示中範囲が出る
- `対象範囲` は現在の表示ページのみ。pagination 外の全件を再取得したり、backend query を別途実行して export するものではない
- `表示中設定` には、表示中行の `案件名 (案件コード)`、`同意文面 / 版`、`必須タイミング`、`状態` が並ぶ
- 現在の filter で 0 件の場合は `表示中設定: なし` と読む。設定全体が存在しない保証ではなく、現在条件に一致する表示中行がないという意味
- table preferences は列の表示設定であり、handoff 条件や出力項目の選定には使わない
- `共有リンク閲覧前（予約）` / `共有リンクダウンロード前（予約）` は、summary 内でも将来拡張用の予約値として読む。現時点の共有リンク同意 enforcement 済み入口ではない
- summary には同意本文、利用者同意履歴、個人情報、CSV一括 export は含まれない

日常確認ポイント:

- 案件ごとに `enabled` な設定だけが残っているか
- `consent_term` の版が current 運用で要求したいものと一致しているか
- `required_on` が intended な入口と一致しているか
- 大量の案件や文面から対象を探すときは、一覧 filter の検索可能 select で `案件名 (案件コード)` や `タイトル / 版` を絞ってから見る
- 件数が多いときは `検索結果` と `表示中` を分け、今見ている page だけで全件を判断しない
- 担当者へ現在の確認範囲を渡すときは、`引き継ぎ用 summary` の条件、件数、表示中設定をそのまま共有し、pagination 外や同意本文の網羅資料として扱わない
- 削除前の confirm は、一覧行に出ている `案件名 (案件コード)`、`同意文面 / 版`、`必須タイミング` を押下前に読み返すための対象識別 cue として読む
- confirm に出る情報は削除対象の識別補助であり、削除 workflow、soft delete、audit log、同意 enforcement の新仕様を追加するものではない

変更時の注意:

- `shared_link_access` と `shared_link_download` は将来拡張用の予約値で、現時点の shared link 実装を意味しない
- 同じ案件・同じ必須タイミングには 1 つの文面しか結び付けられない
- 検索可能 select は候補選択と filter 補助であり、同意文面の active 条件、required_on の enforcement、通知、法務承認、再同意催促 workflow を追加するものではない
- `引き継ぎ用 summary` は一覧の read-only copy 補助であり、CSV/JSON export、同意本文・履歴出力、設定変更、一括編集、通知や法務承認 workflow を定義するものではない

- 利用者画面での同意 enforcement は [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)

### 4. Webhook

`admin/webhook_endpoints` は、外部通知先 endpoint の設定と、最近の送信結果を確認する画面。

- 設定一覧では `名称 / 送信先URL / イベント / 有効` を確認できる
- フォームでは `name` `target_url` `secret_token` `active` `event_types[]` を更新できる
- 新規作成時の既定イベントは `document_updated` と `document_published`
- 送信履歴には `作成日時 / 設定 / イベント / ステータス / HTTP / エラー` が出て、current filter に一致する recent 50 件を確認できる

#### empty state の読み分け

- `Webhook設定` が 0 件のときは、まだ通知先 endpoint 自体が未登録の状態を示す。上段の `新規登録` を入口にして、`名称` `送信先URL` `通知対象イベント` を埋めて最初の 1 件を保存する
- `最近の送信履歴` が 0 件のときは、送信失敗とは限らない。Webhook を保存した直後や、まだ通知対象イベントが発生していない間は、この欄が空のままでも current behavior として自然
- `failed / pending / succeeded` で絞った結果が 0 件のときは、送信履歴全体が空とは限らない。表示を `すべて` に戻し、別ステータスの履歴が recent 50 件内に残っているかを確認する
- `Webhook送信履歴検索` で条件適用中に 0 件になった場合は、結果カード側の `すべてのWebhook送信履歴を見る` から全条件を解除できる。初期 0 件状態では解除対象がないため、この結果側 link は出ない
- 初回セットアップでは `Webhook設定` と `最近の送信履歴` の両方が 0 件になりうるが、前者は「登録前」、後者は「まだ送っていないか、送る機会がない」を表す別文脈として読む

#### status filter と recent 50 件の読み方

- `最近の送信履歴` の `表示` は `すべて / 失敗 / 送信待ち / 成功` を切り替える current filter。URL 上は `delivery_status=failed / pending / succeeded` で、未指定または範囲外の値は `すべて` として扱う
- 件数表示の `表示範囲` は、current filter に一致する送信履歴の総数と、そのうち画面に出ている件数を示す
- 一覧に出るのは recent order の先頭 50 件まで。50 件外の履歴は current first slice では探索導線がなく、必要なら後続 slice で詳細条件を決めて追加する
- ステータス別の件数サマリは全送信履歴の `status` 集計であり、表示中 50 件だけの内訳ではない
- status filter は送信履歴の調査入口であり、endpoint 設定の有効 / 無効や通知対象イベントの設定を絞り込むものではない

#### Webhook送信履歴検索の読み方

`Webhook送信履歴検索` は、送信履歴を探して詳細へ進むための read-only 検索画面です。Webhook 設定一覧の recent 50 件表示や、失敗履歴のまとめて再送とは役割を分けて読みます。

- `Webhook設定`、`イベント`、`ステータス`、`HTTP status`、`エラー断片`、`作成日From / To` を組み合わせて履歴を絞り込める
- `HTTP status` は 100〜599 の整数だけを条件として使う。範囲外や数字以外は条件に使われず、検索欄にも採用されない
- `エラー断片` は保存済みエラーメッセージの短い一部で探すための条件で、エラー全文の表示欄ではない
- `作成日From` は指定日の 00:00 以降、`作成日To` は指定日の 23:59 までを含める。日付として解釈できない値は warning を見て、その条件だけ未適用として読む
- 検索結果の `表示範囲` は、条件一致件数と現在ページの範囲を示す。page 移動では現在の検索条件を維持する
- 条件適用中に 0 件になった場合、フォーム側の `条件をリセット` だけでなく、結果カード内の `すべてのWebhook送信履歴を見る` からも条件なし一覧へ戻れる
- 検索結果から送信履歴詳細へ進んだ場合、戻りリンクは検索条件とページを保った `Webhook送信履歴検索` へ戻る。Webhook設定一覧や recent 50 件から直接開いた場合は、従来どおり `Webhook一覧へ戻る` 導線として読む
- この検索画面は read-only なので、まとめて再送の対象範囲や endpoint 設定の有効 / 無効は変えない

#### 失敗履歴の再送範囲

- 1 件ずつの `再送` は、失敗した送信で、現在の Webhook 設定が有効な場合だけ表示される
- `失敗` filter のときだけ、表示中の failed delivery のうち再送可能なものをまとめて再送できる
- まとめて再送の対象は、画面に表示されている recent 50 件内の再送可能な failed delivery に限られる。50 件外の failed delivery や停止中 endpoint の履歴は対象外
- まとめて再送の説明には対象件数、設定名、イベント名が出るので、受信先側で重複処理できるかを確認してから押す
- `pending` や `succeeded` は、まとめて再送の対象に含めない

#### 送信履歴詳細で見える情報

`Webhook送信履歴詳細` では、一覧の `詳細` から 1 件分の調査に進む。

- 概要では `設定 / イベント / ステータス / HTTPステータス / 送信日時 / 作成日時 / 送信先URL` を確認できる
- 失敗調査では `エラー` と `レスポンス本文` を確認できる
- `リクエスト本文` は first slice では非表示。secret や個人情報を含む可能性があるため、表示範囲とマスク方針が決まるまでは runbook 上も見える情報として扱わない
- 詳細画面の戻り先は、検索結果から開いた場合は条件と page を保持した `Webhook送信履歴検索`、Webhook一覧や recent 50 件から開いた場合は `Webhook一覧へ戻る` として読み分ける。どちらも表示文脈を戻すだけで、retry 対象や検索条件の保存仕様は変更しない

日常確認ポイント:

- `active` な endpoint が intended な通知先だけに絞られているか
- `Webhook設定` が 0 件なら、まず上段 `新規登録` から最初の endpoint を作る
- `最近の送信履歴` が 0 件なら、設定漏れだけでなく通知対象イベント未発生の可能性も含めて判断する
- `failed` な配信が増えていないか。増えている場合は `失敗` filter で再送可能な対象と停止中 endpoint の履歴を分けて見る
- `response_status` や `error_message`、詳細画面の `レスポンス本文` から、受信側障害か設定誤りかを切り分けられるか
- 50 件外の履歴を追いたい場合は、`Webhook送信履歴検索` で条件を絞って詳細へ進む。詳細から戻るときは検索条件と page が保たれているかを確認する。ただし、まとめて再送は従来どおり表示中 recent 50 件の failed delivery に閉じる

変更時の注意:

- current 実装には自動再送キューがないため、失敗配信の再送要否は送信履歴を見て個別または表示中 failed filter の範囲で判断する
- `event_types` は `WebhookEndpoint::EVENT_TYPES` に含まれる値だけが有効
- 署名シークレットを設定した場合は `X-Docs-Portal-Signature-256` を付与する
- request body の表示は current UI の外側なので、実装済みとして運用しない
- 詳細から検索結果へ戻る導線は表示文脈の復元であり、まとめて再送の対象範囲、endpoint 設定、通知履歴の保存条件は変更しない

## 迷ったときの切り分け

- 利用者に何を同意させるか、どの入口で止めるかを見直したい: 同意仕様 docs を先に確認する
- どの案件にどの文面を適用するか迷う: `同意文面` と `案件同意設定` を対で確認する
- 外部通知の失敗原因を追いたい: `Webhook` の recent deliveries と通知仕様 docs を対で確認する
- 権限付与の是非そのものが曖昧: この runbook では判断せず、案件運用や権限仕様の人間確認へ戻す

# 案件所属・文書権限運用runbook

この runbook は、管理画面の `案件所属` と `文書権限` を日常運用で見直すときの入口をまとめる。

新しい権限ポリシーはここでは定義しない。current 実装と既存仕様 docs を前提に、どの画面で何を確認するか、変更時にどこへ戻るかを整理する。

## 先に見るもの

1. 権限モデル全体の正本は [基本モデルと権限](./specs/基本モデルと権限.md)
2. 利用者からの申請キューや同意管理の見方は [アクセス申請・同意管理・Webhook運用runbook](./アクセス申請・同意管理・Webhook運用runbook.md)
3. 管理画面の位置関係は `app/views/admin/_nav.html.slim` の `案件所属` と `文書権限` を起点に確認する

## 画面ごとの役割

### 1. 案件所属

`admin/project_memberships` は、社外ユーザーがどの案件に入れるかを管理する画面。

- フォームでは `project_id` `user_id` `role` を保存する
- 一覧は `projects.code`、`users.email_address` 順で並ぶ
- 同じ案件と同じユーザーの組み合わせは重複登録できない
- role は current code 上では `閲覧者 / 編集者 / 管理者` の 3 つを持つ

#### 0 件から始めるときの最小手順

初回状態では、空 table や表示設定ではなく `まだ案件所属はありません。` の empty state が出る。最初の 1 件は上段の `新規登録` フォームから作る。

1. `案件` で所属先の案件を選ぶ
2. `ユーザー` で社外ユーザーを選ぶ
3. `権限` で `閲覧者 / 編集者 / 管理者` のいずれかを選ぶ
4. 保存したら empty state は消え、`案件所属一覧` に最初の 1 行が出る。続けて `案件コード` `ユーザー` `権限` が想定どおりかを見る

`案件` と `ユーザー` は remote search combobox です。`案件` は案件コード・案件名、`ユーザー` は表示名・メールアドレスの短い断片で探し、検索語は前後空白を除いて最大 100 文字、候補は最大 20 件までに bounded されます。候補 label は `案件コード / 案件名`、または `表示名 / メールアドレス` として読みます。

edit や validation error 後の再表示では、保存済みまたは入力中の案件・ユーザーが selected option として復元されます。通常候補に出ない場合でも、選択済みの値が form に残っているかを確認してから保存し直します。これは `project_id` と `user_id` を選びやすくする form lookup であり、role model、案件所属の一覧 pagination、表示設定、bulk 付与、CSV export、文書権限やアクセス申請 workflow を変更するものではありません。

一覧が 1 件以上あるときだけ `案件所属一覧の表示設定` と table が出るので、初回状態では「フォームで登録する画面」と読めばよい。

#### 一覧が多いときのページ表示

案件所属一覧は、current 実装では default 25 件ずつ表示し、`per_page` を渡しても最大 100 件までに抑えられる。`page` や `per_page` に 0、負数、非数値、上限超過が入っても controller 側で安全な範囲へ正規化される。

- `表示中: X-Y件 / 全N件` は、現在のページに出ている行の範囲と、全体の案件所属件数を表す
- `Page X / Y` と `前へ` / `次へ` でページを移動する。先頭ページの `前へ` と最終ページの `次へ` は disabled 表示になる
- `前へ` / `次へ` は current `per_page` を保持するので、25 件表示で見ている途中に勝手に 100 件表示へ変わるものではない
- `案件所属一覧の表示設定` は列の見え方だけを変え、件数、ページ、role、所属レコード自体は変えない
- 新規登録に失敗して validation error で一覧へ戻った場合も、bounded page の一覧として表示され、全件表示へ戻るものではない

ページ移動は「大量の所属を分けて見るための表示境界」であり、案件所属の絞り込み、CSV export、bulk 操作、role policy 変更を追加するものではない。

日常確認ポイント:

- 対象ユーザーが intended な案件にだけ所属しているか
- `権限` badge が current 運用で想定している role 表示と合っているか
- 所属付け替えや削除の前に、同じユーザーの `文書権限` や pending な `アクセス申請` も確認すべきか

変更時の注意:

- `案件所属` は社外ユーザーの案件アクセスの入口であり、所属を外すと当該案件の文書導線も見直し対象になる
- `DocumentPermission` が残っていても、案件に所属していない社外ユーザーは案件導線だけでは利用しにくくなるため、所属削除と権限削除を別々に行うときは意図を確認する
- role 名の再定義や権限境界の見直しはこの runbook の対象外。迷ったら [基本モデルと権限](./specs/基本モデルと権限.md) と人間判断へ戻る

戻り先:

- 文書単位の閲覧・ダウンロード許可を見たいときは `文書権限`
- 利用者からの申請を処理したいときは [アクセス申請・同意管理・Webhook運用runbook](./アクセス申請・同意管理・Webhook運用runbook.md)

### 2. 文書権限

`admin/document_permissions` は、どの文書をどの会社またはユーザーへ追加公開するかを管理する画面。

- フォームでは `document_id` `company_id` `user_id` `access_level` を保存する
- `company_id` と `user_id` は片方だけを指定する current 制約で、両方空欄・両方指定は不可
- `access_level` は `閲覧` と `ダウンロード` を持ち、`ダウンロード` は `閲覧` を含む
- 同じ文書に対して、同じ会社単位または同じユーザー単位の重複登録はできない

#### 0 件から始めるときの最小手順

初回状態では、`文書別の権限概要` と `権限一覧` に table はまだ出ず、それぞれ `まだ権限は登録されていません。` の empty state が出る。最初の 1 件は上段の `新規登録` フォームから作る。

1. `文書名` で対象文書を選ぶ
2. `会社` または `ユーザー` のどちらか片方だけを選ぶ。保存時は選んだ側だけを残し、もう一方は空にする
3. `権限` で `閲覧` か `ダウンロード` を選ぶ
4. 保存したら empty state は消え、まず `文書別の権限概要` で文書ごとの付与件数と `閲覧 / ダウンロード` 内訳が出ることを確認する。その後 `権限一覧` で、どの会社またはユーザーへ付与した行かを確認する

初回状態では `文書別の権限概要` は文書ごとの件数を見る前の待機場所、`権限一覧` は付与先を 1 行ずつ見直す一覧として empty state が並ぶ。どちらも登録後に意味を持つ表なので、最初は table header を探すより上段フォームから 1 件作る画面と読めばよい。

`会社` と `ユーザー` を両方入れる運用や、どちらも空のまま保存する運用は current contract に反するため、入力前に対象の切り分けを決めてから登録する。両方を入れて保存した場合は `適用対象` section の alert を読み、会社またはユーザーのどちらか一方だけを残して直す。

#### 文書名 select と適用対象 select

`文書名` は `rails_fields_kit` の combobox として、文書名、URL 識別子、案件名で remote search します。1 文字以上入力すると最大 20 件の候補を表示し、選択済み文書は edit や validation error 後の再表示でも `選択済み文書 / 案件名` として復元されます。候補が見つからない場合は、文書名や slug の断片を短くして検索し直します。

`会社` と `ユーザー` は current 実装では remote search ではなく、画面に読み込まれた候補から選ぶ通常の `rfk_select` です。会社は表示名と domain、ユーザーは表示名と email を見て選びます。会社側は「会社全体に同じ権限を付与する場合だけ」、ユーザー側は「特定の 1 名にだけ権限を付与する場合だけ」選ぶ欄で、選ばない側は空欄にします。会社・ユーザーの remote search、候補 0 件時の server-side search、bulk grant は current support として扱いません。

文書名 select の検索や選択復元は、文書を探しやすくする表示補助です。`会社` と `ユーザー` の排他制約、`閲覧 / ダウンロード` の access level、`DocumentPermission` の作成・更新・削除の意味は変えません。

#### 検索・絞り込みと条件クリア

`検索・絞り込み` は、登録済みの文書権限を案件、文書名・slug、権限、付与先で見直すための表示補助です。条件は下段の `権限一覧` だけでなく、上段の `文書別の権限概要` も同じ対象に絞り込んで読みます。

- 条件未指定の初期状態では解除対象がないため、form action は `検索` だけになり、`条件をクリア` は表示されません
- `案件`、`文書名・slug`、`権限`、`付与先` のいずれかを指定したときだけ `条件をクリア` が表示され、押すと `admin/document_permissions` の条件なし一覧へ戻ります
- 条件適用中は `有効な条件:` summary と `表示中: N件` を合わせて読み、0 件の場合は権限が消えたのではなく条件不一致として、検索語や filter を見直します
- filter 0 件時は `文書別の権限概要` と `権限一覧` それぞれの empty state 近くにも `条件をクリア` が表示されます。概要側だけを見ているときも、一覧側だけを見ているときも、同じ条件なし一覧へ戻る導線として使います
- 絞り込みは表示中の行と概要件数を探しやすくするためのもので、`DocumentPermission` の保存内容、`会社` / `ユーザー` の排他制約、`閲覧 / ダウンロード` の意味、表示設定の列状態は変更しません

`条件をクリア` が出ていない状態は権限不足や画面故障ではなく、解除する条件がない状態です。filter したまま編集・削除する場合も、変更対象の会社 / ユーザー / 権限を行で確認してから操作します。

#### 会社単位とユーザー単位の選び分け

- 同じ会社に属する複数ユーザーへ、同じ文書を同じ条件で見せたいときは `会社` を使う
- 会社全体には広げず、特定の 1 人だけへ追加したいときは `ユーザー` を使う
- 保存時は、会社向け付与なら `ユーザー` 欄を空にし、ユーザー個別付与なら `会社` 欄を空にする。どちらも入力されている場合は、同時付与ではなく修正が必要な状態として読む
- すでに会社単位の付与がある文書で、さらに個別例外が必要かを迷ったときは、まず `権限一覧` に似た付与が増えていないかを確認する
- `会社` と `ユーザー` は排他的な入力欄なので、1 行の中で「会社向けの共通付与」と「個人向けの例外付与」を同時に表現しない

#### 登録後にどちらの表を見るか

- `文書別の権限概要` は、文書ごとの付与件数や `閲覧 / ダウンロード` の偏りをざっと点検したいときに見る
- `権限一覧` は、どの会社・どのユーザーへ付与したかを 1 行ずつ確認し、編集・削除したいときに見る
- 初回登録直後は、概要で文書単位の件数が増えていることを確認してから、一覧で付与先の行を確認すると追いやすい

#### 表示設定で変わるもの / 変わらないもの

`文書別の権限概要` と `権限一覧` は、登録済みの権限があるときだけそれぞれ表示設定を持つ。表示設定は列の見せ方を調整するための補助であり、`DocumentPermission` の作成・更新・削除、`access_level` 判定、閲覧/ダウンロード可否は変更しない。

- `権限概要の表示設定` は `文書名`、`案件`、`公開範囲`、`会社権限`、`ユーザー権限`、`閲覧`、`ダウンロード` の列を対象にする
- `権限一覧の表示設定` は `文書名`、`会社`、`ユーザー`、`権限`、`操作` の列を対象にする
- `文書名` や `操作` のような固定列は、権限の実体確認や編集・削除導線を失わないための列として読む
- 調査時は、表示設定で列を隠す前に `会社` / `ユーザー` / `権限` と、概要側の `閲覧` / `ダウンロード` 件数を確認する

表示設定後に見え方が変わっても、権限付与そのものを変更したい場合は行の `編集` / `削除`、または上段フォームで操作する。列の表示状態だけを根拠に、権限が消えた・増えたとは判断しない。

#### 文書別の権限概要

上段の `文書別の権限概要` は、文書単位で現在どれだけ付与があるかをざっと点検するための一覧。

- `会社権限` と `ユーザー権限`: 文書ごとに何件の付与があるかを見る
- `閲覧` と `ダウンロード`: current 付与件数の内訳を見る
- `公開範囲`: 文書自体の visibility policy を確認する
- 文書名リンク: current の文書詳細へ戻って実物を見直す

件数の偏りや想定外の付与がないかを最初に見るときは、まずこの表を使う。

#### 権限一覧

下段の `権限一覧` は、個々の付与先を編集・削除するときの一覧。

- `会社` が入っている行は会社単位の付与
- `ユーザー` が入っている行は個別ユーザー単位の付与
- `権限` badge は current code の `閲覧 / ダウンロード` 表示に従う
- 削除すると、その会社またはユーザーの閲覧範囲が直ちに変わる

日常確認ポイント:

- 文書ごとの付与件数が intended な公開範囲と合っているか
- 会社単位の付与で足りるのに、個別ユーザー付与が増えすぎていないか
- `ダウンロード` 付与が必要な利用者だけに絞られているか

変更時の注意:

- `restricted_external` 文書では、案件所属と文書権限の両方を対で確認する
- `public_with_login` 文書でも、添付ファイルのダウンロード可否は `DocumentPermission` の見直し対象になりうる
- この画面は付与先の整理が目的であり、文書の公開モデルそのものはここで決めない

戻り先:

- 社外ユーザーの案件所属を見直したいときは `案件所属`
- 公開範囲や `view / download` の意味は [基本モデルと権限](./specs/基本モデルと権限.md)
- 利用者からの権限申請経路を追いたいときは [アクセス申請・同意管理・Webhook運用runbook](./アクセス申請・同意管理・Webhook運用runbook.md)

## 案件権限変更 preview JSON の dry-run 境界

`GET /admin/projects/:code/permission_preview` は、案件単位の権限変更候補を保存前に確認するための JSON preview endpoint。`company_ids` や `user_ids` で対象 viewer を選び、`grant_document_ids`、`revoke_document_ids`、`grant_download_document_ids`、`revoke_download_document_ids`、`grant_project_membership`、`revoke_project_membership` を渡すと、変更した場合の見え方の差分だけを返す。

この endpoint は dry-run なので、呼び出しだけでは `ProjectMembership` や `DocumentPermission` を作成・更新・削除しない。preview 結果は、変更前の確認、影響範囲の読み合わせ、申請・運用判断の材料として扱う。

- preview JSON は権限適用そのものではない。実際に保存したい場合は、`案件所属` または `文書権限` の保存導線で対象レコードを確認してから操作する
- preview JSON は正式承認 workflow、通知、audit log 保存、SLA 判定を追加するものではない
- `changed_viewers`、`gained_documents`、`lost_documents`、`gained_download_documents`、`lost_download_documents` は、保存した場合の差分を読むための補助情報として扱う
- company と user の viewer 展開は確認対象を広げるためのもので、会社・ユーザー selector の remote search や bulk grant を current support として定義するものではない

preview で想定外の差分が出た場合は、そのまま適用済みとみなさず、対象 viewer、対象文書、会社単位 / ユーザー単位の切り分けを見直してから、実際の保存導線へ戻る。

## 迷ったときの切り分け

- まず案件に入れるかを見たい: `案件所属` を確認する
- どの文書を追加で見せるかを見たい: `文書権限` を確認する
- 文書権限フォームで会社とユーザーの両方に値がある: 1 行で同時付与せず、残す対象だけを決めてもう一方を空にする
- 利用者からの要求や承認履歴を追いたい: `アクセス申請` を確認する
- role や access level の定義自体を変えたい: この runbook では決めず、仕様 docs と人間判断へ戻る

# コーディング規約

## 基本原則

- `DRY`
  - 同じ知識、同じ条件分岐、同じ表示ロジックを複数箇所へ重複させない
  - 重複が見えたら helper、model、partial、concern、専用クラスへの集約を優先する
- `CoC`
  - Rails の命名、配置、route helper、REST、partial、layout の慣習を優先する
  - 既存の Rails 規約で自然に表現できるものを独自ルールでねじらない
- `Fat Model, Skinny Controller`
  - controller は認証、認可、パラメータ受け取り、遷移制御に寄せる
  - データ解決、変換、判定、再利用したい処理は model、concern、専用クラスへ逃がす
- 安易に service や独自 abstraction を増やさない
  - まず model、controller、helper、concern、`BaseController` で自然に表現できないかを検討する
  - `app/services` のような曖昧な置き場へ寄せず、Importer / Renderer / Builder など責務名が明確なクラス名・配置を優先する

## Rails 全般

- 共通レイアウトの部品は partial に切り出す
- 共通のリンク生成、テーブル表示、ページタイトル設定は helper に寄せる
- controller は薄く保ち、パス書き換えや HTML 変換などの処理は専用クラスへ逃がす
- 画面系 controller は `BaseController`、管理画面は `Admin::BaseController` を継承する
- API 系 controller は `Api::BaseController` を継承する
- Rails 標準で用意されている route helper、form helper、enum、partial、before_action を優先して使う
- 命名は Rails の標準に合わせ、resource 名、controller 名、view 配置をできるだけ一致させる

## 専用クラスの配置

- import 処理は `app/importers/` に置く
  - 例: `DocumentImporter`
- Docusaurus HTML の変換・配信補助は `app/renderers/` に置く
  - 例: `DocusaurusSiteRenderer`
- dry-run や変更前 preview の組み立ては `app/previews/` に置く
  - 例: `DocumentBulkEdit::ChangeNormalizer`, `DocumentBulkEdit::ItemBuilder`
- 診断・品質確認の check 群は `app/checks/` に置く
  - 例: `ApplicationConfiguration::EnvironmentChecks`, `DocumentVersionQuality::MarkdownReferenceChecks`
- import 配下の走査・展開・manifest 組み立ては `app/importers/<feature>/` に置く
  - 例: `ZipImport::ArchiveExtractor`, `GitImport::DocumentPayloadBuilder`
- seed 専用処理は `db/seeds/support/` に置く
  - 例: `SeedSupport::DocusaurusBuilder`
- `app/services` は原則として増やさない
- ただし、Rails 標準の置き場や責務名付きディレクトリで表現しづらいものだけ、例外的に利用を検討する

### `app/` 直下を増やす判断基準

- 新しいトップレベルカテゴリは、責務が名前で即説明でき、今後も複数クラスで再利用される見込みがある時だけ追加する
- 目安として、同種のクラスが 3 個以上ある、または近いうちに増える見込みがある場合に検討する
- 単発の helper や一時的な分解なら、既存カテゴリ配下へ置くか、service を薄くして近い責務へ寄せる
- `app/services/<use_case>/` は第一選択にしない
  - orchestration と判定・整形・走査が同じ箱へ戻りやすいため
  - 既存の `app/previews/`, `app/checks/`, `app/importers/`, `app/builders/` で表現できるならそちらを優先する
- service を thin にした後、責務名が明確になったクラスは配置も `app/services` から移す
  - 例: hash / markdown formatter は `app/presenters/` や `app/exports/`
  - 例: query object は `app/queries/`
  - 例: preview 結果や dry-run は `app/previews/`

## 認証・権限

- `current_user` に直接ロール判定を散らさず、`User` モデルに意図の分かる問い合わせメソッドを置く
- `internal`、`admin`、`company_master_admin`、`external` の責務を混同しない
- Project / Document の参照判定は model concern に寄せる
- `DocumentVersion#viewable_by?` や `DocumentFile#downloadable_by?` のような resource 固有判定は model へ直書きする
- controller 側の access gate は `BaseController` に `require_*_access!` として寄せる

## public_id

- 外部 URL/API には DB の連番 `id` を直接出さない
- アプリケーション管理テーブルには `public_id` を持たせる
- `public_id` は作成時に自動生成し、原則変更しない
- `public_id` は URL-safe な文字列とし、テーブルごとに prefix を付ける
- `upsert_all` は callback を通さないため、seed や bulk import では `public_id` を明示的に投入する

## View

- 一覧テーブルは原則 `table_tag` を使う
- 編集・削除ボタンは `edit_link_to` `delete_link_to` を優先して使う
- 画面タイトルは `page_title` で設定する
- navbar、flash、エラーメッセージなど、複数画面で使う断片は partial に寄せる

## seed / データ投入

- seed は `create!` の逐次投入より `upsert_all` を優先する
- seed 内では enum を文字列のまま渡さず、整数値へ明示変換する
- seed 用の自然キーは CSV 側で安定させる
- seed 用 CSV は `db/seeds/data/` に置く
- seed 補助クラスは `db/seeds/support/` に置く
- seed 用 Docusaurus build は、元ファイルを直接変更せず、一時 workspace で行う

## ドキュメント

- 実装判断や仕様が安定したら `docs/` に反映する
- `docs/` に文書を増やしたら `docs/README.md` を更新する
- 将来対応・保留事項は仕様本文に混ぜず、`docs/ToDo.md` に分離する

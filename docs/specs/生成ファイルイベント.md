# 生成ファイルイベント

この文書は、生成ファイルの変更通知を受けて後続ジョブを起動する `GeneratedFileEvent` 周辺の仕様をまとめる。

## 目的

生成ファイルイベントは、Docusaurus などの生成物・同期物の変更を直接ジョブ実行へ流し込む前に、以下を行うためのバッファである。

- 同じファイルに対する短時間の連続変更をまとめる
- 後続ジョブへ渡す path / operation / metadata を正規化する
- 失敗したイベントを管理画面から再 dispatch できるようにする
- unsafe path を入口ごとに拒否または無視する

## safe path ルール

イベントで扱う path は、リポジトリまたはアプリケーション内の相対パスだけに限定する。

受け入れる path:

- `docs/source.yml`
- `docs\\source.yml`（保存・検索時は `docs/source.yml` として扱う）
- `./docs/source.yml`（先頭の `./` は落とす）
- `./docs/../docs/source.yml`（cleanpath 後に `docs/source.yml` として扱う）

拒否または無視する path:

- 空文字
- `.`
- `..`
- `../docs/source.yml`
- `/tmp/source.yml`
- `C:/tmp/source.yml`
- cleanpath 後も `..` を含む path

入口によって振る舞いは少し異なる。

- `GeneratedFileEvent` model は validation error にする
- notifier / job / handler / buffer は unsafe path をイベント対象から落とす
- 管理画面の path filter は `\\` を `/` に正規化して検索する

## 主な入口と責務

### `GeneratedFiles::ChangeEventNotifier`

外部の変更検知から非同期 job へ渡す最初の入口。

- `file_events` を `{ path:, operation: }` 形式へ正規化する
- `operation` が空なら `update` として扱う
- 重複イベントは `uniq` でまとめる
- unsafe path は enqueue 前に落とす

### `GeneratedFileChangeEventJob`

非同期 job の入口。

- concurrency key を path / operation から作る
- `changed_files` 形式と `file_events` 形式の両方を受ける
- `debounce_seconds` が正なら `GeneratedFiles::EventBuffer` に渡す
- debounce なしなら `GeneratedFiles::ChangeEventHandler` を直接呼ぶ

### `GeneratedFiles::EventBuffer`

短時間の連続イベントを DB 上の `GeneratedFileEvent` にまとめる。

- event key は `path:operation:event_source` で作る
- pending の同一 event key があれば更新する
- `occurrences_count` を増やす
- `scheduled_at` を debounce 後に更新する
- 1件以上 buffer した場合だけ dispatch job を予約する

### `GeneratedFiles::ChangeEventHandler`

registry の rule とイベントを照合し、実際の後続 job を enqueue する。

- `operations` と `path_patterns` / `paths` を照合する
- `ignore_generated_events` が有効な rule は生成 job 起点のイベントを無視する
- rule params 内の `$changed_files`, `$matched_files`, `$event_source`, `$metadata`, `$operations` を展開する
- rule に debounce があり、buffer dispatch 済みでなければ再度 `EventBuffer` に渡す

### `GeneratedFileEvent` model

buffer 済みイベントの保存先。

- path を保存前に `/` 区切りへ正規化する
- safe relative path 以外を validation error にする
- `due` scope は `pending` かつ `scheduled_at <= now` を対象にする
- `mark_processed!` は `processed` にして `error_message` を消す
- `mark_failed!` は `failed` にして error message と processed_at を残す

## 管理画面での再 dispatch

管理画面では、失敗した生成ファイルイベントを再 dispatch できる。

- 個別再 dispatch は対象イベントを `pending` に戻して dispatch job を enqueue する
- 一括再 dispatch は、現在のフィルタに一致する failed イベントを古い順に最大 100 件まで戻す
- 一括再 dispatch 後もフィルタ条件を維持する
- 0件でも「0 件」の notice を出す

## 確認観点

生成ファイルイベント周辺を変更したときは、少なくとも以下を確認する。

```bash
bundle exec rspec \
  spec/models/generated_file_event_spec.rb \
  spec/jobs/generated_file_change_event_job_spec.rb \
  spec/jobs/generated_file_event_dispatch_job_spec.rb \
  spec/services/generated_files/change_event_notifier_spec.rb \
  spec/services/generated_files/event_buffer_spec.rb \
  spec/services/generated_files/change_event_handler_spec.rb \
  spec/requests/admin_generated_file_events_spec.rb \
  spec/requests/admin_generated_file_event_path_filter_spec.rb \
  spec/requests/admin_generated_file_event_retry_limit_spec.rb \
  spec/requests/admin_generated_file_event_retry_path_filter_spec.rb
```

CI が動いていない場合は、上記のローカル / Docker 実行結果を優先して判断する。

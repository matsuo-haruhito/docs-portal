# 本番運用・インフラ前提

この文書は issue `#148` に対応する、`docs-portal` の本番運用前提を整理したものです。

## 1. 前提構成

現行 repo が前提にしている構成要素は次です。

- Rails アプリ
- PostgreSQL
- `storage/` 配下の添付・Docusaurus build 成果物
- GitHub Actions `ci`
- 内部 import API
- Docusaurus build
- Kroki 連携

本番ではローカル filesystem 固定ではなく、アプリ実行基盤と永続ストレージを分ける前提で考えます。

## 2. ストレージ前提

本番向けの保存設計は次を基準にします。

- 原本に近い配布物: `storage/document_files/`
- Docusaurus 生成成果物: `storage/docs_sites/`
- import 用 staging: `storage/imports/`

GCS 等のオブジェクトストレージを使う場合は、次を分けて扱います。

- bucket
- prefix
- lifecycle
- public access 禁止

推奨例:

```text
gs://docs-portal-production/
  document_files/
  docs_sites/
  imports/
```

## 3. 実行環境前提

Cloud Run / GCE / VM / container 基盤のどれでも、次を満たす必要があります。

- Rails が PostgreSQL に接続できる
- `storage/` 相当の永続保存先がある
- Docusaurus build に必要な Node/npm がある
- import 時の一時展開用ディレクトリを確保できる

### tmp 容量

次の処理は一時領域を消費します。

- import artifact 展開
- Docusaurus build
- ZIP 生成

本番では、同時実行数を絞るか、十分な一時領域を持つ worker に寄せます。

### timeout

次の処理は同期 request で長くなりやすいです。

- import
- Docusaurus build
- 大きな ZIP 生成

現時点では同期実行の箇所が残るため、本番では timeout を短くしすぎない一方、将来的には job 化を前提にします。

## 4. DB 構成

`config/database.yml` では production 向けに次の分離余地があります。

- primary
- cache
- queue
- cable

最初は単一 DB でもよいですが、次のどちらかで分離を検討します。

- queue 負荷が増えた時
- cache / cable の運用責務を分けたい時

## 5. ActiveJob backend 方針

現行 repo は `config/puma.rb` に `plugin :solid_queue if ENV["SOLID_QUEUE_IN_PUMA"]` を持っているため、第一候補は Solid Queue です。

現時点の方針:

1. まずは Rails 標準寄りの Solid Queue を優先検討する
2. 大量ジョブや別 worker 分離が必要なら Sidekiq を比較する
3. Cloud native に寄せる必要が強い場合だけ Cloud Tasks 等を検討する

job 化の優先対象:

- import
- Docusaurus build
- ZIP 生成
- メール送信
- Webhook 送信

## 6. 秘密情報管理

最低限管理すべき secrets は次です。

- `SECRET_KEY_BASE`
- `RAILS_MASTER_KEY`
- `DOC_IMPORT_TOKEN`
- `DOC_IMPORT_ACTOR_EMAIL`
- DB 接続情報
- `KROKI_ENDPOINT`
- 将来の Git token / Exchange credential / webhook secret

`ApplicationConfigurationDiagnostic` でも `DOC_IMPORT_TOKEN` と `KROKI_ENDPOINT` の健全性を診断しているため、本番でも同じキーを正本として扱います。

## 7. import / build / mail / webhook の非同期化方針

現状でも `PublishJob` が import の状態記録を持っています。

本番では次を段階的に進めます。

1. 実行履歴を残す
2. 失敗理由を残す
3. 再実行単位を分ける
4. request 内の重い処理を job へ退避する

特に優先度が高いのは import と build です。

## 8. health check

本番で最低限見る health check は次です。

- Rails 起動可否
- DB 接続可否
- `storage/document_files` または相当ストレージの疎通
- Docusaurus workspace の存在
- Kroki 必須構成時の endpoint 設定

アプリ内の `ApplicationConfigurationDiagnostic` は、少なくとも次の確認材料を提供します。

- 必須 env
- 数値 env
- `DOC_IMPORT_TOKEN`
- `ACTIVE_STORAGE_SERVICE`
- `storage/document_files` writable
- Docusaurus workspace
- `KROKI_ENDPOINT`

## 9. バックアップ / restore 前提

本番運用では、DB と `storage/` の両方を戻せることが前提です。

詳細手順は次を参照します。

- [バックアップ・リストア手順](./バックアップ・リストア手順.md)
- [リリース・デプロイ・rollback手順](./リリース・デプロイ・rollback手順.md)

## 10. rate limit / 外部依存

現時点で注意すべき外部依存は次です。

- GitHub Actions / GitHub API
- Kroki
- 将来の Exchange / SMTP
- 将来の webhook 送信先

本番では、失敗時に即時リトライを繰り返すのではなく、job 側で retry 制御を持たせます。

## 11. maintenance mode の考え方

完全停止ではなく、段階的な制御を前提にします。

- import 停止
- build 停止
- 閲覧のみモード
- 管理画面の危険操作停止

文書閲覧そのものを止めずに、変更系だけ止められる構成を優先します。

first slice では `READ_ONLY_MAINTENANCE` を使い、生成ファイル実行履歴の再実行要求だけを停止します。この状態でも実行履歴の一覧・詳細閲覧は継続し、停止中の単体再実行 / 一括再実行は job を enqueue せず、管理者へメンテナンス中であることを表示します。

### 変更系操作 inventory

この表は `config/routes.rb` から見える代表的な POST / PATCH / DELETE entrypoint を、maintenance mode での検討入口として棚卸しするものです。ここに載っているだけでは停止済みを意味しません。`current` は既存 runbook で確認できる読み方、`候補` は個別 Issue / PR で停止可否を決める対象、`要判断` は docs だけでは停止方針を確定しない対象です。

## 12. 現時点の運用ルール

- 本番では sample token / sample key を使わない
- import root と `storage/` は永続化前提で扱う
- import / build / ZIP 生成は将来の job 化を見越して timeout を設計する
- GCS 等を使う場合も、prefix 設計と lifecycle を文書化してから導入する

# ローカル編集からポータル更新までの最小運用案

`ローカルで文書編集 -> git push -> GitHub Actions -> Rails ポータル更新` を、このサンプルを起点に回すための最小案です

## 1. 全体像

役割は 3 つに分けます

- 文書 repo
  - `Markdown` `PDF` `Excel` `Word` などの編集正本を管理する
- GitHub Actions
  - 文書を build し、公開用 artifact と `publish.json` を作る
- Rails ポータル
  - artifact を取り込み、版管理、権限制御、閲覧、配布を行う

## 2. 推奨する文書 repo の置き方

最低限、次の 3 系統に分けます

- `docs-src/`
  - Docusaurus で公開する Markdown 原稿
- `attachments/`
  - PDF、Excel、Word などの配布ファイル
- `publish/`
  - Actions で生成する manifest や補助ファイル

例:

```text
docs-src/
  pj001/
    dispatch-operation-manual/
      index.md
    dispatch-api-spec/
      index.md
attachments/
  pj001/
    dispatch-operation-manual/
      manual.pdf
    dispatch-api-spec/
      api-spec.xlsx
      api-spec.pdf
publish/
  documents.json
  manifest/
    publish.json
```

## 3. publish.json の役割

`publish.json` は、Rails 側へ「今回何を公開対象として取り込むか」を渡す正本です

最低限、次を持たせます

```json
{
  "source_repo": "example/docs-repo",
  "source_branch": "main",
  "source_commit_hash": "abc123",
  "documents": [
    {
      "project_code": "pj001",
      "slug": "dispatch-operation-manual",
      "title": "配車運用マニュアル",
      "category": "manual",
      "document_kind": "markdown",
      "visibility_policy": "restricted_external",
      "version_label": "v1.1.0",
      "status": "published",
      "changelog_summary": "画面説明を追加",
      "markdown_entry_path": "pj001/dispatch-operation-manual/v1.1.0",
      "site_build_path": "pj001/dispatch-operation-manual/v1.1.0",
      "pdf_snapshot_path": "pj001/dispatch-operation-manual/v1.1.0/manual.pdf",
      "files": [
        {
          "file_name": "manual.pdf",
          "content_type": "application/pdf",
          "storage_key": "pj001/dispatch-operation-manual/v1.1.0/manual.pdf",
          "file_size": 234567
        }
      ]
    }
  ]
}
```

`documents` に載っていないものは、今回の公開対象にしない方針にします

## 4. 文書 repo 側の公開ルール

- `main` push で Actions を起動する
- `publish/documents.json` のうち `status: "published"` の文書だけ `publish.json` に入れる
- `draft` は build しても import しない
- `archived` は公開 manifest に入れない
- 同じ `slug + version_label` の再取込はエラーとする
- 修正版は `v1.1.1` のように版を上げる
- バージョン管理しない Document は、同一 Document を上書きする

## 5. GitHub Actions の最小構成

運用を安定させるなら、最初は 1 本の workflow に寄せた方が安全です

1. checkout
2. Node セットアップ
3. Docusaurus build
4. `publish.json` 生成
5. 配布ファイルを artifact 用ディレクトリへ集約
6. `tar.gz` 化
7. Rails import API を呼ぶ

理由は次です

- `workflow_run` をまたぐ artifact 受け渡しは壊れやすい
- build と import を同一 workflow にすると失敗箇所が追いやすい
- 初期運用では `build 成功したが import 失敗` を同じログで見たい

## 6. Rails 側で必要な処理

Rails 側では `DocumentImporter` が次を行います

- artifact 展開先の一時ディレクトリを受け取る
- `publish.json` を読む
- `docusaurus/build` を DocumentVersion の site root 配下へ配置する
- 添付ファイルを `storage/document_files/<storage_key>` へ配置する
- 同一 `slug + version_label` が既に存在する場合は失敗させる
- import 成功後に `DocumentVersion` と `DocumentFile` を登録する
- published version のうち semantic version 比較で最も新しいものを `latest_version` とする
- バージョン管理しない Document は上書きする

最低限の方針はこれで十分です

- 公開版は immutable にする
- 同じ `version_label` の再取込はエラー
- 修正版は `v1.1.1` のように版を上げる

## 7. ポータル側で見る情報

ポータルでは次が見えれば実務で回しやすいです

- 案件
- 文書名
- 版
- 公開状態
- 更新概要
- 閲覧可能ユーザーまたは企業
- 配布ファイル一覧
- 元 commit hash

今の model 構成はこの方向に合っています

## 8. 最小の実装順

1. 文書 repo に `docs-src/` `attachments/` `publish/` を置く
2. `status: "published"` の文書を拾って `publish.json` を作るスクリプトを作る
3. Actions を 1 本化して `build -> manifest 生成 -> import API` まで通す
4. Rails 側に artifact 配置処理を足す
5. `slug + version_label` の重複ルールを確認する
6. 権限制御と表示確認を行う

## 9. この sample を使うときの判断

この sample は次の用途に向いています

- 文書版管理ポータルの骨組み確認
- 文書公開の権限制御確認
- Actions から Rails へ取り込む流れの試作

逆に、次はまだ別途詰める必要があります

- 認証基盤
- 文書 repo 側の原稿規約
- manifest 自動生成
- artifact の永続保存方式

## 10. 結論

このサンプルを起点に、`ローカル編集 -> git push -> Actions -> ポータル更新` は十分組めます

ただし、先に固めるべき正本は `Rails 側の画面` ではなく、文書 repo 側の `publish.json` 生成規約です

ここが固まれば、Actions と Rails の実装は比較的素直につながります

# バックアップ・リストア手順

この文書は issue `#150` に対応する、現行 `docs-portal` の標準バックアップ / リストア手順です。

## 1. バックアップ対象

本アプリでは、永続化対象を次の 2 系統に分けます。

1. DB
2. ファイル実体

具体的には次を対象にします。

- PostgreSQL
- `storage/document_files/`
- `storage/docs_sites/`
- `storage/imports/` のうち監査上残す必要があるもの

## 2. 再生成可能なもの

次は原則として再生成可能です。

- Docusaurus build 成果物
- diagram SVG
- import 時の一時展開物
- root / Docusaurus の依存 install 結果

ただし、再生成可能でも「復旧時間を短くするためにバックアップ対象へ含める」判断は許容します。

現時点の推奨は次です。

- `storage/docs_sites/` は再 build 可能だが、短時間復旧のためバックアップ対象に含める
- `storage/imports/` は長期保管の正本とは見なさず、必要性が明確なものだけ保持する

## 3. バックアップ取得タイミング

最低限、次のタイミングでバックアップを取得します。

- 本番リリース直前
- 破壊的 migration の直前
- 大量 import / 一括削除 / データ補正の直前
- 定期バックアップ

## 4. DB バックアップ

PostgreSQL は論理バックアップを標準とします。

取得対象:

- production primary
- production cache / queue / cable を分離している場合はそれぞれ

基本例:

```bash
pg_dump -Fc -h <db-host> -U <db-user> <db-name> > docs_portal_production.dump
```

必要に応じて次も残します。

- schema-only dump
- role / extension 情報
- バックアップ取得時点の commit SHA

## 5. ファイルバックアップ

ファイル実体は `storage/` 配下を論理区分ごとに扱います。

優先度:

1. `storage/document_files/`
2. `storage/docs_sites/`
3. `storage/imports/`

ローカル / VM / コンテナ volume なら tar か rsync、オブジェクトストレージなら prefix 単位の snapshot / versioning を使います。

基本例:

```bash
tar -czf storage-backup-YYYYMMDDHHMMSS.tar.gz storage/document_files storage/docs_sites
```

## 6. 保存先と命名

バックアップ名には最低限次を含めます。

- 環境名
- 日時
- commit SHA または release 識別子

例:

```text
prod-db-20260505T013000Z-3f1e79e.dump
prod-storage-20260505T013000Z-3f1e79e.tar.gz
```

## 7. リストアが必要になる典型ケース

- 誤 import で Document / DocumentVersion / DocumentFile が壊れた
- 誤削除や誤 archive により業務継続に必要なデータが消えた
- migration 失敗で schema / data が不整合になった
- `storage/` 消失で添付や HTML が配信できなくなった

## 8. リストア手順の原則

リストアは次の順で行います。

1. 障害発生時点の現状を保全する
2. 復旧に使うバックアップを特定する
3. アプリを停止またはメンテナンス状態にする
4. DB を戻す
5. `storage/` を戻す
6. 必要なら再 build / 再 import する
7. 整合性確認を行う
8. アプリを再開する

## 9. DB リストア

論理バックアップを戻す標準例:

```bash
dropdb --if-exists <db-name>
createdb <db-name>
pg_restore -d <db-name> docs_portal_production.dump
```

production で cache / queue / cable DB を分離している場合は、それぞれも同じ release 時点に揃えます。

## 10. ファイルリストア

`storage/` バックアップを戻す例:

```bash
tar -xzf storage-backup-YYYYMMDDHHMMSS.tar.gz
```

オブジェクトストレージ利用時は、対象 prefix を指定時点へ戻すか、復旧先 prefix を新設して切り替えます。

## 11. 再生成で対応するケース

次は backup restore より再生成の方が安全な場合があります。

- `storage/docs_sites/` だけ消えた
- diagram 生成物だけ壊れた
- import し直せば再現できる build 成果物だけ壊れた

その場合でも、DB と原本ファイルが健全であることを確認してから再生成します。

## 12. 復元後の整合性確認

復元後に最低限見る項目は次です。

- Rails が起動する
- ログインできる
- 案件一覧が表示できる
- 文書詳細が表示できる
- HTML 表示が開ける
- 添付ファイルがダウンロードできる
- `Document.latest_version` が主要文書で期待どおりになっている
- `storage/document_files/` と DB の `storage_key` が噛み合っている
- `storage/docs_sites/` と `DocumentVersion.site_build_path` が噛み合っている

## 13. 誤 import / 誤削除時の扱い

誤 import / 誤削除では、すぐに上書き操作を重ねないことを優先します。

1. 問題発生時点の DB dump を追加取得する
2. 問題の範囲を `project` `document` `document_version` 単位で特定する
3. 全体 restore が必要か、対象限定 restore / 再 import で済むか判断する
4. 必要なら別環境で restore 検証してから本番へ戻す

## 14. 復元確認の運用

バックアップは「取れている」だけでは不十分です。

定期的に次を確認します。

- dump ファイルが破損していない
- `pg_restore --list` などで読める
- `storage` アーカイブが展開できる
- ステージング等で最低限の復元手順を再現できる

## 15. 現時点の運用ルール

- リリース前バックアップなしで migration を実行しない
- DB と `storage/` のどちらか片方だけ戻して復旧完了と見なさない
- `storage/docs_sites/` だけの問題でも、DB 健全性確認を省略しない
- restore 後は、少なくとも閲覧・ダウンロード・主要管理導線を目視確認する

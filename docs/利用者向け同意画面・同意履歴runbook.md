# 利用者向け同意画面・同意履歴runbook

この runbook は、利用者が `同意済み文面・注意事項` と `利用上の注意事項への同意` を current UI に沿って見返すときの入口をまとめる。

新しい同意 policy や文面運用ルールはここでは定義しない。current 実装にある表示項目、戻り方、管理画面側へ確認を戻す境界だけを整理する。

## 先に見るもの

1. 文面の管理責務や `consent_scope` / `requirement_timing` の意味を確認したいときは [利用規約・秘密保持の同意管理](./利用規約・秘密保持の同意管理.md)
2. 管理者が `同意文面` や `案件同意設定` をどう確認するかは [アクセス申請・同意管理・Webhook運用runbook](./アクセス申請・同意管理・Webhook運用runbook.md)
3. current route と redirect 条件を確かめたいときは `config/routes.rb` と `app/controllers/consents_controller.rb`

## 画面ごとの役割

### 1. 同意済み文面・注意事項

`GET /consents` は、現在有効な注意事項と、自分がどの文面に同意したかを 1 画面で見返す一覧。

- 上段の `現在有効な注意事項` では、現在 `active` な文面だけを `タイトル / 版 / 種別 / 本文` つきで確認できる
- active 文面が 1 件以上ある場合は、section 冒頭の件数 cue で現在有効な文面数を確認し、文面ごとの `種別` / `版` と本文を読み分ける
- active 文面は `現在有効な注意事項` の section 内で list item として並ぶ。文面ごとの nested card ではなく、複数の active 文面を同じ section の中で読み比べる表示補助として扱う
- 各 active 文面の本文は `この文面の本文` ラベルの下に表示される。履歴の本文ではなく、いま提示されている active 文面の本文として読む
- 下段の `自分の同意履歴` では、同意した順に `同意日時 / 文面 / 種別 / 版 / 対象` を確認できる
- `対象` が `全体` の行は、案件や文書にひも付かない global な同意を表す
- `対象` に `Project` `Document` `DocumentFile` `DocumentVersion` が出ている行は、その対象に対して記録された同意を表す
- 同意履歴は current user がどの文面のどの版へ同意したかを残す個人ユーザー単位の確認履歴であり、会社間契約、法務承認、契約締結状況の代替としては扱わない

empty state の見方:

- `現在有効な注意事項はありません。` は、今表示すべき active 文面が無い状態を示す。エラーや権限不足ではなく、下段の同意履歴とは別に読む
- `まだ同意履歴はありません。` は、自分の同意記録がまだ 0 件であることを示す。利用不可や権限不足を示すものではない

日常確認ポイント:

- いま有効な文面として何が提示されているかを確認したいときは、まず上段を見る
- active 文面が複数ある場合は、件数 cue、文面ごとの `種別` / `版`、`この文面の本文` ラベルを手掛かりに、現在読むべき本文と履歴 table を混同しない
- どの版に同意したか、対象付きの同意だったかを見返したいときは、下段の `種別`、`版`、`対象` を見る
- 同じタイトルでも版が上がっている場合があるので、履歴の確認では `文面` 名だけでなく `版` も一緒に見る

### 2. 利用上の注意事項への同意

`GET /consents/new` は、閲覧やダウンロードの直前に必要な同意が残っているときだけ表示される確認画面。

- 画面上部では、「このポータルで閲覧・ダウンロードする情報の取り扱いを確認してください」と案内される
- 必要な文面があるときは、文面ごとに `タイトル / 版 / 種別 / 本文` が card で並ぶ
- `確認して同意する` を押すと、未同意の文面だけが current user に記録される
- 同意後は `return_to` に安全な相対 path が入っていればその画面へ戻り、無い場合や unsafe な値だった場合は `案件一覧` へ戻る

current `main` での戻り方:

- `確認して同意する` 後は `return_to` の安全性を確認してから戻る
- `必要な同意は完了しています。` と判定された場合も、同じ戻り先ルールでリダイレクトされる
- `同意せず戻る` も同じ戻り先ルールを使う。`return_to` に安全な相対 path が入っていればその画面へ戻り、無い場合や unsafe な値だった場合は `案件一覧` へ戻る
- missing term が無い状態で表示された `戻る` ボタンも、`return_to` が安全な相対 path ならその path、無い場合や unsafe な値だった場合は `案件一覧` へ戻る

日常確認ポイント:

- 文面を読んで同意するだけでよいのか、まだ別の権限申請が必要なのか迷ったときは、同意後の戻り先画面で次の action を確認する
- `同意せず戻る` は `確認して同意する` と同じ safe return_to 境界で戻るため、戻り先が対象文書や案件になるか `案件一覧` になるかは `return_to` の安全性で決まる
- すでに同意済みのときは確認画面に留まらず戻されるので、履歴を見返したいだけなら `GET /consents` を開く

## 管理画面側へ戻す論点

- どの文面を active にするか
- global 文面の `requirement_timing` として `first_view` / `every_version_change` / `every_download` をどう使い分けるか
- 案件同意設定の `required_on` として `first_access` / `download` をどう使い分けるか
- `shared_link_access` / `shared_link_download` は共有リンク実装時の拡張用として扱い、current support の同意導線として読まない
- 法務文面の妥当性や版上げの判断
- 通知、期限、再同意催促、法務承認 workflow の要否

これらは利用者向け runbook では決めず、管理画面側の runbook と仕様 docs を正本にする。

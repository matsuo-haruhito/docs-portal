# 文書一括編集dry-run運用runbook

この文書は issue `#1505` に対応する、`docs-portal` の `文書一括編集` 運用メモです。

## 1. この runbook が扱う画面

この runbook は、internal admin 向けの文書一括編集 dry-run / 実行画面を扱います。

- `GET /admin/bulk_edit_dry_runs/new`
- `POST /admin/bulk_edit_dry_runs`
- `GET /admin/bulk_edit_dry_runs/:public_id`
- `PATCH /admin/bulk_edit_dry_runs/:public_id`

一括編集は、対象文書と変更内容を選んで `事前確認` を作成し、確認画面で preview summary、警告、エラー、変更前後を見てから `確認して実行` する運用です。`admin/documents` の文書マスタ一覧は、検索・状態確認・行単位操作の入口として使い、一括編集そのものの正本はこの画面に分けます。

## 2. 最初の切り分け順

1. まず `文書マスタ` で対象文書の案件、文書名、URL識別子、公開範囲、状態を確認する
2. 同じ変更を複数文書へ適用する必要がある場合だけ、`文書一括編集` に進む
3. `対象文書` で変更したい文書だけを選ぶ。対象が多い場合は画面内の検索、選択済み件数、`選択済みだけ表示` を使って選択漏れや選びすぎを見直す
4. `変更内容` で、実際に変えたい項目だけを入力する。空欄や `変更しない` の項目は変更対象にしない
5. `事前確認を作成` し、確認画面の概要、警告、エラー、明細を読む。`警告 / エラー` が 0 でない場合は、概要の案内どおり明細の `確認目安` で `エラーあり` / `警告あり` の行から先に見る。警告・エラーの表示欄は調査入口の preview であり、secret-like value、private-looking path、長い診断文は mask / truncate されることがある
6. エラーがなく、対象と変更後の値が意図どおりと判断できる場合だけ `確認して実行` する
7. 実行後は `実行結果` の成功 / 失敗 / スキップを確認し、必要なら文書マスタや公開側文書へ戻る

## 3. 対象文書の選び方

`対象文書` には案件名、文書名、カテゴリ、種別、公開範囲、状態が表示されます。複数文書を同時に選べますが、選択前に次を揃えてください。

- 同じ業務目的で変更してよい文書か
- 案件をまたぐ場合に、同じメタデータ値を入れてよいか
- アーカイブ済み文書を含める場合、復元や再アーカイブの意図が明確か
- 公開範囲や snapshot 属性を変える場合、利用者への見え方を確認済みか

対象選択画面には、選択済み件数と現在表示中の件数が表示されます。検索欄は案件名、文書名、カテゴリ、種別、公開範囲、状態の表示テキストを対象にした画面内の表示補助です。検索しても backend query や submit 対象は変わらず、checkbox で選んだ `bulk_edit[document_ids][]` だけが dry-run 作成対象になります。

`選択済みだけ表示` は、すでに選んだ行を見直すための表示切り替えです。行が検索条件で一時的に見えなくても、checkbox の選択状態そのものを解除する操作ではありません。`事前確認を作成` の前に、選択済み件数と表示中の行を見比べて、意図しない案件や公開範囲の文書が混ざっていないか確認します。

対象文書を 1 件も選ばないと dry-run は作成されません。権限外・変更不能な対象が混ざる可能性は、dry-run の警告 / エラーで確認します。

## 4. 入力できる変更内容

### 文書メタデータ

次の `Document` 属性を一括で変更できます。

- `カテゴリ`
- `文書種別`
- `公開範囲`
- `重要度`
- `推奨表示順`
- `保管期限`
- `廃棄候補日時`

公開範囲や期限系の値は、公開側の見え方や運用判断に影響します。特に `公開範囲`、`保管期限`、`廃棄候補日時` は、文書マスタで対象を絞ってから dry-run へ進む方が安全です。

### 最新版属性

次の `latest_version` 属性を一括で変更できます。

- `スナップショット種別`
- `公開開始日時`
- `公開終了日時`

対象文書に最新版がない、または入力値が current code の validation に合わない場合は、dry-run 明細や実行結果のエラーで確認します。

### タグ追加 / 削除

`追加するタグ` と `削除するタグ` は、カンマ、読点、改行区切りで複数指定できます。

- 追加時は、既存タグがなければタグを作成し、文書へ紐づけます
- 削除時は、該当タグが存在する場合だけ文書との紐づけを外します
- 同じタグを追加と削除の両方へ入れないよう、dry-run 前に入力を見直してください

### アーカイブ操作

`アーカイブ操作` は次の 3 つです。

- `変更しない`
- `アーカイブする`
- `アーカイブを解除する`

アーカイブは通常導線から外す操作、復元は運用対象へ戻す操作です。削除や retention policy の最終判断ではありません。

## 5. 事前確認の読み方

### 概要

確認画面の `概要` では次を見ます。

- `事前確認ID`: 問い合わせや監査ログ確認時の識別子
- `状態`: dry-run が実行可能な状態か
- `作成者`: dry-run を作った admin user
- `対象件数`: 選択した文書数
- `変更予定件数`: 実際に変更が発生する文書数
- `警告 / エラー`: dry-run 全体と明細に出た警告・エラー数
- `有効期限`: dry-run を実行できる期限

current code では dry-run の有効期限は既定で作成から 1 日です。期限切れ後は同じ dry-run を実行せず、最新状態で作り直します。

`警告 / エラー` が 0 でない場合、概要 card には「警告・エラーのある行を先に確認」する案内が出ます。この案内は実行可否を自動判定するものではなく、次に見る明細行の優先順を示す表示補助です。

### 警告 / エラー

`警告` は、実行前に人が確認すべき注意です。警告があるだけで必ず停止とは限りませんが、対象文書と変更後の値が意図どおりかを確認します。

`エラー` は、実行しても失敗する可能性が高い状態です。エラーがある場合は `確認して実行` に進まず、対象や入力値を直して別の事前確認を作成します。

`警告` / `エラー` に表示される文字列は、保存値や外部 path の raw 照合場所ではなく、一次切り分けのための safe preview です。token、secret、authorization、password などの key / value、private-looking path、長すぎる診断文は画面上で mask / truncate されうるため、preview だけで原文や保存 payload を断定しないでください。件数、確認目安、変更前後、作り直し導線を見て、実行できる状態かを判断します。

### 事前確認明細

`事前確認明細` では、文書ごとに次を見ます。

- `文書名`: 対象文書
- `確認目安`: 行ごとの優先確認 cue。`エラーあり`、`警告あり`、`変更予定`、`変更なし` のいずれかが表示される
- `変更フィールド`: 実際に差分が出る項目
- `変更前`: 現在値
- `変更後`: dry-run 入力を適用した後の値
- `警告`: 文書単位の注意
- `エラー`: 文書単位の失敗理由

`確認目安` は表示補助です。`エラーあり` はその行の `エラー` を先に読み、`警告あり` は `警告` と変更前後を確認します。`変更予定` は差分がある行、`変更なし` は入力値を適用しても差分がない行です。

明細行の `警告` / `エラー` も safe preview として表示されます。raw 値の照合ではなく、どの行を止めるか、どの入力や対象を作り直すかを決める入口として読みます。

`変更フィールド` が `-` の文書は、入力値を適用しても差分がありません。`確認目安` が `変更なし` の行が想定より多い場合は、入力値や対象選択を見直してください。

## 6. 実行してよい状態 / 止める状態

### 実行してよい状態

- 対象件数と文書名が想定どおり
- `変更予定件数` が作業目的と一致している
- `確認目安` の `変更予定` / `変更なし` の行を確認し、意図しない unchanged がない
- `変更後` のカテゴリ、種別、公開範囲、最新版属性、タグ、アーカイブ状態が期待どおり
- `警告あり` の行は警告の内容を人が確認し、実行してよいと判断できる
- `エラーあり` の行がない
- 有効期限内で、dry-run 作成後に対象文書の状態が大きく変わっていない

### 止める状態

- 対象文書が多すぎる、または案件をまたいだ理由を説明できない
- `公開範囲`、`公開開始日時`、`公開終了日時` の変更がレビューされていない
- タグの追加 / 削除で、同じタグ名や表記ゆれが混ざっている
- `アーカイブする` / `アーカイブを解除する` の意図が文書ごとに違う
- `確認目安` が `エラーあり` の行が 1 件でもある
- dry-run の有効期限が切れている
- dry-run 作成後に対象文書や最新版が更新されている可能性がある

止める場合は `別の事前確認を作成` から作り直します。既存 dry-run の入力をあとから編集する画面、条件保存、予約実行は current support ではありません。

## 7. 実行結果の読み方

実行後は同じ確認画面に `実行結果` が表示されます。

- `success`: 変更が適用された
- `failed`: 実行時に失敗した。対象文書が削除済み、validation 失敗、保存失敗などを確認する
- `skipped`: dry-run の再計算時点で差分がなく、変更しなかった

実行結果の table では、事前確認明細と同じように文書単位の警告 / エラーを確認します。ここでも警告 / エラーは safe preview であり、raw secret-like value や private path を確認する場所ではありません。事前確認時点の `確認目安` は実行前の優先確認 cue であり、実行結果の `success` / `failed` / `skipped` を置き換えるものではありません。

実行後は `BulkEditDryRun` に実行 summary と明細が保存され、`AccessLog` には `action_type=bulk_edit`、`target_type=BulkEditDryRun`、dry-run ID と件数 summary が残ります。監査ログで追うときは事前確認IDを手がかりにします。

## 8. 文書マスタ一覧との責務分担

### 文書マスタ一覧でやること

- 対象文書を探す
- 現在の状態、公開範囲、保管期限、廃棄候補を確認する
- 1 件だけの登録 / 編集 / アーカイブ / 復元 / 削除を判断する
- 公開側文書へ戻って見え方を確認する

### 文書一括編集でやること

- 複数文書へ同じメタデータ変更を適用する
- 対象選択画面の検索、選択済み件数、選択済みだけ表示で dry-run 対象を見直す
- 最新版属性、タグ、アーカイブ / 復元をまとめて dry-run する
- 実行前に確認目安、変更前後、警告 / エラーを一覧で確認する
- 実行結果を文書ごとに確認する

文書一括編集は、文書マスタ一覧の代替ではありません。対象を探す、公開側で確認する、1 件だけ直す、という作業は文書マスタ一覧から行います。

## 9. current support の境界

- 一括編集は admin user のみが使う画面です
- operation type は current code では `document_metadata` を扱います
- 対象は `Document` メタデータ、`latest_version` の一部属性、タグ追加 / 削除、archive / restore です
- 対象選択画面の検索と `選択済みだけ表示` は client-side の表示補助です。server-side filter、条件保存、文書マスタ一覧からの条件引き継ぎではありません
- dry-run は保存され、確認画面から同期的に実行します
- dry-run の既定有効期限は 1 日です
- 実行時は現在の文書状態で plan を再計算し、失敗・スキップ・成功を結果に残します
- 警告 / エラーの表示は safe preview であり、secret-like value、private-looking path、長い診断文は mask / truncate されることがあります。保存済み payload、raw path、secret の照合導線としては扱いません
- CSV import/export、条件保存、予約実行、非同期実行、rollback 実行、bulk delete は current support として書きません
- retention policy の最終判断、自動削除、非可逆 discard は別 issue / 別 runbook の領域です

# 文書利用状況運用runbook

この runbook は、管理画面の `文書利用状況` を日常運用で見返すときの入口をまとめる。

新しい KPI や集計ルールはここでは定義しない。current 実装を前提に、案件単位の集計をどう読み、必要に応じてどこへ戻るかを整理する。

## 先に見るもの

1. 文書公開モデルや権限前提を確認したいときは [アプリケーション仕様](./アプリケーション仕様.md)
2. アクセス証跡の見方を先に確認したいときは [監査ログ運用runbook](./監査ログ運用runbook.md)
3. 管理画面の位置関係は `app/views/admin/_nav.html.slim` の `文書利用状況` を起点に確認する

## 画面の役割

`admin/document_usage_reports` は internal admin だけが使える案件単位の集計画面で、文書ごとの利用有無と総量をざっと見返すために使う。

current 実装の前提:

- 最初に `project_id` で案件を 1 つ選んでから集計する
- `案件` は remote search 対応の combobox で、案件コードまたは案件名を 100 文字までに正規化して検索し、候補は最大 20 件表示する。空検索時も先頭 20 件を候補として返す
- URL や戻り導線で選択済みの案件が現在の候補 20 件に含まれない場合でも、`selected_project` の復元 endpoint で `code / name` の selected option として戻る。存在しない、または削除済みの `project_id` は選択済み案件として解決されない。集計や CSV は案件横断には広がらず、案件未選択時と同じ確認順へ戻す
- 集計対象は選んだ案件に属する文書で、`利用状況` は `すべて / 利用あり / 未利用` から切り替える
- `文書名 / slug` は、選択案件内の文書 title または slug に含まれる検索語で絞り込む。画面入力と server-side の上限は 100 文字で揃っており、長すぎる検索語は正規化後の検索語として表示・集計・CSV 条件に使う
- `並び順` は `タイトル順 / 最終アクセスが新しい順 / 最終アクセスが古い順` を選べる
- `開始日` / `終了日` を指定すると、指定した日の 0:00:00 から 23:59:59 までを日単位で集計する
- 不正な日付入力は 500 にせず、その条件だけ無視する
- 集計条件 form 上部の `すべての条件をクリア` は、解除できる条件があるときだけ表示される。案件選択も含めて `文書名 / slug`、期間、`利用状況`、`並び順` を初期状態へ戻す導線として読む。案件未選択かつ `文書名 / slug`、期間、非 default の `利用状況` / `並び順`、無効日付 warning がない初期状態では表示されない
- `利用あり` は、現在の期間条件を適用した後に `閲覧` `ダウンロード` `既読確認` のどれかが 1 回以上ある文書を指す
- `最終アクセス` は、現在の期間条件を適用した AccessLog の `accessed_at` を元にした最新時刻で、既読確認時刻はここには入らない
- 画面は `集計条件`、`集計サマリ`、文書ごとの一覧の順で読む。文書が 1 件以上あるときだけ `文書利用一覧の表示設定` が一覧の直前に出る
- 案件選択後は、集計サマリの `CSV本体` action group にある `CSV出力` から、同じ案件・文書名 / slug 検索・利用状況・並び順・期間条件の一覧を CSV として出力できる。案件未選択時や invalid `project_id` では全件出力せず、画面へ戻して案件選択を促す
- `条件確認用 metadata` action group の `JSON metadataを確認` は、CSV と同じ正規化後の条件、無視された日付条件、行数、summary を JSON で確認する補助導線である。CSV 本体、非同期 export、全件 export、案件横断レポート、KPI 定義変更として読まない

## 期間指定との読み分け

文書利用状況の期間指定は、案件内の文書ごとの利用傾向を「その期間に限って」見るための first slice である。誰がいつ何を見たかを 1 件ずつ追う正本は引き続き監査ログを使う。

期間指定時の集計対象:

- `閲覧`: 期間内の AccessLog `view` 件数
- `ダウンロード`: 期間内の AccessLog `download` 件数
- `既読確認`: 期間内の ReadConfirmation 件数
- `利用あり` / `未利用`: 期間内の閲覧・ダウンロード・既読確認の有無
- `最終アクセス`: 期間内 AccessLog の最新 `accessed_at`

期間指定なしの場合は、案件全体の累積集計として読む。期間指定ありの場合は、過去に利用実績がある文書でも、指定期間内に実績がなければ `未利用` 側に回る。画面上の `未利用` hint も、期間外の実績を含まない補助表示として読む。

監査ログへのリンクは、current main では案件または文書の既存絞り込みだけを引き継ぐ。文書利用状況で指定した `開始日` / `終了日` は、監査ログ側の期間 filter が merge されるまで先取りして渡さない。

## 集計サマリの見方

### 1. 期間

- `指定なし（案件全体の累積）` は、期間条件を適用していない状態
- `YYYY-MM-DD 以降` は開始日だけ指定した状態
- `YYYY-MM-DD まで` は終了日だけ指定した状態
- `YYYY-MM-DD から YYYY-MM-DD まで` は開始日と終了日の両方を指定した状態

期間指定時は、集計サマリの数字、一覧の `利用`、`閲覧`、`ダウンロード`、`既読確認`、`最終アクセス` がすべて同じ期間条件に従う。

### 2. 文書数 / 利用あり / 未利用

- `文書数` は案件配下の文書総数
- `利用あり` は、現在の期間条件で少なくとも 1 つの利用実績がある文書数
- `未利用` は、現在の期間条件で閲覧・ダウンロード・既読確認が 1 度もない文書数

### 3. 閲覧 / ダウンロード / 既読確認

- `閲覧` と `ダウンロード` は AccessLog 由来の件数
- `既読確認` は ReadConfirmation の件数
- 期間指定なしでは累積、期間指定ありでは指定期間内の件数として読む

### 4. 案件の監査ログへ / 案件の既読確認内訳へ / CSV本体 / 条件確認用 metadata

- `案件の監査ログへ` は、選択中の案件全体を `project_id` で監査ログに戻す導線。上部の案件リンクは、特定の文書だけでなく案件全体のアクセス証跡を調べ始める入口として読む
- `案件の既読確認内訳へ` は、選択中の案件全体の ReadConfirmation 明細へ戻る導線。文書利用状況の AccessLog 期間集計とは別物として読む
- `CSV本体` の `CSV出力` は、現在の `project_id`、`q`、`usage_filter`、`sort_order`、`from`、`to` を引き継いで、表示中の文書利用一覧と同じ条件の CSV を出す導線。`q` は画面に表示されている正規化後の検索語を使う。画面上の補助文どおり、棚卸し用ファイルとして読む
- `条件確認用 metadata` の `JSON metadataを確認` は、同じ `project_id`、正規化後の `q`、`usage_filter`、`sort_order`、`from`、`to` を引き継いで、CSV と同じ条件の棚卸し metadata を JSON で確認する補助導線。`report_type`、`export_scope`、説明文、filters、無視された日付条件、行数、summary を見るためのもので、CSV 行そのものは含まない
- CSV と JSON metadata はどちらも案件選択後だけ使う。案件未選択時や invalid `project_id` では案件横断の CSV / JSON export にはならず、画面へ戻して案件選択を促す
- CSV は `文書名`、`slug`、`カテゴリ`、`種別`、`公開範囲`、`利用`、`閲覧`、`ダウンロード`、`既読確認`、`最終アクセス` の列で出力される。列の表示設定を CSV の集計対象変更として扱わない
- JSON metadata の `row_count` と `summary` も、表示列設定ではなく現在の filter 済み一覧を基準に読む。table preferences で画面上の列を隠していても、CSV / JSON metadata の対象条件は変わらない
- CSV の filename は `document-usage-report-<案件コードまたはpublic_id>-<日付>.csv` で、案件未選択時や invalid `project_id` では全件 CSV にならず `CSV出力には案件選択が必要です。` として画面に戻る

### 5. 表示中 / 利用状況 / 並び順

- `表示中` は、現在の `利用状況` filter と `文書名 / slug` 検索を適用した後に一覧へ残っている文書数
- `利用状況` は今どの filter で見ているかをそのまま表示する
- `並び順` は current 一覧の並べ替え条件を表す。`最終アクセス` 系を選んだときは、時刻がある文書を並べたあと、アクセス未記録の文書が末尾に続く
- `検索` は、文書 title または slug に含まれる検索語を表す。長い検索語は画面と server-side の上限に揃えて短くなり、集計サマリ、0 件 state、CSV 出力、JSON metadata でも同じ正規化後の検索語として扱う

## 監査ログ導線の使い分け

文書利用状況には、案件単位の導線と文書単位の導線がある。どちらも AccessLog を確認する入口であり、ReadConfirmation の内訳画面ではない。

- 集計サマリの `案件の監査ログへ`: 選択中の案件に絞った監査ログへ戻る。閲覧・ダウンロードの対象文書をまだ絞り込まず、案件全体のアクセス証跡を確認したいときに使う
- 文書名の下に出る行内 `監査ログへ`: その文書に閲覧またはダウンロードがあるときだけ表示される。リンク先は選択中の案件に加えて文書 slug を `document_q` として渡し、文書単位の実績へ進む
- 既読確認だけの文書: `利用` は `あり` になるが、行内 `監査ログへ` は表示されない。既読確認自体は AccessLog の文書アクセス証跡ではなく、`最終アクセス` にも入らないためである

既読確認だけの文書を見つけた場合は、監査ログ上の閲覧・ダウンロードが無い文書として扱う。既読確認の確認日時、確認者、会社、文書 slug を追いたいときは `admin/read_confirmations` の `既読確認内訳` を使う。

## 既読確認内訳の見方

`admin/read_confirmations` は、ReadConfirmation の内訳を案件単位で確認する internal admin 向けの画面である。文書利用状況の集計で `既読確認` が動いている文書を見つけたあと、確認日時や確認者を新しい順で見返す入口として使う。

current 実装の前提:

- `project_id` で案件を選ばない限り、既読確認の一覧は表示しない
- `案件` の候補 select は文書利用状況と同じく上限付きで、基本候補は案件名 / id 順の先頭 100 件。選択済み案件が上限外でも、その案件は候補に戻されて `対象条件` や戻り導線に残る
- invalid / deleted `project_id` は選択案件として扱わない。CSV も案件横断にはせず、案件選択を促す画面へ戻す
- `document_slug` は任意。空欄または whitespace-only なら未指定として扱い、選択した案件内の既読確認を新しい順に表示する
- `document_slug` 入力欄は `文書名またはURL識別子` と表示され、直下の `最大100文字まで。選択した案件内の文書名またはURL識別子の一部一致で探します。` を入力前の cue として読む。指定した場合は、前後空白を除去した検索語を 100 文字までに正規化し、選択した案件に属する文書名または slug の一部に一致する文書へ絞り込む。画面表示、CSV link、CSV 出力条件もこの正規化後の値を使う
- `document_slug` が複数文書に部分一致した場合は、対象条件に `一致文書: N件` と先頭 5 件までの候補が出る。1 件に絞りたいときは、候補の文書名または URL 識別子を見て検索語を追加する
- `確認者` と `会社` は任意。どちらも選択案件内で既読確認がある候補だけを表示し、併用した場合は両方に一致する既読確認だけを表示する
- `会社` 候補は、選択案件内で既読確認がある会社に閉じる。初期候補は上限 50 件、検索結果は 20 件で、検索語は会社名またはドメインを 100 文字までに正規化して探す
- `確認者` 候補は、選択案件内で既読確認がある user に閉じる。初期候補は上限 50 件、検索結果は 20 件で、検索語は確認者名、メール、会社名、ドメインを 100 文字までに正規化して探す。会社を指定している場合、確認者候補と selected user 復元もその会社境界を尊重する
- URL や戻り導線で指定済みの会社 / 確認者が候補上限外にある場合でも、選択案件内の既読確認候補として解決できる値は selected option として復元される。案件外、削除済み、会社条件外の値は復元されず、0 件理由の表示へ戻す
- `既読確認日時（開始日）` / `既読確認日時（終了日）` は ReadConfirmation の `confirmed_at` を日単位で絞り込む。文書利用状況の閲覧・ダウンロード集計期間とは別条件として読む
- 指定した文書名または slug が選択案件に見つからない場合は、文書候補なしとして 0 件扱いにする
- 指定した確認者または会社が選択案件の既読確認候補に見つからない場合も 500 にはせず、0 件扱いで理由を表示する
- 一覧は `confirmed_at desc, id desc` の新しい順に、1ページ200件まで表示する。条件一致が200件を超える場合は `前へ` / `次へ` で page 移動し、2ページ目以降も同じ sort order で続きの明細を見る
- `表示中` は現在ページに出ている件数、`表示範囲` は条件一致全体の何件目を見ているか、`条件一致` は filter 後の総件数を表す。`Page X / Y` を見れば、最新200件入口だけでなく200件外へ進めるかを判断できる
- CSV 出力は、同じ `project_id`、`document_slug`、会社、確認者、既読確認日時 filter と、現在の page 範囲を反映する。条件一致全体や非同期 export ではなく、画面で見ている page の棚卸しとして読む
- 表示列は `確認日時`、`文書`、`確認者`、`会社`、`文書URL識別子` で、table preferences により画面上の列表示だけを切り替えられる。filter、page、1ページ200件境界、CSV の対象範囲は table preferences では変わらない
- `文書利用状況へ戻る` は、選択中の `project_id` を引き継いで案件単位の文書利用状況へ戻る導線である。`document_slug` 指定中だけ、正規化後の検索語を文書利用状況の `q` として渡す。既読確認日時の `from` / `to`、`company_id`、`user_id`、`document_slug` param 自体、`page` は文書利用状況へ引き継がない

文書利用状況の行内 `監査ログへ` が出ない文書でも、`既読確認` 件数がある場合は `既読確認内訳` で ReadConfirmation 側の明細を確認できる。逆に `既読確認内訳` は AccessLog の閲覧・ダウンロード明細を出す画面ではないため、閲覧者のアクセス証跡を追うときは監査ログへ戻す。

## 文書利用一覧の表示設定

`文書利用一覧の表示設定` は、一覧で見たい列だけを残して案件単位の傾向を読みやすくするための editor である。

この設定で変わるのは一覧テーブルの見え方だけで、次は変わらない。

- 選択中の案件
- 文書名 / slug 検索
- 期間条件
- `利用状況` filter
- `並び順`
- 集計サマリの数字
- CSV 出力の列と集計対象
- `JSON metadataを確認` の filters、row_count、summary

使い分けの目安:

- まず案件全体の傾向を見たいときは、`利用` `閲覧` `ダウンロード` `既読確認` `最終アクセス` を残して、利用量の偏りを見る
- 文書の棚卸しや公開設定の見直しをしたいときは、`カテゴリ` `種別` `公開範囲` を残して、どの属性の文書に偏りがあるかを読む
- 詳細画面へ戻りながら確認するときは、`文書名` を残したまま他の列を絞り、必要な文書だけを辿りやすくする
- CSV で棚卸しを渡すときは、表示設定で一時的に隠した列も CSV には出る前提で、画面の列幅や表示列ではなく集計条件を確認する
- JSON metadata を共有するときは、表示設定ではなく現在の案件・検索・利用状況・並び順・期間条件が正しいかを先に確認する

一覧で列を一時的に減らしても、上の `集計サマリ` は同じ案件・同じ期間条件の数字を出し続ける。まず summary で全体感をつかみ、次に表示設定で列を絞って individual row を読む流れを基本にすると迷いにくい。

## 一覧列の見方

一覧本体は `table_preferences_table_tag(...)` で描画され、見せる列は `文書利用一覧の表示設定` で調整できる。特に理由がなければ、戻り先になる `文書名` は表示したまま使う。

- `文書名`: project/document detail への戻り口。閲覧またはダウンロードがある文書では、同じセルに文書単位の `監査ログへ` も表示される
- `カテゴリ`: current document category の表示
- `種別`: current document kind の表示
- `公開範囲`: visibility policy の表示
- `利用`: 現在の期間条件で `あり` / `なし` を示す。既読確認だけでも `あり` になる。補助ラベルの `既読のみ` は閲覧・ダウンロードがなく既読確認だけがある状態で、既読確認内訳へ戻る cue として読む。`未利用` は期間内の閲覧・ダウンロード・既読確認がすべてない状態を示し、期間外の実績は含めない。どちらも新しい KPI ではない
- `閲覧`: 現在の期間条件での AccessLog view 件数
- `ダウンロード`: 現在の期間条件での AccessLog download 件数
- `既読確認`: 現在の期間条件での ReadConfirmation 件数。件数が 1 以上なら `内訳へ` から同じ案件と文書 slug の既読確認内訳へ戻れる
- `最終アクセス`: 現在の期間条件で最後に記録された AccessLog 時刻。アクセスが無ければ `-`

CSV では同じ列名が固定順で出る。table preferences で画面上の列を隠していても、CSV の列や集計条件は変わらない。

## 絞り込みと empty state の使い分け

- `文書名 / slug` 検索は、選択案件内の文書 title または slug に検索語を含む行だけを残す。検索語は画面と server-side で同じ上限に正規化され、長すぎる入力を使った場合でも表示中の検索語と CSV 条件、JSON metadata の `filters.q` は同じ値として読む
- `利用あり` は、公開済み文書のうち指定期間内に実際に見られているものへ絞って優先順位を付けたいときに使う
- `未利用` は、公開したのに指定期間内の反応がない文書や、周知漏れが疑われる文書を探したいときに使う
- `タイトル順` は棚卸し向き、`最終アクセス` の新しい順 / 古い順は最近動いた文書や長く触られていない文書を見たいときに向く
- `条件に一致する文書はありません。` は、選択した案件には文書があるが、現在の filter 条件で一覧に残る文書が 0 件になった状態を示す
- 0 件時は `文書利用一覧の表示設定` ではなく、現在の期間、`文書名 / slug` 検索、`利用状況`、`並び順` を見直す。案件を残して一覧条件だけ戻す場合は empty state 近くの `案件だけ残して条件を解除` を使う。この clear link は `project_id` を維持し、`文書名 / slug`、期間、`利用状況`、`並び順` を初期条件に戻す導線として読む
- `案件を選択すると集計結果を表示します。` は、まだ案件未選択で集計自体を始めていない状態を示す。この初期状態では、解除対象がないため上部の `すべての条件をクリア` は表示されない。URL に invalid / deleted `project_id` が残っている場合も、まず案件を選び直す
- `案件を選択すると既読確認の内訳を表示します。` は、`既読確認内訳` でまだ案件未選択のため ReadConfirmation を検索していない状態を示す。invalid / deleted `project_id` が残っている場合も、案件を選び直すところから始める
- `指定した文書はこの案件に見つかりません。` は、入力した `document_slug` が選択案件の文書名または slug と一致しない状態を示す。文書利用状況の文書名リンクや文書 detail の URL で slug を確認してから再入力する
- `指定した確認者はこの案件の既読確認候補に見つかりません。` と `指定した会社はこの案件の既読確認候補に見つかりません。` は、選択案件外の ID や候補外の条件を指定した状態を示す。500 にはならず、0 件として扱う
- `既読確認はありません` は、選択案件または現在の document slug / 確認者 / 会社 / 期間条件に一致する ReadConfirmation が無い状態を示す。閲覧・ダウンロードの有無は監査ログまたは文書利用状況で確認する
- 既読確認内訳で page が総ページ数を超える場合は、current 実装では最終 page に丸められる。0 件時は表示範囲も `0-0件目` と読み、条件一致があるのに前後 link が出ない場合は `Page 1 / 1` の範囲に収まっている状態として扱う

## 日常確認ポイント

- 案件候補に目的の案件がすぐ出ないときは、候補が remote search の最大 20 件に閉じていることを前提に、案件コードまたは案件名の特徴的な断片で検索し直す。既存の戻り導線や URL の `project_id` から選択済み案件が復元されているかも確認し、invalid / deleted `project_id` なら案件を選び直す
- 初期状態で `すべての条件をクリア` が見当たらないときは、解除対象がない状態として読む。案件、検索語、期間、非 default の利用状況 / 並び順、無効日付 warning のいずれかがある場合だけ上部 clear link が出る
- 案件内で、公開中の文書に未利用が偏っていないか
- 文書名 / slug 検索で対象文書を絞ったとき、表示中の検索語と CSV 出力条件、JSON metadata の `filters.q` が同じになっているか
- 送付後 1 週間や問い合わせ前後など、期間を区切ったときに閲覧・ダウンロード・既読確認が動いているか
- `利用あり` と `最終アクセスが古い順` を組み合わせたときに、長く更新確認されていない文書がないか
- 閲覧だけ多い文書と、ダウンロードまで進む文書の差が大きすぎないか
- 既読確認が必要な運用で、確認数が極端に少ない文書がないか
- 既読確認の確認者や会社を見たい場合、文書利用状況の件数だけで判断せず `既読確認内訳` に戻る必要があるか
- `既読確認内訳` で会社や確認者が候補にすぐ出ない場合、候補が選択案件内の既読確認に閉じた remote search であることを前提に、会社名 / ドメイン、確認者名 / メール / 会社名 / ドメインで検索し直す必要があるか
- `既読確認内訳` で条件一致が200件を超える場合、最初の page だけで判断せず `次へ` で200件外の確認日時・確認者・会社も見直す必要があるか
- `既読確認内訳` で CSV を使うときは、現在 page の範囲だけが出ることを前提に、必要なら page を移動してから出力する必要があるか
- `既読確認内訳` で文書名または URL 識別子が複数文書に一致したときは、候補の先頭 5 件を見て検索語を狭める必要があるか
- 気になる文書があれば文書 detail へ戻り、公開状態や添付構成を確認する必要があるか
- 一覧が広くて読みづらいときは、表示設定で列を絞ったうえで同じ案件・同じ期間・同じ filter を見直す必要があるか
- `利用あり` なのに行内 `監査ログへ` が無い文書は、既読確認だけで利用ありになっていないかを切り分ける必要があるか
- CSV を使うときは、画面と同じ案件・同じ検索語・同じ期間・同じ利用状況 filter の棚卸し用出力であり、表示列設定や新しい集計定義の反映ではないことを確認する
- `JSON metadataを確認` を使うときは、CSV と同じ filter 条件、無視された日付条件、行数、summary を共有・棚卸しする補助出力であり、CSV 本体や全件 export ではないことを確認する

## 変更時の注意

- current UI は案件を 1 つずつ見る画面で、会社横断や利用者横断の比較画面ではない
- 案件の remote search は全案件を無制限に初期表示する入口ではない。候補は最大 20 件で、検索語は案件コードまたは案件名を 100 文字までに正規化して探す。選択済み案件が候補上限外の場合は selected option として復元されるが、この候補上限を CSV 対象や集計対象の上限として読まない
- invalid / deleted `project_id` は案件未選択と同じ安全側に寄せて扱う。docs では案件横断集計や案件横断 CSV / JSON metadata export が実行されるように書かない
- `文書名 / slug` 検索は title / slug の部分一致に閉じる。full-text search、本文検索、案件横断検索、CSV 専用検索条件を current support として書かない
- 長い検索語は画面入力と server-side の上限に揃えて正規化される。集計サマリ、0 件 state、CSV link / CSV 出力条件、JSON metadata の filters は、raw request params ではなく画面に表示された正規化後の検索語を基準に読む
- 上部の `すべての条件をクリア` は解除できる条件がある場合だけ表示される。default の `usage_filter=all` と `sort_order=title` だけを解除対象として扱わず、表示されていない clear link を missing action として書かない
- `既読確認内訳` も案件を 1 つずつ見る画面で、document slug、確認者、会社、既読確認日時の filter は選択案件内の明細を絞るためだけに使う
- 会社 / 確認者候補は selected value 復元を含めて、選択案件内で既読確認がある値だけを扱う。候補初期表示 50 件、検索結果 20 件、検索語 100 文字の境界を、CSV 対象、pagination 対象、案件横断検索、user / company master 全体検索の上限として読まない
- `既読確認内訳` は filter 後の明細を1ページ200件で表示し、`page` は正の整数として扱われる。存在しない page を current support のエラーや別条件として書かず、最終 page へ丸められる前提で説明する
- `既読確認内訳` の CSV は current page の `@read_confirmations` を出す。filter 全体 export、全ページ一括 export、非同期 export job、CSV 専用 query を current support として書かない
- `既読確認内訳` で `document_slug` を指定して `文書利用状況へ戻る` を使うと、文書利用状況の `q` に同じ正規化後の検索語だけが渡る。既読確認日時、会社、確認者、page filter を文書利用状況の期間条件や利用条件として引き継ぐように書かない
- 確認者と会社の候補は、選択案件内で既読確認がある値に閉じる。案件外 ID や候補外 ID を指定しても、他案件の既読確認を混ぜない
- `最終アクセス` はアクセスログ由来なので、既読確認だけが行われた文書では空欄のままになりうる
- 行内 `監査ログへ` は閲覧またはダウンロードがある文書だけに出る。既読確認だけの文書にも表示される前提で docs を書かない。文書行の他のリンクも、その文書で確認できる実績がある場合だけ表示される補助導線として扱う
- `文書利用一覧の表示設定` は一覧の表示列を切り替えるだけで、集計対象や利用判定そのものは変えない
- CSV は current HTML と同じ filter 済み `@report_hash[:documents]` を出す。CSV 専用の集計、案件横断 export、table preferences 連動、非同期 export job を current support として書かない
- JSON metadata は current HTML / CSV と同じ filter 済み `@report_hash[:documents]` の条件、無視された日付条件、行数、summary を返す補助出力として読む。JSON schema の変更、案件横断 metadata、table preferences 連動、非同期 export job を current support として書かない
- 集計ロジックや KPI 定義を増やす docs ではなく、いまの画面の読み方だけに留める
- 監査ログ一覧側の期間 filter 実装は #1175 / PR #1285 に委ねる。未 merge の間は、文書利用状況から監査ログへ期間 params を先取りして渡さない
- company master admin や external user は使えない画面なので、社外利用者向け runbook と混ぜない

## 迷ったときの切り分け

- 案件全体で未利用文書や利用偏りを見たい: `文書利用状況` を使う
- 案件が候補にすぐ見えない: remote search は最大 20 件ずつの候補表示なので、案件コードまたは案件名で検索し直す。既存の文書 detail や既読確認内訳から戻る導線で `project_id` が選択済みになっているかを確認し、invalid / deleted `project_id` なら案件を選び直す
- `すべての条件をクリア` が表示されていない: 案件未選択かつ実入力条件がない初期状態、または default の `usage_filter=all` / `sort_order=title` だけの状態として読む。条件を解除したい場合は、案件や検索語、期間、非 default filter が実際に入っているかを先に確認する
- 文書名 / slug で対象文書を探したい: `文書利用状況` の `文書名 / slug` 検索を使う。長い検索語は画面上の表示値に揃えて短くなるため、CSV と JSON metadata もその値で出ると読む
- 指定期間内の利用偏りを見たい: `文書利用状況` で `開始日` / `終了日` を指定する
- 同じ条件の棚卸しを外へ持ち出したい: 案件選択後に `CSV出力` を使う。表示設定で隠した列や列幅ではなく、現在の集計条件を確認する
- 同じ条件の共有用 metadata を確認したい: 案件選択後に `JSON metadataを確認` を使う。CSV と同じ案件・検索・利用状況・並び順・期間条件、無効日付の扱い、行数、summary を見る補助出力であり、CSV 本体や全件 export ではない
- 誰がいつ見たかを追いたい: `監査ログ` に戻る。案件全体なら集計サマリの `案件の監査ログへ`、文書単位の閲覧・ダウンロードなら行内 `監査ログへ` を使う
- その文書が見える理由や公開条件を確認したい: 仕様 docs と文書 detail に戻る
- 既読確認だけの利用状況を見たい: 行内 `監査ログへ` が無いことを AccessLog 未記録のサインとして読み、`既読確認内訳` で確認日時・確認者・会社・文書 slug を確認する
- `既読確認内訳` で会社または確認者を絞りたい: 案件を選んだうえで remote search を使い、会社名 / ドメイン、確認者名 / メール / 会社名 / ドメインで探す。候補上限外でも選択済み値が案件内の既読確認候補なら復元されるが、案件外や会社条件外の値は 0 件として扱う
- `既読確認内訳` で条件一致が200件を超える: `表示範囲` と `Page X / Y` を確認し、`次へ` で200件外の明細へ進む。CSV は現在 page の範囲だけなので、必要な page で出力する
- `既読確認内訳` で文書名または URL 識別子を入れたときに複数文書へ一致する: 先頭 5 件の候補を見て、1 件に絞るなら文書名または slug を追加する
- `既読確認内訳` から文書利用状況へ戻りたい: `document_slug` 指定中は正規化後の検索語だけが文書利用状況の `q` に渡る。既読確認日時、会社、確認者、page 条件は戻り先へ渡らないため、文書利用状況側で必要な期間や利用状況 filter を選び直す
- `既読確認内訳` で 0 件になる: 案件未選択か、選択案件内に ReadConfirmation が無いか、入力した `document_slug` が選択案件の文書名または slug と一致していないか、確認者・会社・期間条件が狭すぎないかを分けて見る
- 古いアクセス証跡を期間で追いたい: #1175 / PR #1285 の merge 状況を確認し、未 merge なら current 監査ログの最新 200 件制限を前提にする
- 期間比較や高度な集計をしたい: current 画面の責務外として扱う

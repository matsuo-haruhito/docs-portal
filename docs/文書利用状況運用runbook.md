# 文書利用状況運用runbook

この runbook は、管理画面の `文書利用状況` を日常運用で見返すときの入口をまとめる。

新しい KPI や集計ルールはここでは定義しない。current 実装を前提に、案件単位の集計をどう読み、必要に応じてどこへ戻るかを整理する。

## 先に見るもの

1. 文書公開モデルや権限前提を確認したいときは [アプリケーション仕様](./アプリケーション仕様.md)
2. アクセス証跡の見方を先に確認したいときは [監査ログ運用runbook](./監査ログ運用runbook.md)
3. 管理画面の位置関係は `app/views/admin/_nav.html.slim` の `文書利用状況` を起点に確認する

## 画面の役割

`admin/document_usage_reports` は internal admin だけが使える案件単位の集計画面で、文書ごとの利用有無と総量をざっと見返すために使う。

current 実装の前提:

- 最初に `project_id` で案件を 1 つ選んでから集計する
- 集計対象は選んだ案件に属する文書で、`利用状況` は `すべて / 利用あり / 未利用` から切り替える
- `文書名 / slug` は、集計済みの文書行を文書名または slug の断片でさらに絞るための検索欄で、案件未選択時に案件横断検索としては動かない
- `文書名 / slug` の検索は、文書ごとの集計値を作った後の行 filter として扱う。閲覧・ダウンロード・既読確認の集計定義は変えない
- `並び順` は `タイトル順 / 最終アクセスが新しい順 / 最終アクセスが古い順` を選べる
- `開始日` / `終了日` を指定すると、指定した日の 0:00:00 から 23:59:59 までを日単位で集計する
- 不正な日付入力は 500 にせず、その条件だけ無視する
- `利用あり` は、現在の期間条件を適用した後に `閲覧` `ダウンロード` `既読確認` のどれかが 1 回以上ある文書を指す
- `最終アクセス` は、現在の期間条件を適用した AccessLog の `accessed_at` を元にした最新時刻で、既読確認時刻はここには入らない
- 画面は `集計条件`、`集計サマリ`、文書ごとの一覧の順で読む。文書が 1 件以上あるときだけ `文書利用一覧の表示設定` が一覧の直前に出る
- 案件選択後は `CSV出力` から同じ案件・文書名 / slug 検索・利用状況・並び順・期間条件の一覧を CSV として出力できる。案件未選択時は全件出力せず、画面へ戻して案件選択を促す

## 期間指定との読み分け

文書利用状況の期間指定は、案件内の文書ごとの利用傾向を「その期間に限って」見るための first slice である。誰がいつ何を見たかを 1 件ずつ追う正本は引き続き監査ログを使う。

期間指定時の集計対象:

- `閲覧`: 期間内の AccessLog `view` 件数
- `ダウンロード`: 期間内の AccessLog `download` 件数
- `既読確認`: 期間内の ReadConfirmation 件数
- `利用あり` / `未利用`: 期間内の閲覧・ダウンロード・既読確認の有無
- `最終アクセス`: 期間内 AccessLog の最新 `accessed_at`

期間指定なしの場合は、案件全体の累積集計として読む。期間指定ありの場合は、過去に利用実績がある文書でも、指定期間内に実績がなければ `未利用` 側に回る。

監査ログへのリンクは、current main では案件または文書の既存絞り込みだけを引き継ぐ。文書利用状況で指定した `開始日` / `終了日` は、監査ログ側の期間 filter が merge されるまで先取りして渡さない。

## 集計サマリの見方

### 1. 期間

- `指定なし（案件全体の累積）` は、期間条件を適用していない状態
- `YYYY-MM-DD 以降` は開始日だけ指定した状態
- `YYYY-MM-DD まで` は終了日だけ指定した状態
- `YYYY-MM-DD から YYYY-MM-DD まで` は開始日と終了日の両方を指定した状態

期間指定時は、集計サマリの数字、一覧の `利用`、`閲覧`、`ダウンロード`、`既読確認`、`最終アクセス` がすべて同じ期間条件に従う。

### 2. 文書数 / 利用あり / 未利用

- `文書数` は案件配下の文書総数
- `利用あり` は、現在の期間条件で少なくとも 1 つの利用実績がある文書数
- `未利用` は、現在の期間条件で閲覧・ダウンロード・既読確認が 1 度もない文書数

### 3. 閲覧 / ダウンロード / 既読確認

- `閲覧` と `ダウンロード` は AccessLog 由来の件数
- `既読確認` は ReadConfirmation の件数
- 期間指定なしでは累積、期間指定ありでは指定期間内の件数として読む

### 4. 案件の監査ログへ / 案件の既読確認内訳へ / CSV出力

- `案件の監査ログへ` は、選択中の案件全体を `project_id` で監査ログに戻す導線
- `案件の既読確認内訳へ` は、選択中の案件全体の ReadConfirmation 明細へ戻す導線。文書利用状況の AccessLog 期間集計とは別物として読む
- `CSV出力` は、現在の `project_id`、`q`、`usage_filter`、`sort_order`、`from`、`to` を引き継いで、表示中の文書利用一覧と同じ条件の CSV を出す導線
- CSV は `文書名`、`slug`、`カテゴリ`、`種別`、`公開範囲`、`利用`、`閲覧`、`ダウンロード`、`既読確認`、`最終アクセス` の列で出力される。列の表示設定を CSV の集計対象変更として扱わない
- CSV の filename は `document-usage-report-<案件コードまたはpublic_id>-<日付>.csv` で、案件未選択時は全件 CSV にならず `CSV出力には案件選択が必要です。` として画面に戻る

### 5. 表示中 / 利用状況 / 並び順 / 検索

- `表示中` は、現在の `利用状況` filter と `文書名 / slug` 検索を適用した後に一覧へ残っている文書数
- `利用状況` は今どの filter で見ているかをそのまま表示する
- `並び順` は current 一覧の並べ替え条件を表す。`最終アクセス` 系を選んだときは、時刻がある文書を並べたあと、アクセス未記録の文書が末尾に続く
- `検索` は `文書名 / slug` 検索が適用中かどうかを示す。検索なしの場合は `なし` と表示される

## 監査ログ導線の使い分け

文書利用状況には、案件単位の導線と文書単位の導線がある。どちらも AccessLog を確認する入口であり、ReadConfirmation の内訳画面ではない。

- 集計サマリの `案件の監査ログへ`: 選択中の案件に絞った監査ログへ戻る。閲覧・ダウンロードの対象文書をまだ絞り込まず、案件全体のアクセス証跡を確認したいときに使う
- 文書名の下に出る行内 `監査ログへ`: その文書に閲覧またはダウンロードがあるときだけ表示される。リンク先は選択中の案件に加えて文書 slug を `document_q` として渡す
- 既読確認だけの文書: `利用` は `あり` になるが、行内 `監査ログへ` は表示されない。既読確認自体は AccessLog の文書アクセス証跡ではなく、`最終アクセス` にも入らないためである

既読確認だけの文書を見つけた場合は、監査ログ上の閲覧・ダウンロードが無い文書として扱う。既読確認の確認日時、確認者、会社、文書 slug を追いたいときは `admin/read_confirmations` の `既読確認内訳` を使う。

## 既読確認内訳の見方

`admin/read_confirmations` は、ReadConfirmation の内訳を案件単位で確認する internal admin 向けの画面である。文書利用状況の集計で `既読確認` が動いている文書を見つけたあと、確認日時や確認者を新しい順で見返す入口として使う。

current 実装の前提:

- `project_id` で案件を選ばない限り、既読確認の一覧は表示しない
- `document_slug` は任意。空欄なら選択した案件内の既読確認を新しい順に表示する
- `document_slug` を指定した場合は、選択した案件に属する文書 slug と一致したときだけその文書へ絞り込む
- 指定した slug が選択案件に見つからない場合は、`指定した文書URL識別子に一致する文書がないため、既読確認は表示されません。` として 0 件扱いにする
- 一覧は `confirmed_at desc, id desc` の最新 200 件までを表示する
- 表示列は `確認日時`、`文書`、`確認者`、`会社`、`文書URL識別子` で、table preferences や user / company filter は current UI にはない
- `文書利用状況へ` は、選択中の `project_id` を引き継いで案件単位の文書利用状況へ戻る導線である

文書利用状況の行内 `監査ログへ` が出ない文書でも、`既読確認` 件数がある場合は `既読確認内訳` で ReadConfirmation 側の明細を確認できる。逆に `既読確認内訳` は AccessLog の閲覧・ダウンロード明細を出す画面ではないため、閲覧者のアクセス証跡を追うときは監査ログへ戻す。

## 文書利用一覧の表示設定

`文書利用一覧の表示設定` は、一覧で見たい列だけを残して案件単位の傾向を読みやすくするための editor である。

この設定で変わるのは一覧テーブルの見え方だけで、次は変わらない。

- 選択中の案件
- 期間条件
- `文書名 / slug` 検索
- `利用状況` filter
- `並び順`
- 集計サマリの数字
- CSV 出力の列と集計対象

使い分けの目安:

- まず案件全体の傾向を見たいときは、`利用` `閲覧` `ダウンロード` `既読確認` `最終アクセス` を残して、利用量の偏りを見る
- 文書の棚卸しや公開設定の見直しをしたいときは、`カテゴリ` `種別` `公開範囲` を残して、どの属性の文書に偏りがあるかを読む
- 詳細画面へ戻りながら確認するときは、`文書名` を残したまま他の列を絞り、必要な文書だけを辿りやすくする
- CSV で棚卸しを渡すときは、表示設定で一時的に隠した列も CSV には出る前提で、画面の列幅や表示列ではなく集計条件を確認する

一覧で列を一時的に減らしても、上の `集計サマリ` は同じ案件・同じ期間条件の数字を出し続ける。まず summary で全体感をつかみ、次に表示設定で列を絞って individual row を読む流れを基本にすると迷いにくい。

## 一覧列の見方

一覧本体は `table_preferences_table_tag(...)` で描画され、見せる列は `文書利用一覧の表示設定` で調整できる。特に理由がなければ、戻り先になる `文書名` は表示したまま使う。

- `文書名`: project/document detail への戻り口。閲覧またはダウンロードがある文書では、同じセルに文書単位の `監査ログへ` も表示される
- `カテゴリ`: current document category の表示
- `種別`: current document kind の表示
- `公開範囲`: visibility policy の表示
- `利用`: 現在の期間条件で `あり` / `なし` を示す。既読確認だけでも `あり` になる。補助ラベルの `既読のみ` は閲覧・ダウンロードがなく既読確認だけがある状態、`未利用` は閲覧・ダウンロード・既読確認がすべてない状態を見分けるための表示で、新しい KPI ではない
- `閲覧`: 現在の期間条件での AccessLog view 件数
- `ダウンロード`: 現在の期間条件での AccessLog download 件数
- `既読確認`: 現在の期間条件での ReadConfirmation 件数。件数が 1 以上なら `内訳へ` から同じ案件と文書 slug の既読確認内訳へ戻れる
- `最終アクセス`: 現在の期間条件で最後に記録された AccessLog 時刻。アクセスが無ければ `-`

CSV では同じ列名が固定順で出る。table preferences で画面上の列を隠していても、CSV の列や集計条件は変わらない。

## 絞り込みと empty state の使い分け

- `文書名 / slug` 検索は、選択済み案件内で目的の文書行だけを見つけたいときに使う。検索対象は文書名と slug のみで、監査ログ本文や既読確認者を探す欄ではない
- `利用あり` は、公開済み文書のうち指定期間内に実際に見られているものへ絞って優先順位を付けたいときに使う
- `未利用` は、公開したのに指定期間内の反応がない文書や、周知漏れが疑われる文書を探したいときに使う
- `タイトル順` は棚卸し向き、`最終アクセス` の新しい順 / 古い順は最近動いた文書や長く触られていない文書を見たいときに向く
- `条件に一致する文書はありません。` は、選択した案件には文書があるが、現在の filter 条件で一覧に残る文書が 0 件になった状態を示す
- 0 件時は `文書利用一覧の表示設定` ではなく、現在の期間、`文書名 / slug` 検索、`利用状況`、`並び順` を見直す。案件全体へ戻す場合は empty state 近くの `条件をクリア` を使う
- `案件を選択すると集計結果を表示します。` は、まだ案件未選択で集計自体を始めていない状態を示す
- `案件を選択すると既読確認の内訳を表示します。` は、`既読確認内訳` でまだ案件未選択のため ReadConfirmation を検索していない状態を示す
- `指定した文書はこの案件に見つかりません。` は、入力した `document_slug` が選択案件の文書 slug と一致しない状態を示す。文書利用状況の文書名リンクや文書 detail の URL で slug を確認してから再入力する
- `既読確認はありません` は、選択案件または選択文書に ReadConfirmation が無い状態を示す。閲覧・ダウンロードの有無は監査ログまたは文書利用状況で確認する

## 日常確認ポイント

- 案件内で、公開中の文書に未利用が偏っていないか
- 送付後 1 週間や問い合わせ前後など、期間を区切ったときに閲覧・ダウンロード・既読確認が動いているか
- 文書名や slug が分かっている場合、`文書名 / slug` 検索で目的の行だけを残し、期間・利用状況・並び順の条件と混同していないか
- `利用あり` と `最終アクセスが古い順` を組み合わせたときに、長く更新確認されていない文書がないか
- 閲覧だけ多い文書と、ダウンロードまで進む文書の差が大きすぎないか
- 既読確認が必要な運用で、確認数が極端に少ない文書がないか
- 既読確認の確認者や会社を見たい場合、文書利用状況の件数だけで判断せず `既読確認内訳` に戻る必要があるか
- 気になる文書があれば文書 detail へ戻り、公開状態や添付構成を確認する必要があるか
- 一覧が広くて読みづらいときは、表示設定で列を絞ったうえで同じ案件・同じ期間・同じ filter を見直す必要があるか
- `利用あり` なのに行内 `監査ログへ` が無い文書は、既読確認だけで利用ありになっていないかを切り分ける必要があるか
- CSV を使うときは、画面と同じ案件・同じ文書名 / slug 検索・同じ期間・同じ利用状況 filter の棚卸し用出力であり、表示列設定や新しい集計定義の反映ではないことを確認する

## 変更時の注意

- current UI は案件を 1 つずつ見る画面で、会社横断や利用者横断の比較画面ではない
- `既読確認内訳` も案件を 1 つずつ見る画面で、user / company filter や table preferences は current UI にはない
- `文書名 / slug` 検索は report rows の表示絞り込みであり、AccessLog / ReadConfirmation の算出方法や KPI 定義を変えるものとして扱わない
- `最終アクセス` はアクセスログ由来なので、既読確認だけが行われた文書では空欄のままになりうる
- 行内 `監査ログへ` は閲覧またはダウンロードがある文書だけに出る。既読確認だけの文書にも表示される前提で docs を書かない
- `文書利用一覧の表示設定` は一覧の表示列を切り替えるだけで、集計対象や利用判定そのものは変えない
- CSV は current HTML と同じ filter 済み `@report_hash[:documents]` を出す。CSV 専用の集計、案件横断 export、table preferences 連動、非同期 export job を current support として書かない
- 集計ロジックや KPI 定義を増やす docs ではなく、いまの画面の読み方だけに留める
- 監査ログ一覧側の期間 filter 実装は #1175 / PR #1285 に委ねる。未 merge の間は、文書利用状況から監査ログへ期間 params を先取りして渡さない
- company master admin や external user は使えない画面なので、社外利用者向け runbook と混ぜない

## 迷ったときの切り分け

- 案件全体で未利用文書や利用偏りを見たい: `文書利用状況` を使う
- 指定期間内の利用偏りを見たい: `文書利用状況` で `開始日` / `終了日` を指定する
- 文書名や slug が分かっている文書だけを探したい: `文書利用状況` の `文書名 / slug` 検索で表示行を絞る
- 同じ条件の棚卸しを外へ持ち出したい: 案件選択後に `CSV出力` を使う。表示設定で隠した列や列幅ではなく、現在の集計条件を確認する
- 誰がいつ見たかを追いたい: `監査ログ` に戻る。案件全体なら集計サマリの `案件の監査ログへ`、文書単位の閲覧・ダウンロードなら行内 `監査ログへ` を使う
- その文書が見える理由や公開条件を確認したい: 仕様 docs と文書 detail に戻る
- 既読確認だけの利用状況を見たい: 行内 `監査ログへ` が無いことを AccessLog 未記録のサインとして読み、`既読確認内訳` で確認日時・確認者・会社・文書 slug を確認する
- `既読確認内訳` で 0 件になる: 案件未選択か、選択案件内に ReadConfirmation が無いか、入力した `document_slug` が選択案件の文書 slug と一致していないかを分けて見る
- 古いアクセス証跡を期間で追いたい: #1175 / PR #1285 の merge 状況を確認し、未 merge なら current 監査ログの最新 200 件制限を前提にする
- 期間比較や高度な集計をしたい: current 画面の責務外として扱う

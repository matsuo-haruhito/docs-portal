# 文書マスタ運用runbook

この文書は issue `#892` に対応する、`docs-portal` の `文書マスタ` 運用メモです。

## 1. この runbook が扱う画面

この runbook は、internal admin 向けの `文書マスタ` 画面を扱います。

- `GET /admin/documents`
- 一覧上部の `新規登録`
- `検索・絞り込み`
- `文書マスタ一覧の表示設定`
- 一覧行の `編集` / `アーカイブ` / `復元` / `削除`

ここでやることは、文書の基本メタデータを管理し、公開側へ戻る前に状態を見直すことです。版差分、添付ファイル確認、案件配下の閲覧導線、ZIP 出力は別画面の責務です。

## 2. 最初の切り分け順

1. 目的の文書がどの案件にあるか分かるなら、まず `キーワード` で案件名・案件コード・文書名・URL識別子を探す
2. 候補が多いときは `カテゴリ` `種別` `公開範囲` `アーカイブ状態` を足して絞る
3. 保管期限や廃棄候補の確認が目的なら `保管期限` と `廃棄候補` の filter を先に使う
4. 一覧で `状態` `最新版/HTML` `古い版候補` `保管期限` `廃棄候補` を見て、公開側文書へ戻るか、`編集` / `アーカイブ` / `復元` / `削除` を判断する
5. 表示列が多いと感じたら `文書マスタ一覧の表示設定` で一覧の見え方を調整する

## 3. 検索・絞り込みの意味

### キーワード

`キーワード` は current code で、少なくとも次を対象に検索します。

- 案件名
- 案件コード
- 文書名
- URL識別子

文書名が曖昧でも、案件コードや URL識別子の断片が分かっていればここから入るのが最短です。

### カテゴリ / 種別 / 公開範囲

- `カテゴリ`: 文書の業務分類で見たいときに使う
- `種別`: Markdown、PDF など文書の種類で切りたいときに使う
- `公開範囲`: `internal_only` / `restricted_external` / `public_with_login` の違いを見直したいときに使う

まず `キーワード` で候補を出し、その後に 1 つずつ足すと、どの条件が効いているかを追いやすいです。

### アーカイブ状態

- `有効` を見たいとき
- `アーカイブ済み` を見たいとき
- 両方まとめて見たいとき

一覧の `状態` 列と合わせて使うと、通常導線から外した文書だけを追いたい場面で迷いにくくなります。

### 保管期限 / 廃棄候補

`保管期限` と `廃棄候補` は、どちらも次の 3 種類で絞れます。

- 値が設定されているもの
- 値が未設定のもの
- 期限日が到来済みのもの

期限系の見直しが目的なら、`アーカイブ状態` と組み合わせて「すでに退避済みで、次の処置が必要な文書」だけに絞る使い方がしやすいです。

current first slice は、ここで候補を一覧確認し、必要に応じて `編集` で期限系メタデータを直すか、行単位の `アーカイブ` / `復元` を手動で判断するところまでです。期限間近の独自閾値、通知条件の確定、自動削除、非可逆 discard はこの画面の current support には含めません。

後続 issue は次の役割で読み分けます。#1264 は画面上で次に何を見ればよいかの案内 copy、#1265 は filter と一覧列の request spec による contract 固定、#1266 は廃棄候補レビュー記録の proposal / feature slice です。いずれも retention policy の最終決定、自動通知、自動削除、非可逆 discard をこの runbook で実装済み扱いにするものではありません。

### 条件を戻したいとき

- `検索`: 現在の条件で一覧を更新する
- `条件をクリア`: filter をまとめて外して `文書マスタ` 全体へ戻る
- 初期状態など解除対象の filter がないとき、`条件をクリア` は表示されません
- 条件に一致する文書が 0 件の場合は、0 件 message の近くにも `条件をクリア` が表示され、結果を読んだ場所から全体へ戻れます
- `検索結果: N件`: 今の条件で hit している総件数を確認する
- `表示中: X-Y件 / N件`: 現在のページに表示されている範囲を確認する
- `前へ` / `次へ`: 現在の検索条件を保ったままページを移動する

検索結果が 1〜50 件のときは、現在の候補を `文書一括編集` の dry-run 候補として開けます。この件数境界はページ内の表示件数ではなく、filter 後の検索結果総件数で判断します。0 件または 51 件以上の場合は、対象が広がりすぎないように条件を見直してから進みます。

## 4. 一覧の読み方

### 案件 / 文書名 / URL識別子

- `案件`: どの案件配下の文書かを示す
- `文書名`: 公開側の文書表示へ戻るリンク
- `URL識別子`: 公開側 route で使う `slug`

公開側の見え方を確認したいときは、一覧の `文書名` から戻ります。`文書マスタ` 自体は管理運用の画面であり、公開側表示をここで置き換えるものではありません。

### カテゴリ / 種別 / 公開範囲

ここは文書メタデータの current 値をそのまま確認する列です。検索条件と列の見え方を見比べると、意図しない公開範囲や分類の混在に気づきやすくなります。

### 状態

- `有効`: 現在の通常運用対象
- `アーカイブ済み`: 通常一覧や社外導線から外している文書
- `実行者: ...`: アーカイブ済み行で、だれがアーカイブしたかを示す

アーカイブ済みの文書でも、この画面から `復元` できます。だれが退避したかを見たいときは、`状態` 列を先に確認します。

### 最新版/HTML と古い版候補

`最新版/HTML` は、公開側で主に戻る版と HTML preview の状態を読む列です。

- `最新版なし`: まだ公開版が選ばれていない状態
- `HTML表示可`: 生成済み HTML へ戻れる状態
- `HTML未生成` / `プレビュー生成中` / `プレビュー失敗` / `HTML未確認`: 公開側表示へ戻る前に preview / build 側を確認する状態

`古い版候補` は、`latest_version` 以外の DocumentVersion を read-only に数える補助列です。

- `N件`: latest 以外の版が残っている件数
- `候補なし`: latest 以外の版がない状態
- 表示される代表情報は、版 label、status、source path または `manual upload` 由来の可能性、更新日時です
- ここで候補が出ても、削除・archive・retention policy の判断を自動で意味するわけではありません

古い版候補は、「この文書には過去版や下書き・アーカイブ版が残っているか」を最初に見るための目印です。版の本文、添付、差分、download 可能性を詳しく確認する場合は、版詳細や添付確認の runbook へ戻してください。

### 保管期限 / 廃棄候補

- 日付が入っている: その文書に期限系メタデータが保存されている
- `-`: まだ設定されていない

`保管期限` は「いつまで保持するか」、`廃棄候補` は「次に廃棄判断を見直す候補日」として読むと切り分けやすいです。どちらも一覧で確認できますが、値の更新自体は `編集` から行います。

一覧での期限確認や古い版候補の表示は、削除や通知を自動実行する合図ではありません。期限到来済みの行や古い版候補がある行を見つけたら、公開側文書への戻り導線、`状態`、`最新版/HTML`、`古い版候補`、`保管期限`、`廃棄候補` を確認し、必要な場合だけ行単位で `アーカイブ` または `復元` を選びます。

### 表示設定

`文書マスタ一覧の表示設定` は、一覧列の見え方を current 作業目的に合わせるための editor です。期限確認に寄せたいときは `保管期限` や `廃棄候補` を前面に出し、版の状態確認に寄せたいときは `最新版/HTML` や `古い版候補` を残し、公開側導線確認に寄せたいときは `文書名` `URL識別子` `公開範囲` を中心に見る使い方ができます。

この表示設定と検索条件は `文書マスタ` 一覧の列表示、列幅、表示件数、候補の絞り込みにだけ効きます。文書詳細の左ペインにある文書ツリーの `表示中` badge、current document の `aria-current`、展開状態とは別の文脈なので、一覧の表示設定を変えても文書ツリー側の現在位置や展開状態を調整する操作にはなりません。

## 5. 操作の使い分け

### 新規登録

上段の `新規登録` は、文書の基本メタデータを追加する入口です。ここで扱うのは文書マスタの登録であり、版差分確認や添付の見直しは別画面です。

フォームの `案件` は、案件コード・案件名の断片で探す remote search combobox です。検索語は前後空白を除いて最大 100 文字まで、候補は最大 20 件までに bounded されます。候補 label は `案件コード / 案件名` として読み、文書を保存する `document[project_id]` の選択だけに使います。

### 編集

`編集` は文書の基本情報を見直す導線です。`タイトル` `slug` `カテゴリ` `種別` `公開範囲` `保管期限` `廃棄候補` など、一覧に出ているメタデータを更新したいときはこちらを使います。

保存済み文書の編集や validation error 後の再表示では、選択済み案件が selected option として復元されます。案件を変更する場合も新規登録と同じ remote search を使い、案件コード・案件名の短い断片で候補を探します。これは文書マスタ form の案件 lookup であり、一覧の `キーワード` filter、案件マスタの作成、Project / DocumentPermission / ProjectMembership の権限設計を変更するものではありません。

### 一括編集候補として開く

検索結果が上限内のとき、文書マスタ一覧の候補を `文書一括編集` の dry-run 入口へ引き継げます。引き継ぎ先では候補文書が選択済みで表示されるため、不要な文書のチェックを外してから dry-run を作成します。

この導線は実行を直接行いません。必ず既存の dry-run 作成と確認画面を経由し、実行は確認画面から行います。大量 ID の無制限引き継ぎ、条件保存、CSV import/export、バックグラウンド実行は含みません。

### アーカイブ

`アーカイブ` は、その文書を通常一覧や社外導線から外したいときに使います。current 一覧では、文書に保存済みの `保管期限` と `廃棄候補` を引き継いだままアーカイブする前提です。

実行前は、対象行の `文書名`、`状態`、`最新版/HTML`、`古い版候補`、`保管期限`、`廃棄候補` を見て、通常導線から外す判断で足りるかを確認します。保管期限が到来済みでも、自動削除や非可逆 discard には進みません。

### 復元

`復元` は、`アーカイブ済み` 文書を運用対象へ戻す操作です。まず `アーカイブ状態 = archived` で絞り、対象文書の `状態` 列を確認してから戻すと誤操作を減らせます。

復元後も `保管期限` と `廃棄候補` の値は期限確認の材料として残るため、必要なら `編集` で current 運用に合う値へ直します。

### 削除

`削除` は最終手段に寄せて扱います。確認文でも、関連する版・添付・コメントも削除される可能性があることが明示されています。current code では関連データ制約に引っかかると `関連データがあるため削除できません。` で止まるため、まずは `アーカイブ` で十分かを見直してから使う方が安全です。

古い版候補があることだけを削除理由にしないでください。latest 以外の版をどう保持・archive・削除するかは、この一覧の補助列だけでは決めず、版詳細、添付、公開履歴、顧客要件、retention policy を合わせて確認する論点です。

## 6. この画面でやること / やらないこと

### この画面でやること

- 文書マスタの検索と状態確認
- 基本メタデータの登録 / 編集
- フォームの案件 field で、案件コード・案件名の断片から保存先案件を選ぶ
- 公開側文書への戻り導線確認
- アーカイブ / 復元 / 削除の判断
- 最新版と HTML preview 状態の確認
- latest 以外の古い版候補の read-only 確認
- 保管期限 / 廃棄候補の確認
- 期限系 filter と一覧列を使った、手動 archive / restore 判断
- 上限内の検索結果を、既存の文書一括編集 dry-run 候補へ引き継ぐ

### この画面でやらないこと

- 案件配下での閲覧可能文書探索
- 案件マスタの作成 / 編集、案件所属や文書権限の付与
- 版詳細、差分、添付ファイル確認
- ZIP 出力
- `案件所属` や `文書権限` の管理
- 文書一括編集 dry-run を経由しない直接一括実行
- 古い版の自動削除 / 自動 archive / 一括整理
- 自動通知、自動削除、非可逆 discard
- 期限間近を判定する新しい閾値定義

## 7. current support の境界

- この runbook は current `main` の `admin/documents` 一覧と周辺操作だけを扱います
- 検索条件は `キーワード`、各 enum filter、`アーカイブ状態`、`保管期限`、`廃棄候補` までです
- 一覧は bounded 表示で、`検索結果` は filter 後総件数、`表示中` は現在のページ範囲を示します。ページ移動では現在の検索条件を維持します
- 文書マスタ form の `案件` は、Project `code` / `name` だけを検索対象にする bounded remote combobox です。候補検索と selected option 復元に閉じ、文書一覧の keyword filter、案件作成、権限付与、一覧 pagination / filter semantics は変更しません
- `文書マスタ一覧の表示設定` と検索条件は一覧側の列表示、列幅、表示件数、候補絞り込みのためのものです。文書詳細の左ペインにある文書ツリーの `表示中`、current cue、展開状態は別文脈として扱います
- 文書マスタから一括編集へ渡せるのは、検索結果が上限内の文書 ID 候補だけです。ページ内表示件数ではなく filter 後総件数で判定し、引き継ぎ後も dry-run の確認画面を経由します
- `最新版/HTML` と `古い版候補` は read-only な一覧確認のための列であり、公開版選択ルール、promote、manual upload / importer contract、版整理 policy を変更するものではありません
- 保管期限 / 廃棄候補の first slice は、一覧確認、表示列、行単位の手動 archive / restore 判断までです
- #1264 / #1265 / #1266 は、画面 copy、filter contract spec、レビュー記録 proposal を分けて扱う後続 issue です。この runbook では要件や実装詳細を重複定義しません
- 文書一括編集 dry-run を経由しない bulk archive、bulk restore、bulk delete、discard candidate marking、古い版の bulk operation、全文検索の高度化、追加 filter の提案までは扱いません
- 自動通知、自動削除、非可逆 discard、期限間近の新閾値定義、retention policy の最終決定は別 issue で扱います
- route 識別子や公開側 URL 契約の将来変更は、この文書で先回りして定義しません
- 版差分、添付、ZIP 出力、権限管理は別 runbook を正本にします

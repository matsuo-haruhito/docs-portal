# 監視・アラート設計

この文書は issue `#159` に対応する、`docs-portal` の監視・アラート設計の現行方針です。

現時点の current support は、alert を受けた後に最初に見る docs / runbook guidance と監視対象の整理です。通知 channel、alert rule、監視サービス連携は current repo ではまだ具体実装がないため、実装済みの運用機能として扱わず、必要になった時点で別 issue に分けて確認します。

## 1. 監視対象の分類

監視対象は次の 5 系統に分けます。

1. Rails アプリ
2. DB
3. storage
4. import / build / job
5. 外部依存

## 2. Rails アプリ監視

最低限見る指標:

- 5xx rate
- 4xx rate
- response time
- login failure

特に alert 対象にするもの:

- 5xx の急増
- ログイン不能
- 文書 HTML 表示または添付配信の広範囲失敗

## 3. DB 監視

最低限見る項目:

- 接続可否
- 接続数
- 遅いクエリ
- DB 容量

production で primary / queue / cache / cable を分ける場合は、primary 以外も同様に監視します。

## 4. storage 監視

対象:

- `document_files`
- `docs_sites`
- `imports`

見る項目:

- 使用量
- file not found 件数
- 権限エラー
- GCS 利用時の API 失敗

current repo で画面から確認できる storage 使用量は、管理ダッシュボードの `Storage使用量` section です。これは local `Rails.root/storage` 配下の `storage/document_files` / `storage/docs_sites` / `storage/imports` の file count と概算使用量を read-only に表示します。各領域では `大きい内訳` として直下項目の上位 5 件だけを file count、概算使用量、最終更新つきで確認できます。

`storage/document_files` の所有者別当たりは `DocumentFile 実体の Project / Document 上位` で読み、`storage/docs_sites` と `storage/imports` は direct child preview から build artifact や import staging の増加元を確認します。どちらも削除、archive、cleanup、retention policy、customer billing / quota report を決める画面ではありません。

この summary は GCS bucket 使用量、signed URL / public access policy、cleanup 候補、retention policy を判断する alert ではありません。外部監視サービスや alert rule を実装する場合は、画面で確認できる read-only 指標とは別 issue で閾値と通知先を決めます。

`DocumentFile#absolute_path` と importer の attachment/build copy は、ファイル欠落時に明示的なエラーを返すため、その失敗件数は監視対象に含めます。欠落実体の確認は `文書ファイル健全性` / `欠落ファイル詳細`、容量の概算確認は `Storage使用量` として分けて見ます。

## 5. import / build / job 監視

### import

現行 repo では `PublishJob` が import 状態を持っています。

見る項目:

- pending 件数
- failed 件数
- 最終成功 import 時刻
- `log_message` の失敗理由

### Docusaurus build

見る項目:

- build failed 件数
- 最終成功 build 時刻
- Kroki 未設定 / Kroki 接続失敗

### 将来の job 監視

job 基盤導入後は次を対象にします。

- failed jobs
- retry count
- queue latency
- stuck jobs

運用で管理画面を使うときは、次の切り分けを基準にします。

- 定期起点の失敗確認は `定期ジョブ`
- 生成依頼の入口確認は `生成ファイルイベント`
- 実際の生成処理と retry 履歴の確認は `生成ファイル実行履歴`

生成ファイル実行履歴の継続失敗候補は `GeneratedFiles::RunFailureAlertCandidates` が read-only service として返せます。同一 `job_id` / `generator` / `output_writer` / `event_source` の最新 run が閾値以上連続で failed の場合だけ候補になり、後続に success がある古い失敗 streak は候補にしません。既定は 3 回以上 / 最大 20 件ですが、呼び出し側が relation、threshold、limit、lookback_limit を狭められる内部 contract です。

管理ダッシュボードの `運用失敗入口` では、生成ファイル card に継続失敗候補を read-only に表示します。dashboard では候補を最大 5 件、取得 window を最新 200 run に bounded し、候補数、identity、連続失敗数、最終失敗時刻、短い error message 断片を調査入口として見せます。保存済み failed 件数 summary と継続失敗候補は別物として読み、候補 link は `生成ファイル実行履歴` の failed filter へ戻るための入口です。

この alert candidate は dashboard 上の read-only 調査入口であり、通知送信、alert rule、外部監視サービス連携、自動 retry を実装済みにするものではありません。どの通知 channel へ流すか、どの閾値を本番 alert rule にするか、候補から自動 retry へ進めるかは別 issue で決めます。

通知前の handoff が必要な場合は、`GeneratedFiles::RunFailureAlertHandoff` で同じ候補抽出から identity、連続失敗数、最終失敗時刻、短い error message preview、failed 実行履歴への確認 path、runbook path を read-only payload として取り出せます。この payload は dry-run / 運用引き継ぎ用であり、通知済み状態、ack / escalation、再通知抑制、外部送信の記録ではありません。本番通知先、再通知間隔、通知済み重複抑制は別 issue で人間が決めます。

管理ダッシュボードの `運用失敗入口` は、保存済み履歴から `Git同期`、`生成ファイル`、`Webhook送信`、`外部フォルダ同期` の failed / skipped / partial 件数を横断して、最初に見る運用画面へ戻すための入口です。各 card の `最新の対象履歴` は対象履歴の `updated_at` 最大値であり、発生時刻や alert 発火時刻を断定する値ではありません。7 日より古い対象履歴だけが残っている場合は `古い失敗のみ` と表示されるため、最近の障害か、古い未解消履歴が残っているだけかを分けて読みます。

`運用失敗入口` は通知 channel、alert rule、自動 retry、専用失敗一覧を実装済みとして扱うものではありません。件数や `古い失敗のみ` が気になる場合は、表示された link から Git同期履歴、生成ファイル実行履歴 / イベント履歴、Webhook送信履歴、外部フォルダ同期設定へ進み、各 runbook の確認順で原因を切り分けます。

詳細な画面の見方と retry 導線は [生成ファイル再試行と定期ジョブ管理 runbook](./生成ファイル再試行と定期ジョブ管理runbook.md) を参照します。

生成ファイル継続失敗候補の dashboard 表示だけを読み返す場合は [生成ファイル継続失敗候補runbook](./生成ファイル継続失敗候補runbook.md) を参照します。

自動リトライを検討する前の処理別リスクは [自動リトライ安全性棚卸し](./自動リトライ安全性棚卸し.md) を参照します。

## 6. 外部依存監視

対象:

- GitHub Actions / GitHub 連携
- Kroki
- Google Drive 外部フォルダ同期
- Microsoft Graph 接続 / Office preview
- 外部送付履歴（mail delivery 相当の手動確認入口）
- Webhook 設定 / Webhook 送信履歴
- 将来の mail / webhook alert rule と通知 channel

alert 対象:

- Kroki endpoint 不達
- import API 呼び出し失敗の継続
- 外部フォルダ同期の継続失敗
- Microsoft Graph 接続または Office preview の継続失敗
- mail / webhook の継続失敗

current repo では、mail delivery 相当の状況確認は `送付履歴`、webhook delivery の状況確認は `Webhook` と `Webhook送信履歴検索` から手動で始めます。これらは失敗調査の入口であり、通知 channel、alert rule、自動検知、外部監視サービス連携を実装済みとして扱うものではありません。

外部送付履歴の継続失敗候補は、`DocumentDeliveryLogs::FailureAlertCandidates` と `DocumentDeliveryLogs::FailureAlertHandoff` が read-only に返す調査入口です。同じ案件 / 送付方式 / 宛先 preview / 件名 preview の最新 failed streak を見つけ、候補 identity、連続失敗数、最終失敗時刻、短い error message preview、`/document_delivery_logs` の確認用 path を handoff payload として読みます。この payload は通知前の運用引き継ぎ用であり、通知済み状態、ack / escalation、再通知抑制、自動 retry、送付状態変更を意味しません。

候補 0 件は、current 条件で handoff する外部送付履歴候補がないことだけを意味します。mail 全体が正常、外部監視が green、Webhook delivery が正常、本番 alert が発火しない、という保証ではありません。失敗が気になる場合は、送付履歴一覧の `状態=送付失敗`、`方式`、検索、詳細画面に戻って、手動送付結果と失敗理由を確認します。

外部フォルダ同期の継続失敗候補は、管理ダッシュボードの `運用失敗入口` から読む read-only handoff です。latest run から `failed` / `partial` が連続している source だけを候補にし、候補 0 件は provider 全体正常、通知済み、ack 済み、自動 retry 済みを意味しません。原因確認や再実行判断は [外部フォルダ同期継続失敗候補runbook](./外部フォルダ同期継続失敗候補runbook.md) と [外部フォルダ同期dry-run・apply運用runbook](./外部フォルダ同期dry-run・apply運用runbook.md) を分けて確認します。

## 7. アラート重要度

### Critical

- ログイン不能
- DB 接続不能
- 本文または添付の配信が広範囲で失敗
- import / build の連続失敗で業務継続に影響

### Warning

- 単発の import failed
- Kroki 未設定または一時的不達
- storage 使用量の閾値超過
- queue 滞留
- 外部フォルダ同期や Office preview の単発失敗

### Info

- 定期 backup 成功
- build 成功
- import 成功

## 8. 通知先

現時点で具体的な通知先実装や alert rule は repo にありません。この節は、通知 channel 実装済みの一覧ではなく、通知が必要になったときに重要度ごとの運用先を分けるための方針です。

- Critical: 即時通知
- Warning: 運用チャネル集約
- Info: ダッシュボードや定期レポート

通知 channel や alert rule を実装する場合は、少なくとも次を別々に確認します。

- 通知 channel と宛先の管理方法
- alert rule の閾値と発火条件
- 監視サービスや scheduler との接続方法
- alert を受けた後に参照する runbook 更新

## 10. 現時点の監視ルール

- まずはアプリ/DB/import/build/storage の基本健全性を優先監視する
- storage 使用量は、管理ダッシュボードで見える local directory 別の read-only 概算値、各領域の直下上位 preview、将来の alert rule / 外部監視指標を分けて扱う
- 外部依存の retry は無制限にしない
- 失敗原因を追える識別子を log に残す設計を優先する
- 監視だけ増やして runbook がない状態を避ける
- 自動リトライは import / build / mail / webhook の安全性を処理ごとに棚卸ししてから、対象処理 1 つ単位の issue に分ける
- alert 実装を進めるときは、通知 channel、alert rule、監視サービス接続、runbook 更新を同じ issue で混ぜず、docs と実装の正本を確認できる単位に分ける

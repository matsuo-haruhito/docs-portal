# 案件・Git連携・文書セット初回セットアップrunbook

この runbook は、internal admin が current `main` の管理画面で 0 件から案件公開の土台を作るときの順序をまとめる。

新しい運用ルールはここでは定義しない。current route、form、既存 runbook を前提に、「最初の 1 件をどこから作るか」と「次に戻る画面」を整理する。

## この runbook が扱う画面

- `案件`: `/admin/projects`
- `Git連携`: `/admin/git_import_sources`
- `Git同期履歴`: `/admin/git_import_runs`
- `文書セット`: `/admin/document_sets`
- `外部表示プレビュー`: `/admin/projects/:code/external_preview`

## 先に知っておくこと

- 3 画面とも internal admin 向けの `/admin` 配下にある。`company_master_admin` が見られる画面境界は [company_master_admin会社・ユーザー管理runbook](./company_master_admin会社・ユーザー管理runbook.md) を正本にする。
- `文書セット` は、対象案件に文書が取り込まれてから初めて `対象文書` を選べる。最初の 1 件は `案件` と `Git連携` を先に整える方が迷いにくい。
- `外部表示プレビュー` は、案件編集画面から開く公開前確認の導線。実ログイン切り替えではなく、既存の外部ユーザー / company_master_admin 権限判定で、文書と添付ファイルの見え方を確認する。
- Git 以外の経路で文書を入れる場合は、[ローカル編集からポータル更新までの最小運用案](./ローカル編集からポータル更新までの最小運用案.md)、[ZIPインポートdry-run運用runbook](./ZIPインポートdry-run運用runbook.md)、[internal upload API dry-run・apply運用runbook](./internal%20upload%20API%20dry-run・apply運用runbook.md) を使い、文書が入ってから `文書セット` へ戻る。
- `文書セット` 画面そのものの列の見方や、`固定版` と `最新版を使う` の使い分けを見直したいときは [文書セット運用runbook](./文書セット運用runbook.md) を併せて確認する。

## まず見る順序

1. `案件` で取り込み先の Project を作る
2. `Git連携` で repository / branch / path を登録する
3. `手動同期` の結果を `Git同期履歴` で確認する
4. 文書が入ったあとに `文書セット` を作る
5. 公開前に `外部表示プレビュー` で社外ユーザー視点の見え方を確認する

## 1. `案件` で最初の Project を作る

`/admin/projects` の上段 `新規登録` が入口。current form では少なくとも次の項目を見ればよい。

- `コード`
- `案件名`
- `企業`
- `有効`
- `説明`

current form copy では `企業` は `会社名・ドメインで検索（未設定可）` になっている。会社名または domain の断片で候補を探し、会社ひも付けを後回しにしたい案件では空のまま Project 自体を作り始められる。保存済み Project の編集や validation rerender では、候補上限外の会社でも selected option として復元される。

登録後にまず確認すること:

- 一覧に `コード` と `案件名` が出ているか
- `企業` が intended な会社、または `未設定` になっているか
- `状態` が intended な `有効` / `無効` になっているか
- このあと Git 連携や文書セットで選びたい案件として見えるか

案件一覧で `削除` を押す前の confirm は、対象を取り違えないために、案件名に加えて `コード` と `企業` を読み返す cue として扱う。企業が未設定の案件では confirm でも `企業未設定` と表示される。これは削除 route、関連文書や所属設定への影響、soft delete / archive / approval workflow を変更するものではない。

案件が増えてきたら、`案件を探す` でキーワード、状態、企業を使って一覧を絞り込む。企業 filter も会社名または domain の断片で検索し、候補は最大 20 件、検索語は最大 100 文字として扱われる。URL で指定済みの企業 filter は候補上限外でも復元され、`企業未設定` を選ぶと会社ひも付けがない Project だけに絞り込む。条件を入れて 0 件になった場合は、案件が削除されたと決めつけず、結果一覧側の empty state 近くに出る `条件をクリア` から条件なし一覧へ戻る。検索フォーム内の `条件をクリア` と同じく、filter 条件を外すための導線であり、Project の保存内容、企業ひも付け、削除状態、表示設定を変更するものではない。

`案件を探す` のキーワード欄は、コード・案件名・説明の断片で探すための入口。近くに出る `最大100文字` の cue は、長い説明文や URL を丸ごと貼るのではなく、短い断片で絞り込む前提を示している。状態 filter、企業 filter、表示設定とは別の条件なので、表示列を変えても検索語や保存済み Project の内容は変わらない。

Project 登録後は、`案件編集` の `標準文書テンプレート` card も確認する。current UI では `作成予定` / `スキップ予定` の件数と、template item ごとの `状態`、`タイトル`、`slug`、`source path` が表示される。

この preview は read-only の確認であり、表示しただけでは `Document` / `DocumentVersion` は作成されない。`標準テンプレートを適用` を押すと draft 文書を作成する実行 action になるため、confirm に出る作成予定 / スキップ予定件数を読み直してから進める。

全件が `スキップ予定`、または作成予定が 0 件の場合は失敗ではなく、既に同じ slug の template 文書が存在していて追加作成がない状態として読む。標準テンプレート適用は文書の初期 draft を補う導線であり、自動公開、文書セットへの自動登録、Git 連携の自動登録や手動同期までは行わない。

ここで Project ができたら、文書の流入経路に応じて次へ進む。

- GitHub repository から pull 型で取り込みたい: `Git連携` へ進む
- 標準テンプレートを使う案件: preview を確認し、必要なら `標準テンプレートを適用` して draft 文書を作成してから `文書セット` へ進む
- すでに seed / ZIP / internal upload で文書を持ち込む前提がある: その import を済ませてから `文書セット` へ進む

## 2. `Git連携` で最小構成を登録する

`/admin/git_import_sources` の上段には、`GitHubリポジトリからpull型で取り込む` という説明 card と `新規登録` form がある。初回は form の次の項目を順に見る。

- `案件`: どの Project へ取り込むか
- `リポジトリ`: `owner/repo` 形式
- `ブランチ`
- `取込元パス`
- `認証方式`
- `状態`

current controller の初期値は次のとおり。

- `ブランチ`: `main`
- `取込元パス`: `docs`
- `認証方式`: `github_app`
- `状態`: `有効`

通常運用では current form copy のとおり `GitHub App` を優先する。`provider`、`Organization`、`installation_id`、`credential_ref`、`credential_secret` は `詳細設定（管理者・検証向け）` にまとまっているので、初回は必要になったときだけ開けばよい。

登録後に進む順序:

1. 一覧へ戻って `案件`、`リポジトリ`、`ブランチ/パス` が意図どおりか確認する
2. `手動同期` を実行する
3. redirect 先の `Git同期履歴` で最新 run の `状態`、`コミット`、`エラー` を見る
4. 必要なら `Git連携` 一覧へ戻り、`最終同期` が更新されたか確認する

`最終同期` が `未同期` のままなら、まだ初回取り込みが完了していない。`文書セット` に進む前に、まず [Git連携設定と同期失敗確認runbook](./Git連携設定と同期失敗確認runbook.md) で branch / path / 認証方式 / エラー内容を見直す。

## 3. `文書セット` は文書が入ってから作る

`/admin/document_sets` も上段 `新規登録` form が入口。current form では次の項目を見れば、最初の 1 件は作り始められる。

- `案件`
- `名称`
- `種別`
- `公開範囲`
- `表示順`
- `説明`

`案件` を選んだあと、対象案件に文書が入っていれば `対象文書` table が現れ、次を調整できる。

- どの文書をセットに含めるか
- 特定版で固定するか、`最新版を使う` か
- `並び順`
- `メモ`

current code では、案件を選んでもその案件に文書がまだ 0 件なら `対象文書` table は出ず、form 上には `まだ対象文書がありません。` と、`Git連携` / `Git同期履歴` / 他 import 経路を見直してから戻る案内が出る。初回セットアップでここに来て詰まったら、文書セット画面の入力ミスと決めつけず、先に次を見直す。

- `Git連携` の `手動同期` をまだ実行していないか
- `Git同期履歴` が `failed` / `skipped` で止まっていないか
- branch や `取込元パス` が intended な文書ディレクトリを向いているか
- Git 以外の import を使う案件なら、その import がまだ終わっていないだけではないか
- 標準テンプレートを使う案件なら、`案件編集` の preview を確認しただけで止まり、`標準テンプレートを適用` をまだ実行していないだけではないか

文書セット登録後に確認すること:

- 一覧に `案件`、`名称`、`種別`、`公開範囲`、`文書数` が出るか
- `文書数` が選んだ対象と大きくずれていないか
- 初回 0 件状態では `文書セット一覧の表示設定` より上段 form の方が主導線になることをチーム内でも共有できているか

## 4. `外部表示プレビュー` で公開前の見え方を確認する

文書セットまで作れたら、対象 Project の `案件編集` へ戻り、下部の `外部表示プレビュー` card から `プレビューを開く` を選ぶ。current UI では、社外ユーザー視点で文書と添付ファイルの表示可否を確認できる。

preview は実際のログインユーザーを切り替えるものではない。既存の権限判定を使い、指定した外部ユーザーまたは会社に属する有効ユーザーから見た結果を表示する。

使い分け:

- `ユーザー指定プレビュー`: 特定の外部ユーザー 1 名について、表示文書、非表示文書、ダウンロード可ファイルを確認する
- `会社指定プレビュー`: 会社に属する有効ユーザーをまとめて確認し、会社単位で想定外の非表示 / 表示がないかを見る

会社指定プレビューの会社選択欄には、会社を選ぶとその会社に所属する有効な外部ユーザーの表示可否をまとめて確認する、という補助 copy が表示される。current UI の会社 selector は basic select のままで、会社 remote search や RFK selector 化は current support ではない。

preview 実行後に見ること:

- `表示文書` / `非表示文書` の件数が、対象文書セットや公開範囲の意図と大きくずれていないか
- `ダウンロード可` / `ダウンロード不可` のファイル名が、対象ユーザーまたは会社の想定と合っているか
- 会社指定で `対象条件に一致する有効な外部ユーザーがいません。` と出る場合、会社やユーザー整備が先に必要ではないか

preview requested 時は `AccessLog` に `external_preview` として記録される。これは公開前確認の監査 trail として扱えるが、実際の社外ユーザーがログインして閲覧した記録ではない。利用状況や実閲覧の確認とは混同しない。

#1386 / #1388 のような selector 改善や preview 対象解決の runtime 変更が入った場合でも、この runbook では current UI の文言と挙動を確認してから追従する。

## よくある詰まりどころ

### `案件` は作れたが `文書セット` に対象文書が出ない

current code では、文書が 1 件も無い案件では `対象文書` table が出ず、`Git連携` / `Git同期履歴` / 他 import 経路を見直す案内が出る。まず取り込み完了を確認してから戻る。

標準テンプレートを使う案件では、`案件編集` の preview は read-only なので、それを見ただけでは対象文書は増えない。`標準テンプレートを適用` 後も 0 件なら、confirm の作成予定 / スキップ予定や既存 document slug を見直す。

### `Git連携` を作ったのに `最終同期` が `未同期` のまま

設定保存と文書取り込みは別。`手動同期` を実行し、その結果を `Git同期履歴` で確認する。

### `外部表示プレビュー` で対象ユーザーや会社が選べない

current UI の候補は、有効な外部ユーザーと company_master_admin、およびそれらのユーザーが属する会社が前提。preview 対象が出ない場合は、案件や文書セットだけでなく、対象会社に有効な外部ユーザーがいるかを先に確認する。

### preview 結果が想定より少ない、または 0 件になる

preview は既存の権限判定を使っているため、文書セットの作成だけでは表示可にならないことがある。`公開範囲`、案件所属、会社・ユーザー整備、添付ファイルの公開可否を分けて確認する。権限設計そのものを変える必要がある場合は、この runbook で判断せず、仕様確認に戻す。

### どの import 経路で文書を入れるかが Git 以外かもしれない

この runbook は順序整理が目的で、import 方式の優劣は決めない。Git を使わない案件では、[ローカル編集からポータル更新までの最小運用案](./ローカル編集からポータル更新までの最小運用案.md)、[ZIPインポートdry-run運用runbook](./ZIPインポートdry-run運用runbook.md)、[internal upload API dry-run・apply運用runbook](./internal%20upload%20API%20dry-run・apply運用runbook.md) のどれが current flow に合うかを先に決める。

### 会社やユーザーの整備も一緒に必要

`案件` 作成時の `企業` 選択や、その後の user 整備まで含めたいときは [company_master_admin会社・ユーザー管理runbook](./company_master_admin会社・ユーザー管理runbook.md) へ戻る。案件への参加や公開範囲まで見直すなら [案件所属・文書権限運用runbook](./案件所属・文書権限運用runbook.md) を使う。

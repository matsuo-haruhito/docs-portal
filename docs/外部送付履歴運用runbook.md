# 外部送付履歴運用runbook

この runbook は、`送付履歴` を current UI に沿って見返すための運用メモです。

新しいメール運用ルールや送付ポリシーはここでは定義しません。current 実装を前提に、dashboard から入った後に一覧と詳細で何を見分けるか、`メーラーを開く` と status 更新ボタンをどう使い分けるかを整理します。

## 先に見るもの

1. dashboard からの入口は [ダッシュボードと文書ショートカット・確認依頼の使い分け](./ダッシュボードと文書ショートカット・確認依頼の使い分け.md)
2. 権限や公開モデルの前提は [アプリケーション仕様](./アプリケーション仕様.md) と [基本モデルと権限](./specs/基本モデルと権限.md)
3. 文書や文書セット側の公開導線を見直したいときは [文書ライフサイクルと公開](./specs/文書ライフサイクルと公開.md)

## 画面の役割

`GET /document_delivery_logs` は、文書や文書セットの外部送付について、誰に何を送ろうとしているか、送付済みか、失敗として記録したかを見返す画面です。

current 実装の前提:

- internal user は全 `DocumentDeliveryLog` を一覧できます
- internal user 以外は、自分が sender の log だけを見ます
- dashboard の `社内向け導線` から入る `外部送付履歴` は internal user 向けの入口です
- 文書または文書セットから送付下書きを作成した直後も、この詳細画面へ redirect されます
- current draft 作成時の既定 `方式` は `portal_link` です

## 最初の切り分け順

1. internal user が案件横断で送付履歴を見直したいときは dashboard の `外部送付履歴` から入る
2. 直前に作成した送付下書きの宛先や本文を確認したいときは、作成後に開いた詳細画面をそのまま使う
3. 対象が `文書` か `文書セット` かを見たいときは、一覧の `文書名/文書セット名` 列か詳細冒頭の補助行を先に確認する
4. 送信済みか、まだ下書きか、失敗として残したかを見たいときは `状態` と `送信日時` を見る

## 一覧の見方

一覧では次の列を見分けます。

- `作成日時`: その log を作った時刻。一覧は新しい順です
- `案件`: どの案件にひもづく送付かを見分ける列です
- `文書名/文書セット名`: 対象が文書なら文書詳細、文書セットなら文書セット詳細へのリンクになります。どちらにもひもづかない log だけ `-` を表示します
- `宛先`: `to_addresses` をそのまま表示します
- `方式`: current enum の raw value を表示します。少なくとも `portal_link` `shared_link` `attachment` `zip_attachment` を取りえます
- `状態`: current enum の raw value を表示します。`draft` `sent` `failed` のどれかで、詳細へのリンクにもなります

current 一覧には検索、filter、bulk action はありません。0 件のときは `送付履歴はありません。` を表示するだけです。

## 詳細の見方

詳細冒頭の補助行では、まず `案件` と対象の `文書` または `文書セット` を確認します。ここで「どの案件の何を送ろうとしていた log か」を先に切り分けます。

その下の表では次を見ます。

- `宛先`: 実際の送付先
- `CC` / `BCC`: 追加受信者。未設定なら `-`
- `方式`: current enum の raw value
- `状態`: current enum の raw value
- `送信日時`: `sent_at`。未送信なら `-`

続いて、`件名` と `本文` をそのまま確認できます。ここで current draft のメール文面と送付先を見直してから、外部メーラーに渡すか、送付結果だけを記録するかを判断します。

## `メーラーを開く` と status 更新の使い分け

詳細下部には `メーラーを開く` が常に表示されます。current 実装では `宛先` `CC` `BCC` `件名` `本文` を使って `mailto:` URL を組み立てるため、手元のメーラーへ内容を引き継ぎたいときの入口として使います。

`送付済みにする` と `送付失敗として記録` は、status が `draft` のときだけ表示されます。

- `送付済みにする`: 外部送付を終えたあと、log を `sent` に更新して `送信日時` を埋めたいときに使います
- `送付失敗として記録`: 下書きのまま送れなかったことを `failed` として残したいときに使います。current 実装では `error_message` に値がなければ `manual mark` を保存します

どちらのボタンも「送付履歴の記録更新」です。再送キューや bulk retry ではありません。

## 日常確認ポイント

- internal user が dashboard から入ったときに、どの案件の送付が並んでいるか
- 対象が `文書` か `文書セット` か
- `方式` が current contract どおりの raw value で記録されているか
- `draft` のまま残っている log に対して、`メーラーを開く` で送るのか、送付結果だけ記録するのか
- `failed` が残っている場合に、その失敗を運用メモとして残すだけでよいか、対象文書や送付先の見直しが必要か

## 変更時の注意

- current UI は送付履歴の見方を扱うもので、送信承認フローやメール文面ポリシーを定義する画面ではありません
- current 一覧には検索、filter、bulk action がないため、それらを前提にした手順はここへ足しません
- `方式` と `状態` は current UI では raw enum を表示しているため、runbook でも同じ token を正本にします
- internal user と sender 以外の可視範囲は、この runbook で拡張しません

## 迷ったときの切り分け

- まず案件横断で送付履歴を見たい: dashboard の `外部送付履歴` から入る
- 作成した下書きの宛先や本文を確認したい: 送付履歴の詳細を開く
- 実際の送信は手元のメーラーで行いたい: `メーラーを開く` を使う
- 送付結果だけ履歴に残したい: `送付済みにする` または `送付失敗として記録` を使う
- 文書そのものの公開状態や閲覧権限を見直したい: 文書 / 文書セット詳細や仕様 docs に戻る

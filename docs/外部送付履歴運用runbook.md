# 外部送付履歴運用runbook

この runbook は、`送付履歴` を current UI に沿って見返すための運用メモです。

新しいメール運用ルールや送付ポリシーはここでは定義しません。current 実装を前提に、dashboard から入った後に一覧と詳細で何を見分けるか、`メーラーを開く` と status 更新ボタン、対象画面への戻り先をどう使い分けるかを整理します。

## 先に見るもの

1. dashboard からの入口は [ダッシュボードと文書ショートカット・確認依頼の使い分け](./ダッシュボードと文書ショートカット・確認依頼の使い分け.md)
2. 権限や公開モデルの前提は [アプリケーション仕様](./アプリケーション仕様.md) と [基本モデルと権限](./specs/基本モデルと権限.md)
3. 文書や文書セット側の公開導線を見直したいときは [文書ライフサイクルと公開](./specs/文書ライフサイクルと公開.md)
4. 継続失敗候補の handoff payload から来たときは [外部送付履歴継続失敗候補 runbook](./外部送付履歴継続失敗候補runbook.md) で identity、連続失敗数、確認用 path の読み方を確認する

## 画面の役割

`GET /document_delivery_logs` は、文書や文書セットの外部送付について、誰に何を送ろうとしているか、送付済みか、失敗として記録したかを見返す画面です。

current 実装の前提:

- internal user は全 `DocumentDeliveryLog` を一覧できます
- internal user 以外は、自分が sender の log だけを見ます
- dashboard の `社内向け導線` から入る `外部送付履歴` は internal user 向けの入口です
- 文書または文書セットから送付下書きを作成した直後も、この詳細画面へ redirect されます
- current draft 作成時の既定 `方式` は `ポータルリンク` です

外部送付履歴の継続失敗候補は、この一覧と詳細へ戻るための read-only 調査入口です。候補 payload は `status=failed`、送付方式、件名断片を含む確認用 path を返しますが、送付状態の変更、通知 channel への送信、alert rule の発火、自動 retry を実行するものではありません。

## 最初の切り分け順

1. internal user が案件横断で送付履歴を見直したいときは dashboard の `外部送付履歴` から入る
2. 継続失敗候補から来たときは、候補の案件、方式、受信者 preview、件名 preview、連続失敗数、最終失敗時刻を入口にし、確認用 path で failed 一覧へ進む
3. 直前に作成した送付下書きの宛先や本文を確認したいときは、作成後に開いた詳細画面をそのまま使う
4. 文書セットから送付下書きを作る前なら、文書セット詳細の `外部送付を準備` 近くに出る補助文で、現在の利用者に表示できる文書が対象であること、使用版、`利用可能な版なし` の行、文書 0 件時の empty state を先に読む
5. 対象が `文書` か `文書セット` かを見たいときは、一覧の `文書名/文書セット名` 列か詳細冒頭の補助行を先に確認する
6. 特定の案件名、案件コード、宛先/CC/BCC、件名、失敗理由で探したいときは、一覧上部の検索欄に語句を入れる。検索語は前後の空白を落とし、最大 100 文字まで使われます
7. 日次・週次の確認で作成時期や実送信時期を絞りたいときは、一覧上部の作成日 `から` / `まで` または送信日時 `から` / `まで` を使う
8. まだ送っていない下書きか、送付済みか、失敗として残したかを見たいときは、一覧上部の `状態` filter と列の `状態` を見る
9. `ポータルリンク` と添付系を分けて見たいときは、一覧上部の `方式` filter と列の `方式` を見る
10. 条件一致が 50 件を超えるときは、表示範囲と `前へ` / `次へ` を見て、同じ filter 条件のまま page を移動します
11. 現在の絞り込み条件で監査・共有用の短い一覧を残したいときは `CSV出力` を使い、最新 50 件・固定列・現在見える範囲だけの read-only export として扱います

## 文書セットから送付下書きを作る前

文書セット詳細の `外部送付を準備` は、現在の利用者に表示できる文書を対象に外部送付の下書きを準備する入口として読む。文書セットに登録されている全 item、internal-only 文書、現在の利用者に見えない文書まで自動で送付対象に広げるものではありません。

文書一覧が表示されている場合は、下書きを作る前に文書ごとの使用版を確認します。`固定版` はその行が特定版を使うことを示し、`最新版` は作成時点の current latest version を使う読み方です。`利用可能な版なし` の行がある場合は、送付操作へ進む前に対象文書の版状態を見直します。この表示は送付可否や権限 policy を新しく決めるものではなく、送付前に文書の公開範囲や版の状態を確認するための cue です。

文書セット詳細で `表示できる文書はありません。` と出る場合は、その文書セットに current user が表示できる文書がない状態として読む。外部送付を準備する前に、文書セットの構成、文書の公開範囲、版の有無、案件所属、文書権限を確認します。ここでも自動 retry、通知、承認 workflow、送付下書きの一括作成可否はこの runbook で新しく定義しません。

## 送付下書きの宛先入力

送付下書きの `宛先` `CC` `BCC` は、`;` `,` 改行で複数入力できます。保存時に前後の空白を落とし、lowercase に揃え、重複した address を 1 つにまとめます。

`宛先` は必須です。`CC` と `BCC` は空欄のまま保存できます。

明らかに `local@domain` 形式ではない token が含まれる場合、下書きは保存されず、作成画面に validation error が表示されます。domain の存在確認、MX / SMTP 到達性、アドレス帳との照合は行いません。実際の送信可否は、引き続き手元のメーラー側で確認します。

## 一覧の見方

一覧では、まず上部の検索欄で案件名、案件コード、宛先/CC/BCC、件名、失敗理由を部分一致検索できます。検索語は前後の空白を落としてから最大 100 文字まで適用されます。長い error log や件名をそのまま貼るのではなく、特徴的な断片で探すと、検索条件や詳細からの戻り先も同じ短い語句で維持しやすくなります。検索は current user が見える送付履歴の範囲内で行われ、internal user 以外が他 sender の log を検索結果で見ることはありません。本文全文検索、顧客 master 照合、アドレス帳照合は current support ではありません。

継続失敗候補から開く確認用 path も、同じ検索欄と filter を使います。候補の件名に secret-like value が含まれる場合、確認用 path の `q` は `[FILTERED]` を含む短い断片になります。検索結果が 0 件でも、mail 全体が正常、外部監視が green、候補が解消済みとは断定せず、候補 payload の identity と詳細画面の失敗理由を合わせて確認します。

作成日 `から` / `まで` は `created_from` / `created_to` として送られ、送付履歴の `created_at` を日単位で絞り込みます。開始日だけ、終了日だけでも使えます。無効な日付は絞り込みに使わず、画面上では `作成日は YYYY-MM-DD 形式で指定してください。無効な日付は絞り込みに使いません。` と案内します。

送信日時 `から` / `まで` は `sent_from` / `sent_to` として送られ、送付済み log の `sent_at` を日単位で絞り込みます。未送信の下書きや送信日時が空の log は、送信日時 filter が有効なときは一致しません。無効な日付は絞り込みに使わず、画面上では `送信日時は YYYY-MM-DD 形式で指定してください。無効な日付は絞り込みに使いません。` と案内します。

検索語を入れたまま `作成日`、`送信日時`、`状態`、`方式` を切り替えると、その検索条件を保持したまま絞り込みます。検索だけを外したいときは `検索をクリア`、作成日だけを外したいときは `作成日をクリア`、送信日時だけを外したいときは `送信日時をクリア` を使います。検索結果が 0 件のときは `検索条件に一致する送付履歴はありません。` と表示します。

次に、上部の status buttons を使って `すべて` `下書き` `送付済み` `送付失敗` を切り替えます。件数は current user が見える範囲、検索条件、作成日 filter、送信日時 filter を基準に集計され、`方式` filter が有効なときはその方式に絞った件数になります。

次に、上部の delivery type buttons を使って `すべて` `ポータルリンク` `共有リンク` `添付ファイル` `ZIP添付` を切り替えます。件数は current user が見える範囲、検索条件、作成日 filter、送信日時 filter を基準に集計され、`状態` filter が有効なときはその状態に絞った件数になります。

`検索`、`作成日`、`送信日時`、`状態`、`方式` は併用できます。例えば、特定案件の未送付 portal link 下書きだけを確認したいときは検索語、作成日、`下書き`、`ポータルリンク` を同時に使い、添付系の送付済みだけを日次 review したいときは送信日時、`送付済み`、`添付ファイル` または `ZIP添付` を組み合わせます。

いずれかの filter が有効なときは、filter 群の下に `現在の絞り込み条件` summary が表示されます。summary は検索語、作成日範囲、送信日時範囲、localized status label、localized delivery type label を 1 行で読み返すための補助表示です。個別 clear link や状態・方式の `すべて` button を置き換えるものではなく、今どの条件で何件のうちどの page を見ているかを確認する入口として使います。

summary の下にある表示件数の説明は、現在の filter 条件に一致した履歴のうち、いま画面に出ている範囲を示す補助文です。1 page 目では先頭から表示している件数、2 page 目以降では現在の page に表示している件数として読みます。検索結果が 0 件の場合も summary は残るため、`すべての送付履歴を見る` を押す前に、どの検索語・日付・状態・方式の組み合わせで 0 件になったかを読み返します。

送付履歴があるときは `表示範囲: N件中M件を表示しています。` または `表示範囲: N件中M件を表示しています。現在のページ: A-B件。1ページあたり最大50件です。` の表示を先に読みます。N は現在の検索・作成日・送信日時・状態・方式 filter に一致した総件数、M は現在の page に表示している件数です。50 件を超える場合は `前へ` / `次へ` と `X / Yページ` が表示され、同じ filter 条件を保ったまま 50 件単位で移動できます。`page=bad` や `page=0` は 1 page 目、総 page 数を超える `page` は最後の有効 page に寄せて表示されます。表示設定は `document_delivery_logs` の列の見え方を保存するだけで、`検索`、`作成日`、`送信日時`、`状態`、`方式` filter、page 移動、詳細への `return_to` は変えません。

`CSV出力` は、現在の `検索`、`作成日`、`送信日時`、`状態`、`方式` filter を引き継ぎ、current user が見える送付履歴のうち最新 50 件だけを固定列で出力します。列は `作成日時`、`送信日時`、`案件コード`、`案件名`、`対象種別`、`対象名`、`To`、`CC`、`BCC`、`方式`、`状態`、`失敗理由` です。table preferences の表示列設定や現在の page は CSV 列や出力対象に影響しません。失敗理由は短い preview に閉じ、Authorization / Bearer / Basic / token / secret / password / API key などの secret-like な断片は `[FILTERED]` として出力されます。CSV は read-only export であり、bulk retry、送付状態変更、メール本文全文 export、添付 export、background export、saved report ではありません。

表示設定で調整できる列は次の通りです。

- `作成日時`: その log を作った時刻です。一覧は新しい順です。pin された列として扱います
- `案件`: どの案件にひもづく送付かを見分ける列です
- `文書名/文書セット名`: 対象が文書なら文書詳細、文書セットなら文書セット詳細への link になります。どちらにもひもづかない log だけ `-` を表示します
- `受信者`: `recipients` column key のまま、To / CC / BCC を短い label 付きで分けて表示します。長い address の全量確認は詳細を開きます
- `方式`: helper label で表示します。current `main` では `ポータルリンク` `共有リンク` `添付ファイル` `ZIP添付` を取りえます
- `状態`: helper label で表示します。current `main` では `下書き` `送付済み` `送付失敗` のどれかで、詳細への link にもなります。pin された列として扱います
- `失敗理由`: `送付失敗` のときだけ短い失敗理由を表示します。失敗理由が空なら `詳細で失敗内容を確認してください。` と表示し、それ以外の状態では `-` です

表示設定を使うときの目安:

- 下書き処理では `作成日時`、`案件`、`文書名/文書セット名`、`受信者`、`状態` を残すと、誰に何を送る予定かを確認しやすい
- 失敗調査では `作成日時`、`案件`、`受信者`、`方式`、`状態`、`失敗理由` を残すと、対象と失敗内容を一覧で切り分けやすい
- `状態` の link から詳細へ入る導線は列表示の設定後も同じです。失敗理由が長い、または本文・CC / BCC まで見直したい場合は詳細を開きます

0 件のときは、検索語がある場合は `検索条件に一致する送付履歴はありません。`、検索語がなく `作成日`、`送信日時`、`状態`、`方式` のいずれかで絞っている場合は `条件に一致する送付履歴はありません。` と表示します。検索条件や filter がある場合は `現在の絞り込み条件` summary と併読し、どの条件を外すべきかを確認してから `すべての送付履歴を見る` を使います。検索条件や filter がない場合は `送付履歴はありません。` を表示します。bulk action はありません。

## 詳細の見方

詳細冒頭の補助行では、まず `案件` と対象の `文書` または `文書セット` を確認します。ここで「どの案件の何を送ろうとしていた log か」を先に切り分けます。

その下の表では次を見ます。

- `宛先`: 実際の送付先
- `CC` / `BCC`: 追加受信者。未設定なら `-`
- `方式`: helper label の送付方式
- `状態`: helper label の送付状態
- `送信日時`: `sent_at`。未送信なら `-`

続いて、`件名` と `本文` をそのまま確認できます。ここで current draft のメール文面と送付先を見直してから、外部メーラーへ渡すか、送付結果だけを記録するかを判断します。

継続失敗候補から detail を開いた場合は、候補 payload の `failure_count` と detail の失敗理由を別々に読みます。`failure_count` は最新 log から連続している failed 数であり、detail に表示される 1 件の失敗理由や保存済み failed 件数全体ではありません。原因確認では、同じ案件・方式・宛先 preview・件名 preview の最新 failed log が連続しているかを一覧と詳細で見ます。

## `メーラーを開く` と status 更新の使い分け

詳細下部では、下書きの履歴なら `メール作成`、`手動状態更新`、`対象へ戻る` の操作 group を確認します。`送付済み` または `送付失敗` になった履歴では `手動状態更新` group は表示されず、`メール作成` group 内の補助文で、下書きではないため状態を手動で変更する操作が出ないことを確認します。

`メール作成` の `メーラーを開く` は常に表示されます。current 実装では `宛先` `CC` `BCC` `件名` `本文` を使って `mailto:` URL を組み立てるため、手元のメーラーへ内容を引き継ぎたいときの入口として使います。この操作だけでは送付履歴の状態は変わりません。

`手動状態更新` は status が `下書き` のときだけ表示されます。`送付済み` または `送付失敗` になった履歴では、detail に状態更新 action は出ず、controller も non-draft の手動更新 request を受け付けません。画面上も、送付済み・送付失敗済みの記録は過去記録として確認する旨の補助文を表示します。

- `送付済みにする`: 外部送付を終えたあと、log を `送付済み` に更新して `送信日時` を埋めたいときに使います
- `送付失敗として記録`: 下書きのまま送れなかったことを `送付失敗` として残したいときに使います。`失敗理由` 欄には、宛先確認待ち、添付容量超過、外部メーラー側のエラーなど、後で一覧の `失敗理由` 列や検索で見返せる短い理由を 200 文字以内で入力します。空欄のまま記録した場合は current fallback として `manual mark` が保存されます

どちらのボタンも「送付履歴の記録更新」です。再送キューや bulk retry ではありません。継続失敗候補を見た後でも、ここでできるのは current UI 上の下書きからの手動状態更新だけです。候補 payload を見たことによって、過去の `送付失敗` 履歴を再更新したり、自動 retry や通知済み状態へ進めたりはしません。

## 戻り先の見分け方

詳細下部では、送付結果の確認後に戻る先を 2 種類から選べます。

- `対象の文書へ戻る` / `対象の文書セットへ戻る` / `対象の案件へ戻る`: 今見ている log の対象そのものへ戻りたいときに使います
- `送付履歴一覧へ戻る`: 送付履歴の一覧へ戻って、他の log も続けて見直したいときに使います

文書または文書セットから送付下書きを作って詳細へ来た場合でも、一覧へ戻る link は `GET /document_delivery_logs` に戻ります。作成元の対象へ戻したいときは、対象リンク側を使います。

一覧の filter から詳細へ入ったときは、詳細 link の `return_to` に現在の一覧 path が渡されます。検索語は正規化後の最大 100 文字、作成日 filter と送信日時 filter、状態、方式、page は一覧 path 側にも含まれるため、`送付履歴一覧へ戻る` で同じ絞り込みと同じ page へ戻れます。filter なしで詳細へ入った場合の戻り先は通常の送付履歴一覧です。

一覧の filter から入った下書きを `送付済みにする` または `送付失敗として記録` した場合も、詳細画面の `return_to` は引き継がれます。状態更新後に表示される詳細画面から `送付履歴一覧へ戻る` を押すと、元の検索語・作成日・送信日時・状態・方式 filter と page を含む一覧へ戻れます。不正な `return_to` は通常の送付履歴一覧へ fallback します。

継続失敗候補の `failed_delivery_logs_path` から一覧を開いた場合も、戻り先は通常の一覧導線として扱います。候補 payload の確認用 path は調査入口であり、ack / escalation / 再通知抑制の状態遷移ではありません。

## 日常確認ポイント

- internal user が dashboard から入ったときに、どの案件の送付が並んでいるか
- 継続失敗候補から来た場合、候補の identity、方式、受信者 preview、件名 preview、連続失敗数、最終失敗時刻を一覧・詳細で確認できているか
- 文書セットから送付下書きを作る前に、現在の利用者に表示できる文書、使用版、`利用可能な版なし`、文書 0 件時の empty state を読み、公開範囲や版状態を見直す必要がないか
- 案件名、案件コード、宛先/CC/BCC、件名、失敗理由で探したい送付履歴を絞り込めているか。長い文字列は前後の空白を落とした最大 100 文字までで探す
- 作成日 `から` / `まで` で、日次・週次に作成された下書きや失敗履歴だけへ絞れているか
- 送信日時 `から` / `まで` で、実際に送付済みになった期間だけへ絞れているか。未送信の下書きは送信日時 filter では一致しない
- status buttons で `下書き` だけ、`送付済み` だけ、`送付失敗` だけを切り分けたい状況か
- delivery type buttons で `ポータルリンク` と `共有リンク` / `添付ファイル` / `ZIP添付` を切り分けたい状況か
- `検索`、`作成日`、`送信日時`、`状態`、`方式` の併用 filter で、今日見るべき下書きや失敗だけに絞れているか
- `CSV出力` が現在の絞り込み条件と current user の可視範囲を引き継ぎ、最新 50 件・固定列・masked failure reason preview に閉じていることを理解しているか
- `現在の絞り込み条件` summary で、検索語、日付範囲、状態、方式の組み合わせを読み返せているか。0 件時も summary を見て、解除する条件を決めてから `すべての送付履歴を見る` を使う
- 条件一致が 50 件を超える場合、`表示範囲` と `X / Yページ` を確認し、同じ filter 条件のまま `前へ` / `次へ` で確認範囲を移動できているか
- 範囲外 page や古い link から開いたときに、1 page 目または最後の有効 page に寄せられていることを、表示範囲で読み取れているか
- 表示設定で、下書き処理や失敗調査に必要な列を残せているか
- `受信者` 列で To / CC / BCC の違いを見分けられているか。全量や本文まで確認する必要があれば詳細を開く
- `失敗理由` 列で一覧上の一次切り分けができるか、詳細まで開く必要があるか
- 対象が `文書` か `文書セット` か
- `方式` が current contract どおりの helper label で表示されているか
- `下書き` のまま残っている log に対して、`メーラーを開く` で送るのか、送付結果だけ記録するのか
- `送付失敗として記録` を使う場合、後で検索できる短い `失敗理由` を残せているか。空欄時は `manual mark` だけが残るため、原因調査が必要なら一度立ち止まって理由を入力します
- `送付済み` / `送付失敗` 済みの log では手動状態更新 action が出ず、画面上の補助文も過去記録として確認する案内になるため、過去記録の修正ではなく下書きからの送付結果記録として扱えているか
- filter 付き一覧から detail に入り、状態更新後も `送付履歴一覧へ戻る` で元の作業文脈へ戻れるか
- `送付失敗` が残っている場合に、その失敗を運用メモとして残すだけでよいか、対象文書や送付先の見直しが必要か
- 候補 0 件や検索結果 0 件を、mail 全体正常、外部監視 green、通知済み、ack 済みと誤読していないか

## 変更時の注意

- current UI は送付履歴の見方を扱うもので、送信承認フローやメール文面ポリシーを定義する画面ではありません
- 文書セット詳細の `外部送付を準備` 周辺の補助文は、現在の利用者に表示できる文書、使用版、文書 0 件時の読み方を示す cue です。送付 workflow、権限 policy、document set viewability、公開判定、DB schema を変更するものとして読まないでください
- current 一覧には `検索`、`作成日` filter、`送信日時` filter、`状態` filter、`方式` filter、50 件単位の page 移動、CSV出力、表示設定があります。bulk action はないため、それを前提にした手順はここへ足しません
- `現在の絞り込み条件` summary は active filter を読み返す補助表示です。個別 clear link、状態・方式 buttons、50 件単位の page 移動、詳細 link、`return_to`、表示設定、CSV出力の意味は変えません
- CSV出力は current filter 条件に一致する最新 50 件を固定列で出す read-only export です。table preferences の列表示、現在の page、本文全文 export、添付 export、background export、saved report、送付状態更新、bulk retry とは別物として扱います
- 表示設定は列の見え方だけを変えます。検索・作成日 filter・送信日時 filter・page 移動・詳細 link・状態更新 workflow・CSV列の意味は変えません
- 継続失敗候補は read-only handoff です。候補 payload、一覧検索、手動状態更新、通知 channel、alert rule は別物として扱い、自動 retry、ack / escalation、再通知抑制を current support としてここへ足しません
- `検索` は `projects.name`、`projects.code`、`to_addresses`、`cc_addresses`、`bcc_addresses`、`subject`、`error_message` の first slice です。検索語は前後の空白を落として最大 100 文字に正規化されます。本文全文検索、顧客 master 照合は current support ではありません。CSV出力は同じ検索語を引き継ぎますが、最新 50 件・固定列・masked failure reason preview に閉じます
- `作成日` は `created_at` の日単位 filter です。送付履歴 log が作られた時期を見る入口であり、送信完了時刻ではありません
- `送信日時` は `sent_at` の日単位 filter です。送付済み時期を見る入口であり、`sent_at` が空の下書きは一致しません
- `方式` と `状態` は current UI では helper label を表示しているため、runbook でも localized label を正本にします
- 手動状態更新は draft 起点の記録更新です。`送付済み` / `送付失敗` 済み履歴の再更新、状態 machine 全体、過去記録の編集 workflow、承認 workflow はここへ足しません
- 失敗理由入力は短い運用メモです。taxonomy、template 管理、再送 queue、bulk retry、自動 retry、外部メーラーの送信結果同期は current support として書きません
- 宛先 validation は明らかな形式不正を下書き作成前に止めるためのものです。MX / SMTP 到達性、アドレス帳、顧客 master 照合の説明はここへ足しません
- internal user と sender 以外の可視範囲は、この runbook で拡張しません

## 迷ったときの切り分け

- まず案件横断で送付履歴を見たい: dashboard の `外部送付履歴` から入る
- 継続失敗候補から調査を始めたい: `DocumentDeliveryLogs::FailureAlertHandoff` の payload で identity、案件、方式、連続失敗数、最終失敗時刻、error preview を読み、確認用 path から failed 一覧へ進む
- 文書セットから外部送付下書きを作りたい: 文書セット詳細の `外部送付を準備` 周辺で、現在の利用者に表示できる文書、使用版、`利用可能な版なし`、文書 0 件時の案内を確認してから進む
- 作成した下書きの宛先や本文を確認したい: 送付履歴の詳細を開く
- 特定案件、宛先/CC/BCC、件名、失敗理由から送付履歴を探したい: 一覧の検索欄を使う。長い error log や件名は、特徴的な 100 文字以内の断片に絞る
- 日次・週次で作成された下書きや失敗履歴を見たい: 一覧の作成日 `から` / `まで` を使う
- 実際に送付済みになった期間を見たい: 一覧の送信日時 `から` / `まで` を使う
- portal link / 共有リンク / 添付ファイル / ZIP添付を分けて確認したい: 一覧の `方式` filter を使う
- 複数条件で 0 件になった: `現在の絞り込み条件` summary で検索語・日付・状態・方式を読み返し、外す条件を決めてから `すべての送付履歴を見る` や各 clear link を使う
- 条件一致が 50 件を超えている: `表示範囲`、`X / Yページ`、`前へ` / `次へ` を見て、同じ filter 条件のまま前後の page へ移る
- 古い URL や手入力の `page` で想定外の範囲が出た: 表示範囲を見て、1 page 目または最後の有効 page に丸められていないか確認する
- 現在の filter 条件で短い evidence を残したい: `CSV出力` を使い、最新 50 件・固定列・table preferences 非連動・masked failure reason preview の read-only export として読む
- 下書きや失敗調査で読む列を絞りたい: 一覧の `送付履歴一覧の表示設定` を使う
- 実際の送信は手元のメーラーで行いたい: `メーラーを開く` を使う
- 送付結果だけ履歴に残したい: `送付済みにする` または `送付失敗として記録` を使う。失敗時は、後で検索・一覧確認しやすい短い失敗理由を入れる
- 状態更新後も同じ filtered list で続けて処理したい: 一覧 filter から詳細へ入り、更新後の `送付履歴一覧へ戻る` を使う
- 対象の文書や文書セットへ戻って公開状態や内容を見直したい: 対象に戻る link を使う
- 他の送付 log も続けて確認したい: `送付履歴一覧へ戻る` を使う
- retry、通知先、alert rule、ack / escalation が必要になった: この runbook では決めず、監視 docs と継続失敗候補 runbook を参照して別 Issue で仕様判断する

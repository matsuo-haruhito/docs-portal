# 開発・保守ガイド

この文書は issue `#147` に対応する、`docs-portal` の開発体験と保守性に関する整理です。

## 1. 最初に読む順番

AI や開発者が新規タスクに入る時は、次の順で十分です。

1. `AGENTS.md`
2. [docs/README.md](./README.md)
3. [アプリケーション仕様](./アプリケーション仕様.md)
4. [基本モデルと権限](./specs/基本モデルと権限.md)
5. [テスト方針](./テスト方針.md)
6. タスクに対応する runbook / 仕様書

この順で、権限・公開モデル・運用制約・テスト前提を先に押さえます。

## 2. ディレクトリ責務

主要ディレクトリの役割は次です。

- `app/controllers/`: request 入口、認証、対象 record 解決、response 選択
- `app/controllers/admin/`: 管理画面の CRUD と運用向け導線
- `app/controllers/api/internal/`: token 前提の内部 API
- `app/services/`: import / build / export / preview / 検索補助など、controller から分離した業務ロジック
- `app/previews/`: dry-run や変更前確認の正規化、差分組み立て
- `app/checks/`: 診断、品質確認、health check 系の判定ロジック
- `app/importers/`: import 処理本体
- `app/models/concerns/`: 権限制御や public_id などの横断 concern
- `db/seeds/support/`: seed 用の build / 走査 / 変換補助
- `spec/requests/`: 導線・権限・HTTP レベルの回帰確認
- `spec/services/`, `spec/importers/`: ルールや変換ロジックの回帰確認

## 3. controller を太らせない方針

controller は次に限定します。

- 認証・認可
- path / param から対象を引く
- service や query object を呼ぶ
- HTML / JSON / redirect を返す

次のような処理は service へ寄せます。

- import 実行と publish job 記録
- Docusaurus build や ZIP 生成
- 検索条件の適用
- dry-run / preview 結果の組み立て
- AI 向け export 形式の整形
- 外部フォルダ同期の計画作成・適用・外部API呼び出し

既存の具体例:

- import: `DocumentImporter`, `ImportDryRunValidator`
- export: `AiContextExporter`, `AiContextHashExporter`
- search / 補助: `DocumentSearch`, `RelatedDocumentFinder`, `RecentDocumentsQuery`
- 権限 preview: `ExternalVisibilityPreview`, `PermissionChangeDryRun`
- 外部フォルダ同期: `ExternalFolderSync::Runner`, `ExternalFolderSync::GoogleDriveClient`

新機能を追加する時も、controller に分岐や集計が増え始めたら service 抽出を優先します。

## 4. service class の責務分割

大きく次の系統に分けます。

### 名前空間の判断基準

- `app/services/` は controller や job から呼ぶ単一ユースケースの orchestration を置く
- 読み取り専用の検索・集計は `app/queries/` 候補として扱う
- 成果物生成は `app/builders/` 候補として扱う
- dry-run や表示 preview は `app/previews/` 候補として扱う
- quality check や設定診断は `app/checks/` 候補として扱う
- JSON / Markdown / CSV など外部出力形式は `app/exports/` 候補として扱う
- 外部 CLI / API / SaaS との境界は `app/integrations/` 候補として扱う
- 画面整形に寄った DTO や view 補助は `app/presenters/` 候補として扱う
- 既存実装を一括移動するのではなく、新規実装や関連箇所の改修タイミングで段階的に寄せる

巨大な万能 `Service` を作らず、名前だけで責務が分かる配置を優先します。

### `app/` 直下を増やす時の基準

- まず「責務の種類」で切れるかを見る
- その責務が横断的で、複数クラスへ増える見込みがある時だけ `app/<category>/` を追加する
- 単発の整理では、新しい top-level を増やさず既存カテゴリの下へ入れる
- `app/services/<use_case>/` は最後の選択肢で、既存カテゴリへ置く方が責務の種類を追いやすい
- service の中身を分解して責務が固定化したら、wrapper 以外は `services` から出す
  - 例: `AiContextMarkdownExporter` を `app/exports/`
  - 例: `RecentDocumentsQuery` を `app/queries/`
  - 例: `PermissionChangePreviewHash` を `app/presenters/`

### import / build

- `app/importers/document_importer.rb`
- `db/seeds/support/docusaurus_builder.rb`
- `db/seeds/support/docusaurus_build_runner.rb`

import と seed build は近い責務を持ちますが、現時点では seed 側に Docusaurus 補助実装が寄っています。今後は「workspace 準備」「diagram 変換」「build 実行」「成果物配置」をより分けていく前提です。

### search / navigation

- `DocumentSearch`
- `RelatedDocumentFinder`
- `SourcePathBreadcrumb`
- `RecentDocumentsQuery`
- `DocumentSearch` から切り出した `DocumentQuerying::SearchMatchCatalog`

検索条件の正規化や関連文書導線は service 側で閉じ、controller は filter の受け渡しだけに寄せます。

### export / report

- `AiContextExporter`
- `AiContextHashExporter`
- `DocumentUsageReport`
- `DocumentVersionZipBuilder`
- `DocumentVersionsZipBuilder`
- `AiContextExporter` から切り出した `AiContext::ProjectSectionBuilder`, `DocumentSectionBuilder`
- `DocumentVersionZipBuilder`, `DocumentVersionsZipBuilder` から切り出した `DocumentExport::FileSelector`, `PathResolver`, `ReadmeBuilder`

出力形式ごとに formatter を分け、HTML 画面や API から共通利用できる形を保ちます。

### preview / dry-run

- `ExternalVisibilityPreview`
- `PermissionChangeDryRun`
- `ImportDryRunValidator`
- `DocumentBulkEditPlan` から切り出した `DocumentBulkEdit::ChangeNormalizer`, `DocumentBulkEdit::ItemBuilder`
- `PermissionChangeDryRun` から切り出した `PermissionChange::VisibilitySimulator`
- `ExternalFolderSync::Runner` は外部フォルダ同期の dry-run / apply を同じ計画形式で扱う

「変更前確認」は適用処理と混ぜず、preview 専用 service と presenter を分けます。

### checks / diagnostics

- `ApplicationConfigurationDiagnostic` から切り出した `ApplicationConfiguration::EnvironmentChecks`, `SecretChecks`, `StorageChecks`, `WorkspaceChecks`
- `DocumentVersionQualityChecker` から切り出した `DocumentVersionQuality::MetadataChecks`, `DocumentFileChecks`, `MarkdownReferenceChecks`

一覧 service に判定ルールを抱え込ませず、診断項目ごとの check 群へ分けます。

### import helpers

- `ZipImportStager` から切り出した `ZipImport::ArchiveExtractor`, `ManifestBuilder`
- `ZipImportDocumentScanner` から切り出した `ZipImport::PathClassifier`, `DocumentCandidateBuilder`
- `GitImportManifestBuilder` から切り出した `GitImport::PathCatalog`, `DocumentPayloadBuilder`

import 本体は orchestration に留め、feature ごとの helper を `app/importers/<feature>/` へ寄せます。

### integrations / 境界

- `KROKI_ENDPOINT` を使う図変換
- Docusaurus / Node / npm 実行
- Google Drive 外部フォルダ同期
- 将来の GitHub / Microsoft Graph / Box / webhook 送信
- `WebhookDeliveryDispatcher` から切り出した `WebhookDispatch::PayloadBuilder`, `RequestBuilder`, `DeliveryRecorder`

外部コマンドや外部サービス呼び出しは、Rails の業務ロジック本体と分離しやすい場所に寄せる前提で設計します。

Google Drive外部フォルダ同期は、現時点では追加gemを使わず `ExternalFolderSync::GoogleDriveClient` が標準ライブラリの `net/http` / `openssl` / `json` でService Account JWT認証、OAuth user refresh token認証、Drive API列挙、media download、exportを扱います。

### presenters

- `AccessRequestHash` から切り出した `AccessRequestPresentation::HashBuilder`

Hash 化や画面/API 向けの整形は presenter 寄りの責務として分けます。

### builders

- `NotificationEventPublisher` から切り出した `NotificationPublishing::RecipientResolver`, `ReceiptBuilder`

複数 service から再利用しうる組み立て処理や付随データ生成は builder へ寄せます。

## 5. 依存 gem の採用理由

主要な gem / 依存の意図は次です。

- `rails`, `propshaft`, `importmap-rails`, `turbo-rails`, `stimulus-rails`: Rails 標準寄りの最小構成を維持する
- `pg`, `puma`, `solid_cache`, `solid_queue`, `solid_cable`: Rails 8 系の標準運用基盤
- `slim`: view の記述量を抑える
- `vite_rails`: Vite entrypoint から npm package と gem 提供の Stimulus controller を読み込む
- `pundit`: policy 配置の受け皿。現時点では model / controller 側の明示チェックが主で、Pundit 利用は将来拡張余地として残している
- `rparam`: 文書一覧 filter の型付きパラメータ正規化に使う
- `rtypes`: typed な value object / DTO 導入の余地として採用しているが、現時点での利用は限定的
- `tree_view`: 案件 / 文書のツリー表示と、その Rails helper / render state を担う
- `rails_table_preferences`: 一覧の表示設定保存や関連 controller helper を担う基盤として採用している
- `rails_fields_kit`: `rfk_*` helper と Tom Select 連携を使った検索可能な select / 入力補助を担う

追加 gem を入れる時は、標準 Rails で代替できない理由と、運用時の更新責務を説明できるものに限定します。

Google Drive外部フォルダ同期は、Google API gemを追加せず標準ライブラリで実装しています。Service Account方式とOAuth user方式の両方を `net/http`, `openssl`, `json` で扱い、将来OAuthフローやchanges APIの厳密運用が複雑化した場合だけ、公式gem採用を再検討します。

## 6. 内部 UI gem を更新するときの最短確認順

`tree_view` / `rails_table_preferences` / `rails_fields_kit` を追う task では、詳細を読む前に次の順で十分です。

1. `Gemfile.lock`
   - current `main` が実際に解決している revision を確認します。`Gemfile` の取り込み元より、いまの app の再現根拠として優先します。
2. [関連 gem 連携調査 runbook](./関連gem連携調査runbook.md)
   - 3 gem の責務、upstream docs の入口、app-side verification checklist の正本として使います。
3. 対応する gem repo の README / docs / 関連 issue・PR
   - public API や導入前提の変更なのか、説明不足なのかを切り分けます。
4. `docs-portal` 側の seam
   - helper、partial、initializer、Vite alias、Stimulus entrypoint、対象画面 spec を確認し、app 固有の組み込み差分かどうかを見ます。

補足:

- `#477` の ref / tag 固定方針が未確定でも、この確認順は current `main` の snapshot を追う手順として使えます。
- 入口文書では手順だけを要約し、gem ごとの詳細な確認先や app-side verification 項目は runbook 側を正本にします。
- 3 gem とも、upstream docs を読むだけでは足りず、対象画面や request / system spec を含む app-side verification まで行ってから issue の分類を決めます。

## 7. docs-quality の相対 Markdown link 失敗を読む順番

README / `docs/**/*.md` を触る PR では、`docs-quality` の `Check relative markdown links` が `.github/scripts/check_markdown_links.rb` を実行します。失敗したら、log の `README.md:35 -> ./docs/... (missing: docs/...)` の形から、左から順に「リンクを書いたファイル」「行番号」「Markdown の link destination」「実際に存在確認した path」を読みます。

確認順は次で十分です。

1. 指摘された `file:line` の link text と destination を見る
2. `missing:` の path が repo 内に存在するか、ファイル名の typo かを確認する
3. 日本語ファイル名は、既存 docs と同じ raw Unicode 表記を基本にし、すでに percent-encoded で安定している link は無理に直さない
4. `#anchor` だけの link、`https:` / `mailto:` など外部 scheme は checker の対象外なので、relative path の誤りに絞る
5. README や docs index の入口を直したら、同じ destination を別の index で重複して間違えていないか確認する

この runbook は link checker の仕様変更ではなく、docs-only PR の手戻りを減らすための読み方です。CI failure 個別のリンク修正は、失敗した PR 側で直します。

## 8. AI が無駄に広く読まないための指針

- まず route と対象 controller から入る
- 次に対象 service と model concern を読む
- 最後に対応する request spec / service spec を読む
- `docs/ai/prompt-logs/` は履歴確認が必要な時だけ読む
- 生成物や sample data は、不具合再現や import/build 調査が必要な時だけ触る
- tree / table preference / searchable form を触る時は `doc/frontend_interaction_policy.md` と関連 gem の README (`tree_view-rails`, `rails_table_preferences`, `rails_fields_kit`) を先に確認する

## 9. README と docs の役割分担

- `README.md`: リポジトリの入口、利用者向け概要、開発者向け入口
- `docs/README.md`: repo 正本の索引
- 各 `docs/*.md`: 仕様、運用、設計、保守ルールの詳細

README に詳細仕様を増やしすぎず、詳細は `docs/` に寄せる方針です。

## 10. 今後の整理余地

- import と seed build 補助の責務をさらに分離する
- Pundit を本格利用するか、現行の明示チェック方針を維持するかを決める
- `rtypes` の実利用箇所を増やすか、導入意図を再評価する
- lint / static check の導入は、CI コストと保守効果を見ながら段階導入する

# 文書セット運用runbook

この runbook は、internal admin が current `main` の `文書セット` 画面を日常運用で見返すときの入口をまとめる。

新しい配布ルールや公開ポリシーはここでは定義しない。current form、一覧、filter、request spec を前提に、「どの検索・filter で絞るか」「どの列を見分けるか」「固定版と最新版をどう使い分けるか」「対象文書が多い案件でどう探すか」「文書がまだない案件でどこへ戻るか」を整理する。

## 先に見るもの

1. `文書セット` を 0 件から立ち上げる順序を確認したいときは [案件・Git連携・文書セット初回セットアップrunbook](./案件・Git連携・文書セット初回セットアップrunbook.md)
2. 案件の参加者や文書の公開範囲まで見直したいときは [案件所属・文書権限運用runbook](./案件所属・文書権限運用runbook.md)
3. 文書の流入経路を確認したいときは [Git連携設定と同期失敗確認runbook](./Git連携設定と同期失敗確認runbook.md)、[ZIPインポートdry-run運用runbook](./ZIPインポートdry-run運用runbook.md)、[internal upload API dry-run・apply運用runbook](./internal%20upload%20API%20dry-run・apply運用runbook.md)

## 画面の役割

`/admin/document_sets` は internal admin 向けの管理画面で、上段 `新規登録` / `編集` form と、下段の `文書セット一覧` を 1 画面で扱う。

current 実装の前提:

- `案件` は案件コードまたは案件名で remote search して 1 つ選び、その案件に属する文書だけを `対象文書` として選ぶ
- `案件` の検索語は前後空白を除いた最大 100 文字まで扱われ、候補は最大 20 件である。編集時や validation error 時は保存済み・入力中の選択済み案件が復元される
- `対象文書` が多い場合は、form 内の `文書名 / URL識別子で探す` で対象案件内の文書を remote picker から選び、既存 table row へ移動・絞り込みできる
- form 内の `表示中の対象文書を絞り込み` は、文書名または `URL識別子` を使って、すでに表示されている対象文書 table row だけをローカルに絞り込む
- `選択済みのみ表示` は、保存 payload を変えず、現在チェックしている文書だけを見失わないための表示補助である
- `固定版` は文書ごとに `version_label` で remote search できる。検索対象は対象案件内の同一 Document の versions に閉じ、候補は最大 20 件である
- `種別` と `公開範囲` は一覧・form の両方で user-facing label で表示される
- `文書セット一覧` では `検索` / `種別` / `公開範囲` の filter metadata が current controller behavior につながっており、文書セット名・案件名・案件コード・label の組み合わせで一覧を絞り込める
- 一覧 `検索` 欄直下の補足 copy は、検索対象が案件名・案件コード・文書セット名の断片であることと最大 100 文字であることを、入力中も読める visible cue として示す
- 一覧は filter 後の文書セット集合を page 表示する。`検索結果` は filter 後の総件数、`表示中` はいま表示している範囲であり、複数 page があるときは `前へ` / `次へ` で同じ filter 条件のまま移動する
- 初期状態では一覧 filter form 内の `条件をクリア` は表示されず、`検索` / `種別` / `公開範囲` のいずれかが有効なときだけ form actions に表示される
- filtered 0 件時は、form 内の `条件をクリア` に加えて 0 件メッセージ近くにも `条件をクリア` が出る
- `CSV出力` は現在の絞り込み条件に一致する文書セット集合を、表示中 page や表示設定の列表示・幅とは独立した固定列で出力する
- `GET /admin/document_sets.json` は CSV 本体ではなく、現在の `検索` / `種別` / `公開範囲` 条件、件数、固定 CSV header、summary を確認する companion metadata JSON を返す。表示中 page や表示設定とは独立し、文書セット行データや `public_id` 一覧は返さない
- `文書セット一覧の表示設定` で変わるのは一覧 table の列表示だけで、filter 結果、表示 page、登録済みデータ、`対象文書` の中身は変わらない
- `新規登録` form は current filter 条件付きの一覧から validation error で戻ったとき、`q` / `set_type` / `visibility_policy` の query を保ったまま再描画される
- `編集` / `削除` は list row ごとにあり、edit route も destroy route も numeric id ではなく current `public_id` を使う

## 一覧で見るポイント

一覧には少なくとも次の列が並ぶ。

- `案件`: どの Project に属する文書セットか
- `名称`: 文書セット名
- `種別`: `送付用` `要件` `設計` `運用` `顧客共有` `社内用` `その他`
- `公開範囲`: `社内のみ` `限定公開` `ログインユーザー公開`
- `文書数`: 現在このセットに含まれている文書数。数値の下に、固定版の使い方を読むための `固定版あり（N件）` / `最新版のみ` / `文書なし` cue が出る

日常確認では、まず `検索` で案件名・案件コード・文書セット名の断片から対象候補を絞り、次に `種別` / `公開範囲` が intended な label か、`文書数` が想定より増減していないかを見る。

`文書数` の cue は編集前の判断補助である。`固定版あり（N件）` は、その文書セットに含まれる対象文書のうち、特定の `固定版` を選んでいる item が N 件あることを示す。`最新版のみ` は対象文書が 1 件以上あり、固定版を選んだ item がない状態、`文書なし` は対象文書が 0 件の状態である。

この cue で変わるのは一覧での読み取りだけで、次は変わらない。

- `文書数` の数値は、これまで通りそのセットに含まれる `DocumentSetItem` の件数である
- 固定版として保存されるのは、編集 form の `固定版` select で特定の版を選んだ文書だけである
- `最新版のみ` は「今後も最新版へ追従する item だけで構成されている」という読み取りであり、公開範囲や配布可否を変更するものではない
- `文書なし` はセット内 item が 0 件であることを示す cue で、案件に文書が存在しないことまでは断定しない

### `検索` / `種別` / `公開範囲` filter の見方

一覧の filter は、表示設定より先に「どの文書セット群を見たいか」を狭めるために使う。

- `検索` は文書セット名、案件名、案件コードの部分一致で絞る。文書タイトルや対象文書の `URL識別子` は一覧検索の対象ではない
- `検索` 欄直下の補足 copy は、placeholder が入力中に見えなくなった後も、検索対象と最大 100 文字の上限を読み返すための cue である。検索ロジック、CSV 出力、page 移動、表示設定の contract を広げるものではない
- `種別` だけで絞ると、`送付用` や `設計` など同じ用途の文書セットだけを並べて比較できる
- `公開範囲` だけで絞ると、`社内のみ` や `限定公開` など同じ公開先のセットだけを確認できる
- `検索` / `種別` / `公開範囲` を併用すると、たとえば案件コードの断片を含み、かつ `送付用` / `限定公開` のような current 運用上の組み合わせだけを残せる

filter が有効なときは、一覧上部の `適用中:` に `検索: ...`、`種別: ...`、`公開範囲: ...` が badge として並ぶ。ここは現在どの条件で一覧件数が減っているかを読む場所で、表示設定の列幅や表示列とは別に確認する。

`検索結果: N件` は filter 後の総件数で、今の page に表示されている行数ではない。1 page に収まらない場合は `表示中: X-Y件 / N件` と `A / Bページ` を見て、いま総件数のどこを見ているかを確認する。`前へ` / `次へ` の page link は `q` / `set_type` / `visibility_policy` を保つため、同じ条件のまま古い文書セットや次の候補へ移動できる。`per_page` が query にある場合も page link に引き継がれる。

初期状態では filter form 内の `条件をクリア` は出ない。active filter がある場合だけ、form actions 内の `条件をクリア` から `検索` / `種別` / `公開範囲` をすべて外して一覧トップへ戻れる。CSV 出力はその時点の filter 条件を使うため、出力前に `適用中:` badge、`検索結果`、`表示中` の範囲を確認する。画面上の `CSV出力` 近くにも、CSV は現在の絞り込み条件に一致する文書セット集合を、表示設定とは独立した固定列で出すことが案内される。

CSV の固定列は current controller の `CSV_HEADERS` に従い、`案件コード`、`案件名`、`文書セット名`、`種別`、`公開範囲`、`文書数`、`public_id` を出す。表示設定で table の列を隠したり幅を変えたりしても、CSV header、CSV row、対象集合は変わらない。page 2 などを見ている状態で CSV 出力しても、出力対象はその page の行だけではなく、current filter に一致する全件である。

同じ条件を JSON で見返したいときは、`GET /admin/document_sets.json?q=...&set_type=...&visibility_policy=...` を CSV の companion metadata として読む。metadata では `report_type: document_sets`、`export_scope: current_filters`、`filters`、`ignored_filters`、`row_count`、`csv_headers`、`summary` を確認できるが、CSV row や文書セットの `public_id` 一覧は返らない。詳しい項目の読み方は [CSV条件metadata JSON 運用メモ](./CSV条件metadata_JSON運用メモ.md) の文書セット section を参照する。

`検索結果: 0件` かつ `条件に一致する文書セットはありません。` が出たときは、登録済み文書セットがないとは限らない。まず `検索` の入力ゆれ、次に `種別` / `公開範囲` の組み合わせを見直す。条件を外して全体を見直す場合は、form actions の `条件をクリア` か、0 件メッセージ近くの `条件をクリア` で一覧トップへ戻る。0 件のときは page link は出ない。

新規登録で validation error が起きたときは、一覧も form action も current filter 条件を保ったまま戻る。`per_page` は一覧の表示件数を再現するために使われるが、新規登録 form action へ page / per_page を混ぜない。保存成功後は一覧トップへ戻るので、保存後に同じ slice を見直したいときは必要に応じて filter をかけ直す。

`文書セット一覧の表示設定` は、一覧で見たい列だけを残したいときに使う。ここで変わるのは table の見え方だけで、次は変わらない。

- filter で絞られた対象集合
- page 移動で引き継がれる filter 条件
- CSV 出力の固定列と対象集合
- 文書セット CSV metadata JSON の `row_count`、`csv_headers`、`summary`、対象 filter
- 登録済みの `対象文書`
- `種別` や `公開範囲` の保存値
- `編集` / `削除` の導線

## 新規登録 / 編集 form の見方

上段 form では、まず次の基本項目を埋める。

- `案件`: 案件コードまたは案件名で検索し、文書セットを所属させる Project を選ぶ
- `名称`: 同じ案件の中で見分ける名前
- `種別`: 利用目的の大まかな区分
- `公開範囲`: セットをどう見せるかの current policy
- `表示順`: 一覧や公開側での並び順の基準
- `説明`: 運用メモや補足説明

`案件` の候補は最大 20 件で、検索対象は Project の `code` と `name` である。編集画面や validation error 後に戻った form では、保存済みまたは入力中の選択済み案件だけが初期表示されるため、別案件へ付け替えたいときは案件コードまたは案件名の短い断片で検索する。

`案件` をまだ選んでいないときは、`対象文書` は出ず、`案件を選ぶと対象文書を設定できます。` だけが表示される。

## `対象文書` table の見方

対象案件に文書が 1 件以上入っていると、`対象文書` table が現れる。ここでは文書ごとに次を調整する。

- `対象`: チェックした文書だけをこのセットへ含める
- `文書名`: 対象文書への戻り口。下段には `URL識別子` も出る
- `固定版`: `最新版を使う` か、特定の版を固定するか
- `並び順`: セット内での表示順
- `メモ`: その文書だけに付ける補足

`文書名 / URL識別子で探す` は、対象案件に属する文書を title / slug で探して、選んだ文書の table row へたどり着くための remote picker である。候補は current project scope に閉じ、選んだ値は既存 row を探す補助として使われる。

`表示中の対象文書を絞り込み` は、同じ案件に属する文書が多いときに table の表示行を探しやすくするローカル補助である。入力値は文書名と `URL識別子` に対して部分一致し、右側の件数表示で現在の表示件数を確認できる。

`固定版` select は、各文書 row の中で、その文書に属する version を `version_label` で探す remote search である。検索語は前後空白を除いた最大 100 文字まで扱われ、候補は最大 20 件である。候補は対象案件内の同一 Document の versions に閉じるため、別案件の文書や同じ案件の別 Document の version をここから固定することはできない。

初期表示では `最新版を使う` と、保存済みまたは入力中の固定版だけが出る。目的の固定版が見えないときは、固定する版の `version_label` の短い断片で検索する。

`選択済みのみ表示` は、チェック済み文書、固定版、並び順、メモを確認し直すための表示補助である。検索中も選択済み文書を見失わないために使い、保存対象を自動で増減させるものではない。

これらの検索・表示補助で変わるのは表示中の table row と fixed version 候補の読み出しだけで、次は変わらない。

- 対象案件に属する文書だけが候補になる範囲
- `固定版` select で選べるのは、その row の Document に属する versions だけであること
- `document_set_items[...]` の送信 shape
- チェック済み文書、固定版、並び順、メモの入力内容
- 案件横断の対象文書検索、別 table への置き換え、文書権限や company / user selection の remote search 横展開は current support ではない

絞り込み後に目的の文書が出ない場合は、まず `URL識別子` の入力ゆれ、次に選択中の `案件` が正しいかを見る。別案件の文書や未取り込み文書を、これらの検索欄だけで候補に出すことはできない。

### `固定版` と `最新版を使う` の使い分け

- `最新版を使う`
  - `DocumentSetItem` に版 id を保存せず、その文書の current `latest_version` を使う
  - 継続的に最新版へ追従したいときの既定値として読む
- 特定の版を選ぶ
  - `固定版` select で `version_label` の断片を検索し、`版ラベル (公開状態)` の候補から選ぶ
  - 検索語は最大 100 文字、候補は最大 20 件で、対象案件内の同一 Document の versions だけが候補になる
  - 保存済みまたは入力中の固定版だけが初期表示されるため、別の版へ固定したいときは `version_label` で検索する
  - その版を固定して見せたいときに使う

保存時は、チェックした文書だけがその文書セットに残る。チェックを外した文書は、この save でセット外になる前提で読む。

保存後の一覧で `固定版あり（N件）` が出た場合は、N 件分の item だけが特定版固定で、残りの item は `最新版を使う` のままと読む。全 item が固定版という意味ではないため、必要なら編集 form でどの文書に固定版を選んでいるかを確認する。

## 文書がまだない案件で止まったとき

`案件` を選んでもその案件に文書がまだ 0 件なら、`対象文書` table は出ず、次の案内が表示される。

- `まだ対象文書がありません。`
- `Git連携`
- `Git同期履歴`
- `ZIPインポート`

この状態は、文書セット画面のエラーというより「対象案件にまだ文書が入っていない」ことを示す current empty state と読む。

- GitHub repository から取り込む案件: [Git連携設定と同期失敗確認runbook](./Git連携設定と同期失敗確認runbook.md)
- ZIP で持ち込む案件: [ZIPインポートdry-run運用runbook](./ZIPインポートdry-run運用runbook.md)

## 保存後に見るポイント

保存して一覧へ戻ったら、少なくとも次を確認する。

- `案件` と `名称` が intended な組み合わせで出ているか
- 必要なら `検索` で案件名・案件コード・文書セット名から保存したセットを見つけられるか
- `種別` と `公開範囲` が current label で迷わず読めるか
- 必要なら `検索` / `種別` / `公開範囲` filter で同系統のセットだけに絞って比較できるか
- filter をかけたときは `適用中:` badge、`検索結果`、`表示中` を読み、複数 page があるなら `前へ` / `次へ` で同じ条件のまま移動できるか
- filter を外すときは active filter 時だけ出る `条件をクリア` で一覧トップへ戻れるか
- `文書数` が選んだチェック数と大きくずれていないか
- `文書数` の cue が、固定版を含むなら `固定版あり（N件）`、固定版なしで対象文書があるなら `最新版のみ`、対象文書 0 件なら `文書なし` になっているか
- fixed version を選んだ文書だけを固定し、他は `最新版を使う` のままにできているか

`文書数` が想定と違うときは、保存前にチェックしていない文書があったか、別案件の文書を探していないかを先に見直す。

`固定版あり（N件）` の件数が想定と違うときは、編集 form の `固定版` select で版を選んだ文書だけを数えているかを見直す。`最新版のみ` が出ているセットは、対象文書はあるが固定版 item がない状態なので、固定版が必要な配布セットでは編集 form へ戻って対象文書ごとに選び直す。

## 日常確認ポイント

- `検索` で案件名・案件コード・文書セット名のどれを手掛かりにしているかを `適用中:` の badge で確認できているか
- `送付用` や `顧客共有` など、用途に対して `種別` がずれていないか
- `公開範囲` が current 運用の意図とずれていないか
- 同じ `検索` / `種別` / `公開範囲` の組み合わせだけに絞ると、想定外のセットが混ざっていないか
- `検索結果` は filter 後の総件数、`表示中` は現在 page の範囲として読み分け、複数 page があるときは `前へ` / `次へ` で同じ filter 条件のまま確認できているか
- CSV 出力前に `適用中:`、`検索結果`、`表示中`、画面上の CSV cue を確認し、表示中 page や表示設定ではなく current filter 集合と固定列が出ると読み分けられているか
- `GET /admin/document_sets.json` を確認するときは、CSV row ではなく current filter の `row_count`、`csv_headers`、`summary` を見る companion metadata として読めているか
- 初期状態で `条件をクリア` が出ていないこと、active filter 時だけ form actions に出ること、filtered 0 件時は 0 件メッセージ近くからも一覧トップへ戻れることを確認できているか
- `案件` を付け替えるとき、案件コードまたは案件名で intended な Project を選び、対象文書の候補がその案件内に閉じていることを確認しているか
- `文書名 / URL識別子で探す` や `表示中の対象文書を絞り込み` で対象文書を探すとき、文書名または `URL識別子` が対象案件内で一致しているか
- fixed version にした文書だけを意図して固定できているか
- `文書数` が急に 0 件や想定外の件数になっていないか
- `固定版あり（N件）` / `最新版のみ` / `文書なし` cue が、固定版 select の intended state とずれていないか
- 文書が見つからず empty state になっている案件で、取り込み run の見直しが必要ではないか

## 変更時の注意

- current 画面で選べるのは、その `案件` に属する文書だけである
- `案件` は案件コード / 案件名で remote search できるが、これは文書セットを所属させる Project を選ぶ入口であり、対象文書や文書権限の検索範囲を広げるものではない
- 一覧の `検索` は文書セット名・案件名・案件コードの絞り込みであり、対象文書 title / slug の検索ではない
- 一覧の `検索` 欄直下の補足 copy は、placeholder と同じ検索対象と最大 100 文字の上限を入力中も見える形にするだけで、一覧検索の対象、CSV 対象集合、page 移動、表示設定の責務を変更しない
- 一覧の page 移動は filter 後の文書セット集合を分割して見るための導線で、`検索` / `種別` / `公開範囲` を保つ。保存 payload、CSV 対象集合、対象文書 form の候補範囲は page 移動で変わらない
- form 内の `文書名 / URL識別子で探す` は対象案件内の文書を remote picker で探す補助、`表示中の対象文書を絞り込み` は表示済み row をローカルに絞る補助として読み分ける
- `固定版` select は文書ごとの version remote search であり、対象案件内の同一 Document の versions だけを `version_label` で探す。別案件や別 Document の version は固定候補に出ない
- `選択済みのみ表示` は表示補助であり、未選択文書を保存 payload から勝手に消す contract ではない
- 一覧の `固定版あり（N件）` / `最新版のみ` / `文書なし` cue は、登録済み item の固定版利用状況を読む補助であり、保存値、公開判定、固定版選択ルールを変更しない
- `条件をクリア` は active filter の解除導線である。初期状態に解除対象がないときは form 内に表示されず、filtered 0 件時は form 内と 0 件メッセージ近くのどちらからも同じ一覧トップへ戻る
- `CSV出力` は現在の `検索` / `種別` / `公開範囲` に一致する文書セット集合を固定列で出す。表示中 page、`文書セット一覧の表示設定` の列表示・幅、画面上の表示列、table preference は CSV header や対象集合を変えない
- `GET /admin/document_sets.json` は CSV 条件確認用 metadata であり、CSV 本体、行データ API、全件 export API、scheduled report ではない。unsupported enum は `ignored_filters` に分け、表示設定で table の列を変えても metadata 条件や `csv_headers` は変わらない
- `文書セット一覧の表示設定` は一覧の列表示を切り替えるだけで、filter 結果、page 移動、登録内容そのものは変えない
- empty state は「Git だけが唯一の正解」という意味ではなく、current code が `Git連携` / `Git同期履歴` / `ZIPインポート` と他 import 経路を見直す導線を出している状態として読む
- validation error 時の filter 文脈保持は `新規登録` の再描画を見やすくするための current behavior であり、保存成功後まで filter を固定する契約ではない
- 文書配布ポリシーの再設計や新しい `種別` の定義までは、この runbook に含めない

# ダッシュボードと文書ショートカット・確認依頼の使い分け

この文書は、`dashboard` から辿る日常導線の役割差をまとめる。

新しい運用ルールはここでは定義しない。current 実装を前提に、どの画面を何のために使うか、一般利用者と internal user で見える導線がどこで分かれるかを整理する。`README.md` は入口案内に留め、この文書を `GET /dashboard`、`GET /documents`、`GET /document_bookmarks`、`GET /document_approval_requests` の current UI を読むための正本として扱う。`文書ショートカット` 画面そのものの細かい見方と `解除` / 戻り方は [文書ショートカット運用runbook](./文書ショートカット運用runbook.md) を参照してください。

## 先に見るもの

1. 画面全体の正本は [閲覧画面とUI](./specs/閲覧画面とUI.md)
2. 権限や利用者種別の前提は [基本モデルと権限](./specs/基本モデルと権限.md)
3. dashboard の `保留中の申請` から入る利用者向け一覧は [利用者向けアクセス申請runbook](./利用者向けアクセス申請runbook.md)
4. 文書の外部送付や同意・申請まわりを運用目線で見直すときは [アクセス申請・同意管理・Webhook運用runbook](./アクセス申請・同意管理・Webhook運用runbook.md)

## 導線の全体像

### 1. ダッシュボード

`GET /dashboard` は、次に触る案件や文書へ戻るための入口。

- summary card では `閲覧可能案件` `閲覧可能文書` `保存ショートカット` `保留中の申請` をまとめて確認する
- `保留中の申請` summary card は current user 自身の pending `AccessRequest` 件数を見る入口で、一覧へ入った後の `対象` `要求権限` `状態` `理由` `承認者` と `取消` の見方は [利用者向けアクセス申請runbook](./利用者向けアクセス申請runbook.md) を正本にする
- `閲覧可能文書` summary card からは `GET /documents` の案件横断一覧へ直接入り、案件名が曖昧でもキーワードやタグから文書を探せる
- internal user には summary card が 1 枚追加され、`保留中の確認依頼` から pending の `DocumentApprovalRequest` 一覧へ直接入れる
- 一覧では `案件` `お気に入り` `後で読む` `最近見た文書` `最近更新された文書` を並べて、次の移動先を決めやすくする
- `お気に入り` / `後で読む` に候補がある場合は、その section の `ショートカット一覧でさらに見る` から `GET /document_bookmarks` へ移動して保存済みショートカットをまとめて見直す
- `最近見た文書` / `最近更新された文書` に候補がある場合は、その section の `文書一覧でさらに探す` から `GET /documents` へ移動して案件横断で追加候補を探す
- 画面上部の CTA から `案件一覧へ` と `文書ショートカット一覧` へ移動できる
- internal user は summary card で未処理の確認依頼を見つけ、画面下部の `受付中Q&A候補` と `社内向け導線` から Q&A workspace、`確認依頼一覧`、`外部送付履歴`、`管理ダッシュボード` へ進める

使いどころ:

- どの案件や文書から作業を再開するか決めたいとき
- 案件をまだ 1 つに絞らず、案件横断で文書を探したいとき
- お気に入りや最近見た文書から、すぐ本文へ戻りたいとき
- dashboard の短い候補だけでは足りず、保存済みショートカット一覧や文書一覧へ移って追加候補を見たいとき
- 自分が出した access request の pending 件数や処理結果を見返したいとき
- internal user が pending task、受付中の公開 Q&A、社内向け運用画面へ入りたいとき

短い list の表示上限と scope:

- `案件` は current user が閲覧可能な `Project` を code 順で最大 10 件だけ表示する。0 件でも `案件一覧へ` で一覧側を確認できる
- `お気に入り` と `後で読む` は current user の readable な `DocumentBookmark` だけを新しい順にそれぞれ最大 8 件表示する。権限外になった文書は残っていても出さない
- `お気に入り` / `後で読む` の list がある場合の `ショートカット一覧でさらに見る` は、dashboard の最大 8 件枠から `文書ショートカット` 画面へ移るための導線。0 件時の `文書一覧へ` CTA とは役割が違い、既存 bookmark をまとめて整理したいときに使う
- `最近見た文書` と `最近更新された文書` は current user が今閲覧可能な文書だけをそれぞれ最大 10 件表示する
- `最近見た文書` / `最近更新された文書` の list がある場合の `文書一覧でさらに探す` は、recent history や更新候補の検索ではなく、案件横断の `文書一覧` に戻ってキーワード・タグ・実用フィルタで追加候補を探す導線として読む
- `保留中のアクセス申請` は current user 自身の pending request を新しい順に最大 3 件表示する。他ユーザーの申請や処理済み申請は short list には出さない
- internal user だけに `保留中の確認依頼` summary card、`受付中Q&A候補`、`社内向け導線` が出る。external user の dashboard には確認依頼一覧、受付中 Q&A 候補、管理ダッシュボードへの入口を出さない
- `受付中Q&A候補` は internal user が閲覧できる文書に紐づく public Q&A root のうち、`status: open`、`internal_only: false`、`parent_id: nil` の thread を更新順に最大 5 件だけ表示する。回答済み / クローズ済み Q&A、internal-only 確認事項、返信単体、権限外または archived 文書の metadata は候補に含めない

`最近見た文書` と `最近更新された文書` の読み分け:

- `最近見た文書` は current user 自身の `AccessLog` の `view` action から作る再訪導線。`RecentDocumentsQuery` は文書ごとに最新の閲覧ログを選び、現在も `viewable_by?` を満たす文書だけを最大 10 件返す
- 画面上では `あなたの閲覧履歴から表示しています` と補足され、各行の meta には `あなたが最近閲覧` と出る。更新通知ではなく、作業再開用の個人履歴として読む
- `最近更新された文書` は current user が閲覧可能な `Document` を `updated_at` の新しい順に最大 10 件並べる更新確認導線。自分が開いたかどうかではなく、閲覧可能文書の更新順を見る
- 画面上では `閲覧可能な文書を更新日時の新しい順に表示しています` と補足され、各行の meta には `更新日時` が出る。未読通知ではなく、更新候補を確認する入口として読む
- どちらも current user が今見えない文書は表示対象にしない。recent history に残っていても権限外になった文書は再表示しない
- dashboard の recent list は再開先を素早く選ぶための短い入口。より多くの個人用近道を見直すときは `文書ショートカット一覧` へ移動する
- 未読管理、通知、recommendation、閲覧履歴検索は current support として扱わない。#1596 / #1598 のような runtime 変更が merge された場合だけ、この節を追従する

internal user 向けの `確認依頼` と `受付中Q&A候補` の役割差:

- summary card の `保留中の確認依頼`: まず件数を見て、未処理の依頼があるかを dashboard で即確認したいときの入口
- summary card の `未処理の確認依頼を見る`: pending 件数を確認した流れで、そのまま確認依頼一覧へ進むための入口
- `受付中Q&A候補`: 公開 Q&A のうち受付中の root thread を、通知や担当割当ではなく read-only handoff として見直す入口。文書、案件、版、投稿者、返信有無、最終活動、本文 preview を読み、link から版または文書の Q&A workspace に戻る
- `社内向け導線` の `確認依頼一覧`: `外部送付履歴` や `管理ダッシュボード` と並べて、社内オペレーション全体の一部として辿る入口

補足:

- `保留中の確認依頼` summary card、`受付中Q&A候補`、`社内向け導線` の確認依頼一覧が見えるのは internal user だけ
- external user は `閲覧可能案件` `閲覧可能文書` `保存ショートカット` `保留中の申請` の 4 カード構成のままで、確認依頼一覧や受付中 Q&A 候補を日常導線として前提にしない
- `受付中Q&A候補` の 0 件表示は、通知 green、SLA 達成、問い合わせが存在しない保証ではない。必要に応じて各文書の Q&A workspace、検索、タブを確認する
- dashboard から案件横断で探す入口は `閲覧可能文書` summary card で、案件を 1 つ決めてから配下文書やツリーを見たいときは `閲覧可能案件` から各案件詳細へ進む

初回利用・空状態の読み方:

- dashboard 冒頭の補足文が「権限不足のエラーではありません」と出ている場合は、external user の個人向け section がすべて空の状態として読む。まず `案件一覧へ` / `文書一覧` で現在閲覧できる範囲を確認し、社内の確認依頼や受付中 Q&A 候補は自分向けの通常導線ではないと切り分ける
- internal user に「保留タスクがない場合も」と出ている場合は、pending の確認依頼や承認待ちがない状態。管理画面だけへ寄せず、`案件` や `文書` の一覧から現在の担当範囲を確認できる
- `案件` が 0 件なら、current user が参照できる `Project` がまだない状態。まず `案件一覧へ` で一覧側の表示を確認し、必要なら `保留中の申請` から自分の access request を見直す
- `お気に入り` と `後で読む` が 0 件なら、まだ個人用ショートカットを保存していないだけで、閲覧権限の有無とは切り分けて読む。文書を探すときは `閲覧可能文書` または `案件一覧へ` から始める
- `最近見た文書` が 0 件なら、このアカウントで閲覧履歴がまだない状態。権限不足やエラーではなく、文書詳細を開くと履歴ベースで次回以降の戻り先になる
- `最近更新された文書` が 0 件なら、current user が閲覧できる範囲に更新候補がない状態。権限エラーではなく、案件や文書の存在確認は `閲覧可能案件` / `閲覧可能文書` の summary card と各一覧で切り分ける
- `受付中Q&A候補` が 0 件なら、dashboard の最大 5 件枠に出る open public Q&A root がない状態。問い合わせ全体がない、通知が正常、回答期限を満たしている、という意味ではない
- `保留中の申請` は自分が出した pending `AccessRequest`、`保留中の確認依頼` は internal user が処理する pending `DocumentApprovalRequest`、`受付中Q&A候補` は internal user が見直す open public Q&A root。いずれも dashboard 上の短い入口だが、対象者と次に見る一覧 / workspace が違う

#### `閲覧可能文書` 一覧の見方

`GET /documents` は、current user が閲覧可能な文書を案件横断で探すための discovery 画面です。`閲覧可能案件` から案件を決めて入る導線とは役割が違い、まず文書を見つけたいときに使います。

README では「dashboard から案件横断で文書を探せる」という入口だけを案内し、この節で `GET /documents` の current UI と、案件を決めた後に project-scoped の `文書一覧` へ切り替える判断基準をまとめます。

最初の切り分け順:

1. 案件名や文書名に心当たりがあるなら `キーワード` から入る
2. 候補が多いときは `タグ` `カテゴリ` `ファイル種` `公開範囲` を 1 つずつ足す
3. HTML の有無、添付、PDF、図面系だけを見たいときは `実用フィルタ` を足す
4. `ヒット理由` と `案件` 列を見て、案件横断のまま追うか、対象案件へ絞って見直すかを決める

この画面で current code が見ている主な項目:

- `キーワード`: 案件名、文書名、URL識別子、元パス、版、本文、文書キーワード、添付ファイル名 / パス
- `タグ`: `DocumentTag` ベースの分類
- `カテゴリ`: `Document.categories`
- `ファイル種`: `Document.document_kinds`
- `公開範囲`: `Document.visibility_policies`
- `実用フィルタ`: `HTML生成済み` `添付あり` `PDFあり` `図あり`

一覧の読み方:

- `案件`: どの案件配下の文書かを見分ける列。案件を決め打ちして見直すなら、ここから案件詳細へ移る
- `文書名`: 文書詳細への入口
- `ヒット理由`: 検索語が案件名、文書名、元パス、添付ファイル名などのどこに当たったかを badge 付きで示す
- `最新版` `HTML` `添付`: そのまま HTML 確認に向くか、添付込みの配布物があるかを手早く判断する
- `最終更新`: 同名文書候補が複数あるときの見分けに使う

権限境界の見方:

- external user では、所属案件と portal 表示条件の両方を満たす文書だけが出る
- internal user では `internal_only` 文書も案件横断一覧に出る
- external user で 0 件のときは、画面上でも権限確認を依頼する案内が出る

向いている場面:

- 案件名が曖昧で、まず文書を起点に探したいとき
- 似たタグやカテゴリをまたいで、どの案件に文書があるか見たいとき
- HTML がある文書、PDF を持つ文書、図面系文書だけを案件横断で探したいとき

向いていない場面:

- すでに案件が決まっていて、左の `文書ツリー` を使って階層で絞りたいとき
- current page 選択や ZIP 出力オプションを使って案件配下の文書をまとめて持ち出したいとき
- internal user が source path を前提に manual upload の追加候補を選びたいとき

その場合は、案件詳細配下の `GET /projects/:project_code/documents` へ移ってから確認します。project-scoped の `文書一覧` では、左の `文書ツリー`、ZIP 出力オプション、manual upload panel、source path を使った upload / zip 文脈が加わるため、詳細は [文書一覧の検索・実用フィルタ・ZIP出力 runbook](./文書一覧の検索・実用フィルタ・ZIP出力runbook.md) を正本にしてください。

### 2. 文書ショートカット

`GET /document_bookmarks` は、利用者ごとの近道を最低限の一覧で見直す画面。

- `お気に入り` は継続的に参照する文書を残すための置き場
- `後で読む` は今すぐは開かないが、あとで確認したい文書を一時的に溜める置き場
- `最近見た文書` は操作で追加する一覧ではなく、閲覧履歴ベースで直近の文書を再表示する
- `お気に入り` と `後で読む` は、この画面から `解除` できる
- 細かい見方や `解除` / 戻り方は [文書ショートカット運用runbook](./文書ショートカット運用runbook.md) を正本にする

使いどころ:

- 利用者自身が、よく読む文書と一時保留の文書を分けたいとき
- dashboard では足りない件数を、まとめて見直したいとき
- 案件一覧へ戻る前に、直近の個人用ショートカットを整理したいとき

補足:

- 権限外になった文書は、bookmark や recent history に残っていても一覧には出さない
- 文書詳細の `お気に入りに追加` `後で読むに追加` から、この一覧へ積み上がる

### 3. 確認依頼

`DocumentApprovalRequest` は、文書コメントより一段正式に、担当者を指定して確認を依頼したいときの導線。

- 文書詳細の `正式な確認依頼` セクションから `タイトル` `確認相手` `内容` を入れて依頼を作成できる
- 文書単位の一覧には `タイトル` `依頼者` `確認相手` `状態` が出る
- `GET /document_approval_requests` の全体一覧は internal user 向けで、`日時` `文書名` `依頼名` `依頼者` `確認相手` `状態` を見直せる
- 詳細画面では `状態` `依頼者` `確認相手` `対応者` `OK日時` `Cancel日時` を確認できる
- pending の依頼は、internal user が `OK`、internal user または依頼者本人が `Cancel` できる
- 一覧の `依頼名` link は current 一覧 URL を `return_to` として detail へ渡すため、`対応待ち` / `OK済み` / `Cancel済み` の filter をかけた全体一覧でも、文書配下の一覧でも、`一覧へ戻る` は元の一覧条件へ戻る
- detail で `OK` や `Cancel` を実行したあとも同じ detail に戻り、`一覧へ戻る` の戻り先は保持される

使いどころ:

- 文書に対して、誰に確認してほしいかを明示して依頼を残したいとき
- internal user が、案件横断で pending の確認依頼を見直したいとき
- 依頼者が、自分の依頼が処理済みかを個別詳細で追いたいとき

注意点:

- dashboard 上で `確認依頼` への導線が見えるのは internal user だけ
- `GET /document_approval_requests` の全体一覧と文書配下一覧を日常的に見直せるのは internal user だけで、依頼者本人は自分が作成した依頼の detail を閲覧し、pending の間だけ `Cancel` を使える
- 一般利用者向けの個人導線は `文書ショートカット` が主で、`確認依頼一覧` を日常入口として前提にしない

## 迷ったときの切り分け

- 案件をまたいで文書を探したい: dashboard の `閲覧可能文書` から入る
- 案件を決めてから配下文書やツリーを見たい: `案件一覧` または dashboard の `閲覧可能案件` から入る
- 自分用の近道を整理したい: `文書ショートカット` を使う
- 自分が送った申請の状態や `取消` を見直したい: dashboard の `保留中の申請` から入り、[利用者向けアクセス申請runbook](./利用者向けアクセス申請runbook.md) を見る
- 誰かにレビューや確認を正式に頼みたい: 文書詳細から `確認依頼` を作る
- internal user が未処理の確認依頼件数だけを先に見たい: dashboard の `保留中の確認依頼` から入る
- internal user が受付中の公開 Q&A を短く見直したい: dashboard の `受付中Q&A候補` から該当文書または版の Q&A workspace へ入る
- internal user が案件横断で依頼や送付履歴を追いたい: dashboard の `社内向け導線` から各一覧へ進む
